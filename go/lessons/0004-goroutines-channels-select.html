1 · Goroutines: go and gone

A goroutine is a function running concurrently, started with one keyword:

go checkOne(l)   // returns IMMEDIATELY; checkOne runs... whenever, on some core

Two facts with teeth. First: go f() doesn't wait — the next line runs before f likely even starts. Second: when main returns, every goroutine dies instantly, mid-sentence. So launching is trivial; the real skill is waiting correctly. The standard tool is a WaitGroup:

var wg sync.WaitGroup
for _, l := range links {
    wg.Add(1)                    // +1 BEFORE starting, never inside the goroutine
    go func() {
        defer wg.Done()          // -1 when this worker finishes, even on panic
        // ... work ...
    }()
}
wg.Wait()                        // blocks until the counter is zero

2 · Channels: how goroutines talk

Goroutines don't return values to their starter — they send them over a channel:

results := make(chan result)     // unbuffered: send blocks until someone receives
results <- r                      // send (inside a worker)
r := <-results                    // receive (in the collector)

Unbuffered channels are a meeting point: the send and the receive happen together, or both wait. make(chan result, 8) adds a buffer — sends don't block until it's full. Start unbuffered; add buffers only with a reason (Effective Go).

When no more values are coming, the sender side calls close(ch). Receivers detect it with the comma-ok pattern you already know from maps:

r, open := <-results   // open == false → channel closed AND drained

Same two-value shape, same job: distinguishing "a real zero value" from "there's nothing there".

3 · select: waiting on several things at once

select is switch for channels — it waits until one case is ready:

select {
case r := <-results:              // a worker reported
    out[r.code] = r.ok
case <-time.After(timeout):       // ... or the clock won
    return out                    // partial results beat hanging forever
}

This is the backend heartbeat pattern: result, timeout, or cancellation — whichever arrives first. Tomorrow context replaces time.After for the cancellation half; the select stays identical.

4 · The race detector: your new reflex

Two goroutines touching the same variable, at least one writing, with no coordination — a data race. count++ is secretly read-modify-write, so 1000 goroutines incrementing yields... some number ≤ 1000, different each run. The fix is a mutex, and the detector is a flag:

go test -race ./...

Go instruments every memory access and prints the two guilty stack traces. The mission says this should be a reflex, not an afterthought — you'll trigger it deliberately in a minute (Go blog: race detector).

Step 1 — the checker skeleton. Create checker.go. Given except three TODOs:

package main

import (
    "strings"
    "sync"
    "time"

    "linkshort/link"
)

// result carries one link's health check back over the channel.
type result struct {
    code string
    ok   bool
}

// checkOne simulates a network health check (~50ms). Deterministic on purpose:
// a real http.Get here would make the tests flaky.
func checkOne(l link.Link) result {
    time.Sleep(50 * time.Millisecond)
    return result{code: l.Code, ok: strings.HasPrefix(l.URL, "https://")}
}

// checkAll fans out one goroutine per link, collects results over a channel,
// and gives up (returning partial results) after timeout.
func checkAll(links []link.Link, timeout time.Duration) map[string]bool {
    results := make(chan result)

    // TODO 1: loop over links; for each one wg.Add(1) and launch a goroutine
    //         that defer wg.Done()s and sends checkOne(l) into results.
    //         (Section 1 has this shape almost verbatim.)
    var wg sync.WaitGroup

    // GIVEN — the closer: when every worker is done, close the channel so
    // the receive loop knows nothing else is coming. This runs in its OWN
    // goroutine because wg.Wait() blocks — and we need to start receiving NOW.
    go func() {
        wg.Wait()
        close(results)
    }()

    out := make(map[string]bool, len(links))
    deadline := time.After(timeout)
    for {
        // TODO 2: select over two cases —
        //   receive `r, open := <-results`: if !open return out, else store r
        //   receive from deadline: return out (partial results)
        //   (Section 3 is this select, minus the loop.)
    }
}

Step 2 — the tests (given, create checker_test.go).

package main

import (
    "testing"
    "time"

    "linkshort/link"
)

func TestCheckAllChecksEveryLink(t *testing.T) {
    links := []link.Link{
        {Code: "go", URL: "https://go.dev"},
        {Code: "bad", URL: "http://insecure.example"},
        {Code: "rust", URL: "https://www.rust-lang.org"},
    }
    got := checkAll(links, time.Second)
    if len(got) != 3 {
        t.Fatalf("checked %d links, want 3: %v", len(got), got)
    }
    if !got["go"] || got["bad"] || !got["rust"] {
        t.Fatalf("wrong verdicts: %v", got)
    }
}

func TestCheckAllTimesOut(t *testing.T) {
    links := []link.Link{{Code: "a", URL: "https://a.dev"}, {Code: "b", URL: "https://b.dev"}}
    got := checkAll(links, time.Millisecond) // far below checkOne's 50ms
    if len(got) == 2 {
        t.Fatal("expected partial results under a 1ms timeout, got all of them")
    }
}

Run go test ./... — both should pass once TODOs 1–2 are in. The timeout test is the interesting one: it proves your select really races the clock.

Step 3 — TODO 3, meet the race detector. Create race_test.go with a counter test: 1000 goroutines, each incrementing a shared count guarded by var mu sync.Mutex (mu.Lock(); count++; mu.Unlock()), a WaitGroup, then assert count == 1000. Then:

go test -race ./...        # passes — the mutex serializes the writes

Now delete the three mutex lines and run it again. This is what a production bug looks like when you're lucky enough to catch it in CI:

WARNING: DATA RACE
Read at 0x00c0000941e8 by goroutine 10:
  ... race_test.go:17
Previous write at 0x00c0000941e8 by goroutine 9:
  ... race_test.go:17

Two goroutines, same address, one line of your code named twice. Restore the mutex, watch it pass. From today, -race rides along on every test run.

Step 4 — wire it into main. At the end of main, health-check a demo batch:

health := checkAll([]link.Link{
    {Code: "go", URL: "https://golang.org"},

fmt.Println("health:", health)
Done when: go vet ./... silent · go test -race ./... green (with the mutex back!) · go run . ends with health: map[go:true old:false]. That's a concurrent fan-out with timeout — the shape inside every Go backend you'll interview at.
