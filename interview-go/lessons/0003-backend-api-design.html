Go Screening Kit · Part 3 · 4 questions · pick 2 per screen

Backend & API design

Language depth without service-building judgment makes a good puzzle-solver, not a good backend hire. These questions test whether they've owned a service — structured it, shut it down cleanly, and survived its failure modes. Your own 5 years of fullstack work counts fully here: most answers you can evaluate against your own experience.

Question 10

"Walk me through how you'd structure a new REST API service in Go — project layout, layers, how dependencies get wired together."

Why ask: open-ended by design. You learn what they consider normal, which reveals the codebases they've actually lived in.

Answer guideline — a strong answer covers

  • Some layering with clean direction: handler → service/domain → repository (names vary; the separation matters, not the vocabulary). Handlers do HTTP, domain does rules, repos do storage.
  • Interfaces defined where they're consumed (the service defines the Store interface it needs; postgres implements it) — this is Go-idiomatic and enables testing without mock frameworks.
  • Constructor injection: dependencies passed via New...(...), wired explicitly in a thin main. No global state, no init-magic, usually no DI framework.
  • Pragmatism about layout: cmd/ + internal/ is common, but "start flat, split when it hurts" is an equally strong answer. Framework choice (chi/echo/stdlib) held loosely.

Strong signals

  • Consumer-side interfaces — the single most Go-native structural idea
  • Thin main, explicit wiring, testability as a stated goal
  • Can say what they'd not do (over-layering a small service)

Weak / red flags

  • Recites a "standard layout" template with no reasoning
  • Everything hangs off globals / init() side effects
  • "The framework generates it" — structure outsourced entirely

Follow-up probe

"Why define the interface next to the consumer instead of next to the implementation?" (Good: the consumer states the minimal contract it needs; implementations satisfy it implicitly; tests substitute fakes without import cycles.)

3/5 = clean layering + injection. 5/5 = consumer-side interfaces with the why, plus taste about when structure is overkill.

Question 11

"Your service is behind a load balancer and gets a deploy signal. How do you shut it down without dropping requests?"

Why ask: graceful shutdown is a rite of passage — you implement it the first time your deploys start throwing 502s. Concrete, small, and impossible to bluff.

Answer guideline — a strong answer covers

  • Catch SIGTERM/SIGINT (signal.NotifyContext or signal.Notify).
  • Call server.Shutdown(ctx) with a deadline: stops accepting new connections, lets in-flight requests drain, returns when done or when the deadline expires.
  • Order matters: flip the readiness/health check first so the LB stops routing, then drain, then close DB pools / flush queues / stop background workers.
  • Bonus: the timeout must fit the platform's grace period (e.g. Kubernetes terminationGracePeriodSeconds); long-lived connections (websockets, streaming) need their own strategy.
ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGTERM, syscall.SIGINT)
defer stop()
// ... srv.ListenAndServe() in a goroutine ...
<-ctx.Done()                                  // deploy signal arrives
shutCtx, cancel := context.WithTimeout(context.Background(), 15*time.Second)
defer cancel()
srv.Shutdown(shutCtx)                         // drain in-flight, then exit

Strong signals

  • Names server.Shutdown and what it actually does
  • Thinks about the LB/readiness handshake, not just the process
  • Mentions cleaning up non-HTTP resources (pools, workers, consumers)

Weak / red flags

  • "Just os.Exit / the platform handles it"
  • Confuses Shutdown with Close
  • Has clearly never seen deploy-time 502s or thought about why they happen

3/5 = signal + Shutdown + timeout. 5/5 = readiness-first ordering and platform grace-period awareness.

Question 12

"You're calling a downstream service that's occasionally slow or flaky. How do you add timeouts and retries — and what can retries make worse?"

Why ask: the second half is the real question. Naive retries have taken down more systems than the failures they were meant to survive.

Answer guideline — a strong answer covers

  • Timeouts: per-request context.WithTimeout; a configured http.Client{Timeout: ...} (never the zero-value default client, which has no timeout); budget awareness — the outbound deadline must fit inside the inbound request's remaining deadline.
  • Retries: bounded attempts, exponential backoff with jitter, retry only idempotent operations or ensure idempotency (idempotency keys) before retrying writes.
  • What goes wrong: retry storms/amplification — a struggling downstream gets 3× the traffic exactly when it can least handle it; duplicate side effects on non-idempotent writes (double-charged payment); thundering herd without jitter.
  • Bonus: circuit breakers / failing fast, honoring Retry-After, retrying on the right errors only (timeouts/5xx yes, 4xx no).

Strong signals

  • Says "idempotent" unprompted when discussing retried writes
  • Jitter, and can say why (synchronized retry waves)
  • Knows the default http.Client has no timeout

Weak / red flags

  • "Retry 3 times in a loop" with no backoff, no idempotency thought
  • No concept of deadline budgets across hops
  • Can't name a single way retries make an outage worse

3/5 = timeout + bounded backoff retries. 5/5 = idempotency + retry-amplification reasoning; circuit breaker vocabulary is a bonus, not required at 3–4 yrs.

Question 13

"How do you validate and evolve API request payloads? Say a client needs a new field next month — walk me through the change."

Why ask: tests day-to-day craftsmanship and whether they think about clients they don't control.

Answer guideline — a strong answer covers

  • Dedicated request/response structs per endpoint — never binding JSON straight into DB models (over-posting, accidental field exposure, coupled evolution).
  • Validation after decode: required fields, ranges, enums — via a library (validator tags) or plain code; returning structured 400s that name the bad field.
  • Go-specific nuance: encoding/json ignores unknown fields by default (DisallowUnknownFields to opt out) and zero values are ambiguous — "was count sent as 0 or omitted?" → pointers, or a decode-then-check pattern.
  • Evolution: additive changes are safe (new optional field with a default); breaking changes (rename, type change, new required field) need versioning or a deprecation window. Never repurpose an existing field's meaning.

Strong signals

  • Separate DTOs from domain/DB models, with the why
  • Knows the zero-value-vs-omitted JSON ambiguity
  • "Additive is free, breaking needs a version/deprecation plan"

Weak / red flags

  • Binds JSON directly into GORM/DB models
  • Validation is "the frontend checks it"
  • Would ship a renamed field and "tell the clients"

3/5 = DTOs + server-side validation. 5/5 = JSON zero-value nuance + a coherent compatibility story.
