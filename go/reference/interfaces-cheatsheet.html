Reference sheet · Interfaces & Composition

Go interfaces, on one page

Rules of thumb

  • Satisfaction is implicit. Have the methods → satisfy the interface. No keyword, no registration.
  • Define at the consumer. The package that calls the behavior declares the interface. Implementers return concrete types.
  • Keep them small. 1–2 methods is normal. "The bigger the interface, the weaker the abstraction." — Go proverb
  • Accept interfaces, return structs. Params = behavior you need. Returns = full concrete type.
  • Compose by embedding. Promoted methods, override by redeclaring, delegate with outer.Inner.Method().
  • Don't abstract preemptively. Add an interface when a second implementation (or a test fake) needs it — not before (100 Go Mistakes #5).

Snippets

Consumer-defined interface

// package handler — needs only lookups
type UserGetter interface {
    Get(id string) (User, error)
}

Compose interfaces from interfaces

type ReadWriter interface {
    Reader
    Writer
}

Decorator via embedding (middleware pattern)

type LoggingStore struct {
    Store
    log *slog.Logger
}
func (s LoggingStore) Save(ctx context.Context, l Link) error {
    s.log.Info("saving", "code", l.Code)
    return s.Store.Save(ctx, l)
}

Compile-time satisfaction check

var _ Store = (*MemStore)(nil)  // breaks the build if MemStore stops satisfying Store

Interview one-liners

  • Why implicit satisfaction? Decouples packages: consumers abstract without implementers opting in.
  • Why no inheritance? Embedding gives reuse via composition without fragile hierarchies.
  • nil-interface gotcha: an interface holding a nil pointer is not itself nil — an interface value is (type, value); it's nil only when both are unset. Never return concrete-typed nils as interfaces.
