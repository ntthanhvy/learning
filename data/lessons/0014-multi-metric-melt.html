Melt revisited: multiple metrics at once

Lesson 6 melted a wide table with exactly one value per cell (amount). Real reports rarely stop at one metric — a monthly report wants total and count per customer per month, side by side. Today builds that wider table, and shows why melting it back needs one extra step Lesson 6 didn't.

Recap, in one line

Lesson 6: pivot_table spreads one value column across new columns; melt un-spreads it back. Today: what happens when pivot_table spreads two value columns at once.

1 · Two aggfuncs, one pivot_table call

pivot_table accepts a list of aggregate functions in aggfunc=, not just one. Passing aggfunc=["sum", "count"] over the same values="amount" computes both at once — the SQL equivalent is running SUM(amount) and COUNT(amount) in the same GROUP BY query, which you'd normally just do as two columns in one SELECT. pandas does the same thing, but the result isn't flat:

wide = clean.pivot_table(
    index="customer", columns="order_date",
    values="amount", aggfunc=["sum", "count"], fill_value=0,
)

The output's columns are now a MultiIndex: level 0 is which aggfunc produced the value ("sum" or "count"), level 1 is the date, same as Lesson 6. A single column is no longer just "2026-01-05" — it's the tuple ("sum", "2026-01-05"), and wide.loc["An", ("sum", "2026-01-05")] is how you'd address one cell directly.

2 · Why melt can't touch this directly

melt was built for the one-level column case Lesson 6 used — its var_name expects to record one plain column name per melted row, not a two-part tuple. Calling .melt() straight on a MultiIndex-column DataFrame either errors or produces a confusing multi-column variable_0/variable_1 split you didn't ask for. The fix is to flatten the column MultiIndex into ordinary strings first, joining the levels with a separator:

flat = wide.set_axis(
    ["_".join(map(str, c)) for c in wide.columns.to_flat_index()],
    axis=1,
)
-- ("sum", "2026-01-05") becomes the single string "sum_2026-01-05"

to_flat_index() turns the MultiIndex into a plain sequence of tuples; the list comprehension joins each tuple's parts into one string. Once every column is a flat string, melt works exactly like Lesson 6 — reset_index() (customer is still sitting in the index from the pivot), then .melt(id_vars="customer", var_name="metric_date", value_name="value").

3 · Un-merging the metric from the date

The melt gets every cell into one long column, but metric_date is still a merged string like "sum_2026-01-05" — one melt turned a 2D grid into rows, but the two things that used to be separate column levels are now squashed into one text value. str.split with expand=True un-squashes it back into real columns:

long[["metric", "order_date"]] = long["metric_date"].str.split("_", n=1, expand=True)

n=1 caps it at the first underscore — a safety habit, not a requirement here (today's dates have no underscores either way), but it matters the moment a metric name itself contains one ("avg_price_2026-01-05" should split into "avg_price" and "2026-01-05", not three pieces). expand=True is what turns a Series of two-item lists into an actual two-column DataFrame that can be assigned straight into long[["metric", "order_date"]].

The result is a middle ground between Lesson 6's fully-long and fully-wide shapes: one row per (customer, date, metric) instead of one row per (customer, date) with metrics as separate columns. This "long with a metric column" shape is usually what a downstream groupby or a charting library actually wants — df.groupby(["order_date", "metric"])["value"].sum() reads naturally; a wide table with hand-picked column names per metric doesn't.

4 · The other tool for this: stack()

Lesson 6 flagged stack() as "out of scope for today" — it's the MultiIndex-native alternative to the flatten-then-melt path above. wide.stack(level=0) pulls the aggfunc level directly down into the row index, no string-flattening step needed, because stack already understands MultiIndex columns the way melt doesn't. The trade-off: stack's output keeps metric as an index level, not a plain column like melt's var_name gives you — so it's usually followed by its own reset_index() to get the same tidy shape. Both tools reach the same destination; melt-after-flattening is the more portable habit (works no matter how the wide table was built), stack is the shorter path specifically when you already have MultiIndex columns in hand.

Q: "You pivoted a table to get both a total and a count per group per month, and now need it back in long format for a groupby. Walk me through it." A strong answer, out loud: "Pivoting with two aggfuncs gives MultiIndex columns — one level for the metric, one for the date — and melt only understands flat, single-level columns, so I can't call melt directly on that. I'd flatten the columns first, joining the metric and date into one string per column, then melt normally with var_name capturing that joined string. The one thing melt doesn't do for me is separate the metric back out from the date — that merged string is still one value, so I'd follow it with a str.split on the separator, expand=True, to land the metric and date in their own columns. The alternative is stack(), which handles the MultiIndex natively without the flatten step, but it leaves the metric as an index level rather than a column, so I still need a reset_index() afterward either way." The signal: naming why melt can't run directly on MultiIndex columns (not just reciting the fix), and knowing stack() exists as the native alternative even if melt-after-flatten is the one actually used.
