Unstack: pulling a row-index level back into columns

Lesson 14 named stack() as the MultiIndex-native alternative to melt, and left its inverse, unstack(), as a teaser. Today builds it from the other direction: start from a plain groupby, not a pivot, and watch unstack turn its long result wide.

Recap, in one line

Lesson 14: a MultiIndex can sit on the columns (what two-aggfunc pivot_table produces). Today: a MultiIndex sitting on the rows instead — what a multi-column groupby produces — and unstack as the tool that moves a row level onto the columns.

1 · A groupby on two columns is already a MultiIndex

Group by more than one column and pandas doesn't flatten the result — it stacks the group keys into a MultiIndex row index, one level per column grouped on:

by_customer_date = clean.groupby(["customer", "order_date"])["amount"].sum()
-- customer  order_date
-- An        2026-01-05    120.0
--           2026-01-10     42.0
-- Binh      2026-01-06     35.5
--           2026-01-09    180.0

This is the exact SQL equivalent of GROUP BY customer, order_date — one row per (customer, date) pair, just like SQL returns. The pandas difference is only presentation: SQL always hands back a flat table, but pandas keeps the two group keys as index levels rather than ordinary columns, which is what makes this a MultiIndex — the same concept Lesson 14 taught, just on the row axis instead of the column axis.

2 · unstack moves one of those levels onto the columns

The long shape above is correct but awkward to eyeball — comparing An and Binh side by side means scanning down a tall column. unstack takes one named index level and spreads its values across new columns, the same "long → wide" move Lesson 6's pivot_table made, arrived at from a groupby instead:

wide_by_date = by_customer_date.unstack("order_date", fill_value=0)
--           2026-01-05  2026-01-06  2026-01-09  2026-01-10
-- customer
-- An             120.0         0.0         0.0        42.0
-- Binh             0.0        35.5       180.0         0.0

fill_value=0 matters here the same way it did in Lesson 6's pivot_table call: An never ordered on 2026-01-06, so without fill_value that cell would be NaN — correct in the sense that the combination genuinely doesn't exist, but usually not what a report wants for a currency total. unstack is, in fact, exactly what pivot_table does under the hood for the single-aggfunc case: pivot_table is a shorthand for "groupby, then unstack." Reaching for groupby().unstack() directly instead of pivot_table is worth it whenever the long, grouped result is useful on its own — for a running total, a rank, or a join — before it also gets reshaped wide.

3 · Which level moves is a choice, not a default

unstack takes a level name (or position — 0 for the outermost, 1 for the next) precisely because a 2-level MultiIndex can be unstacked either way, and each choice answers a different question from the exact same grouped result:

wide_by_customer = by_customer_date.unstack("customer", fill_value=0)
-- customer      An     Binh
-- order_date
-- 2026-01-05  120.0     0.0
-- 2026-01-06    0.0    35.5
-- 2026-01-09    0.0   180.0
-- 2026-01-10   42.0     0.0

Same by_customer_date Series, same two calls apart — but unstacking "customer" instead of "order_date" leaves dates in the row index and spreads customers across columns instead, "one row per date, one column per customer" rather than the reverse. Whichever level stays in the row index is the one still easy to filter and sort on afterward (wide_by_customer.loc["2026-01-09"] reads naturally; reaching a single date out of wide_by_date's columns does not) — that's the real deciding factor for which level to unstack, not just which shape looks nicer printed.

fill_value never changes a real total. It only fills gaps that were never there to begin with — the grand total of every cell in wide_by_date (zeros included) is still 377.5, identical to summing the original four amount values before any grouping or reshaping happened. If a total changes after unstack, the bug is upstream, not in fill_value.

Q: "You've got a Series with a 2-level MultiIndex from a groupby. How do you get it into a wide table, and how is that different from just calling pivot_table?" A strong answer, out loud: "unstack() takes a named level of the row index and spreads its values across new columns — so a groupby on customer and order_date, followed by unstack('order_date'), turns the long result into customer-by-date. pivot_table does the same underlying thing in one call, but it hides the intermediate long Series, and I lose the chance to reuse that long form for something else — a rank, a join, a running total — before reshaping it. The other thing unstack makes explicit that pivot_table doesn't: which level moves is my choice, not a default. The same Series unstacked on customer instead of order_date gives me date-by-customer instead, and I pick whichever one leaves the level I still need to filter or sort on inside the row index." The signal: naming unstack and pivot_table as the same underlying operation at different levels of abstraction, not two unrelated tools.
