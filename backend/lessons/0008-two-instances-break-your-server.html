Why two instances break your "stateless" server

Lesson 7 left off here: your service just got a second instance behind a load balancer — what breaks the moment "the server" stops being one process. Lessons 2 and 6 already used the word stateless as a design goal. Today is where that stops being a nice phrase and becomes a hard requirement — because the moment there are two processes, any state you quietly left in memory becomes state that only half your users can see.

1 · The mental model that breaks

Up through Lesson 7, "the server" meant one running process. One process means one place any in-memory value lives — a cache map, a counter, a variable holding the last thing a client did. Even if you never thought about it, the mental model was: whatever this process remembers, every future request from any client will see, because there's only one process to ask.

Horizontal scaling — running two, five, or fifty copies of the exact same Go binary — breaks that model without changing a single line of handler code. A load balancer sits in front, and on every incoming request it picks one instance to send it to, usually round-robin or by whichever instance is least busy right now. Nothing guarantees the same client's next request lands on the same instance. Nothing even guarantees it lands on a different one — it's genuinely arbitrary from the client's point of view. The moment that's true, "whatever this process remembers" quietly becomes "whatever this one out of N processes remembers" — and every other instance has no idea it happened.

2 · Where in-memory state hides, and what it costs you

None of these bugs show up with one instance. All of them show up in production the day someone adds a second one — which is exactly why they're dangerous: they pass every test that only ever ran one process.

  • In-memory session store. Lesson 4 introduced the session store as the place login state actually lives. If a handler kept it as a package-level Go map instead of a database table, a user who logs in and lands on instance A gets a session only instance A knows about. Their very next request — routed by the load balancer to instance B — looks unauthenticated, because instance B's map never heard about that login. From the user's side this looks like "I keep getting randomly logged out," which is one of the most common horizontal-scaling bugs there is.
  • In-memory rate limiter or cache. A rate limiter meant to allow "5 requests per minute per user" that counts in a local variable is actually enforcing 5 requests per minute per instance. Behind 4 instances, the same user can make up to 20 requests a minute by pure luck of which instance answers each one — the limit silently multiplies by your replica count, and nobody configured that on purpose.
  • Local file writes. A handler that saves an uploaded file to local disk (e.g. ./uploads/avatar.png) writes it to whichever instance happened to handle that request. The next request that tries to read it back — an image tag loading the avatar — might land on a different instance whose disk never received that file, and 404s on something that was "just uploaded a second ago."

The common thread: each of these is state that only makes sense if there's exactly one process. The moment there's more than one, it either silently disappears (the session, the file) or silently multiplies (the rate limit) — and it always looks like a bug in something else first, because nobody suspects "the number of instances" as the cause.

// the bug in miniature — this compiles, passes local tests, and is wrong the moment N > 1
var cache = map[string]int{} // lives in THIS process's memory only

func handleHitCount(w http.ResponseWriter, r *http.Request) {
    key := r.URL.Query().Get("page")
    cache[key]++ // instance A and instance B each keep their OWN count
    fmt.Fprintf(w, "%d", cache[key])
}

3 · The fix: push shared state somewhere every instance can reach

The fix is the same idea in all three cases, and it's exactly what "stateless" was promising back in Lesson 2 and Lesson 6, made concrete now that replication is real: a handler is allowed to hold state only for the duration of one request. Anything that must survive past that request — a login, a counter, an uploaded file — belongs somewhere every instance can read and write the same copy of it: the Postgres database, or a shared cache like Redis.

  • Session store → a sessions table in Postgres (or a Redis hash keyed by token) — either way, one shared source that any instance can look up by token.
  • Rate limiter → a counter in Redis with a TTL, incremented atomically per user per window — every instance increments the same counter, so the limit means what it says regardless of which instance handles which request.
  • Uploaded files → written to shared object storage (or at minimum a shared network volume), not a given instance's local disk — any instance can serve the file back, because none of them "own" it.

Once every instance is reading and writing the same shared state instead of its own local copy, the instances become genuinely interchangeable — this is the property sometimes called shared-nothing at the instance level: nothing any one instance holds is unique to it, so losing an instance loses nothing, and the load balancer's arbitrary routing decision stops mattering.

4 · The band-aid: sticky sessions, and why it's usually the wrong fix

There's a shortcut that looks like it solves this without touching any code: configure the load balancer to always send the same client back to the same instance, based on a cookie or their IP — called sticky sessions (or session affinity). If instance A always sees the same user, its in-memory session map "works again."

It's usually the wrong fix. It defeats the actual point of load balancing — traffic can no longer spread evenly, because a client is pinned regardless of how busy their instance is. And it breaks in exactly the moments you need scaling to help most: restart that instance, or scale down during a quiet period, and every client pinned to it loses their session at once, with no other instance able to pick up where it left off — because the state was never anywhere else. Sticky sessions treat the symptom (which instance a client hits) instead of the disease (state that shouldn't have been local in the first place).

Frontend-habit trap: this is the server-side mirror of a bug you'd never make in the browser — you wouldn't store a value in one tab's memory and expect a different tab to see it. Two server instances are exactly two tabs that happen to share a URL; the fix is the same instinct you already have: shared state goes somewhere both can read, not in either one's local memory.

Q: "How would you scale this API horizontally?" A strong answer, out loud: "First I'd check what state the handlers are keeping in-process — sessions, caches, rate limit counters, file writes — because none of that survives having more than one instance unless it's already in the database or a shared cache like Redis. Once that's true, I can put a load balancer in front and add instances freely, since any instance can handle any request. I'd avoid sticky sessions as the fix — it just hides state that's still only on one instance, and it breaks the first time that instance restarts or gets scaled down." The subtext: naming which specific things tend to leak into memory (sessions, caches, uploads) is what separates this from reciting "make it stateless" as a slogan.
