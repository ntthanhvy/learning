Logging & monitoring: how you find out something's wrong

Lesson 12 closed on this teaser. Lesson 7 already introduced two pieces — the correlation ID and structured logs — but only as the escape hatch for a single 500 response. Today zooms out: what a backend logs continuously, in production, all day, and how anyone (not just the request that just failed) notices trouble before a user reports it.

1 · console.log doesn't survive the frontend/backend jump

The frontend habit is console.log and a browser tab you're already looking at. That instinct breaks completely on a server: nobody is watching the terminal of a production process at 3am, the process might be one of many instances (Lesson 8), and by the time a user complains the moment you'd want to inspect is long gone. A backend's logs are not a debugging convenience you glance at — they're the only record of what happened, so they have to be written continuously, to somewhere durable, whether or not anyone is looking that second.

2 · Log levels: not every line deserves the same attention

Lesson 7's request_id-keyed logs answered "what happened during this one failed request." A running server produces log lines constantly, most of them boring, so every line is tagged with a log level so the ones that matter don't drown in the ones that don't:

  • DEBUG — fine-grained detail, off by default in production, turned on only while actively chasing a specific bug.
  • INFO — routine, expected events: a request came in, a job finished, a user signed up. Normal operation, worth keeping, never urgent.
  • WARN — something unexpected but recovered from automatically: a retry succeeded on the third attempt, a cache miss fell back to the DB.
  • ERROR — something failed and did not recover: the exact class of thing Lesson 7's 500 handler logs. Someone should be able to see this without reading everything else.

The level is what lets a dashboard show "only ERROR, last hour" instead of an undifferentiated firehose — the same reason a JS framework separates console.log from console.error, just load-bearing here instead of cosmetic.

3 · Structured logs, machine-readable by design

Lesson 7 introduced structured logging for one failure; it's the default for every line, not a special case:

logger.Info("charge attempt started",
    "request_id", requestID,
    "user_id", userID,
    "amount_cents", 4200,
)
// → {"level":"INFO","msg":"charge attempt started","request_id":"req_8f3a","user_id":"user_123","amount_cents":4200}

A free-text sentence like "charging user_123 $42.00" is fine for a human eyeballing a terminal and useless for a query. The JSON version answers "show me every ERROR for user_123 in the last hour" as a precise field filter — a real query — instead of a hopeful regex over sentences that might quietly change wording next release. Go's standard library has shipped exactly this since 1.21, under log/slog:

var logger = slog.New(slog.NewJSONHandler(os.Stdout, nil))

func chargeCardHandler(w http.ResponseWriter, r *http.Request) {
    requestID := r.Header.Get("X-Request-ID")
    userID := "user_123" // pulled from auth middleware (Lesson 4)

    logger.Info("charge attempt started",
        "request_id", requestID, "user_id", userID, "amount_cents", 4200)

    if err := chargeCard(r.Context(), userID, 4200); err != nil {
        logger.Error("charge failed",
            "request_id", requestID, "user_id", userID, "error", err.Error())
        http.Error(w, "payment failed", http.StatusPaymentRequired)
        return
    }
    w.WriteHeader(http.StatusOK)
}

Every line still carries the same request_id Lesson 7 keyed a single error by — that's not a coincidence. Structured logging and correlation IDs are the same idea applied everywhere at once: put a stable, filterable field on every line, all the time, not just the one line written when something already broke.

What never goes in, even structured: Lesson 7's information-disclosure rule about response bodies applies just as hard to logs — passwords, full card numbers, session tokens, and raw auth headers should never be logged at all, structured or not. A log line is a durable, searchable, often widely-readable record; putting a secret in one doesn't leak it to an attacker on the wire, but it does leak it to everyone with log access, forever, including anyone who breaches the logging system itself later.

4 · Logs answer "what happened"; monitoring answers "is it happening now"

Logs are a record you search after you already suspect something's wrong — reactive, one event at a time. Monitoring is the proactive half: a small set of numbers, tracked continuously, that make a trend visible before a customer files a ticket. The engineering term for one such number is a metric: not a discrete event like a log line, but a number sampled repeatedly and graphed — request rate, error rate, p99 latency, how many jobs are sitting in Lesson 10's queue right now.

The two are complementary, not competing: a metric graph shows error rate spiking at 14:02 — that's the signal something's wrong and roughly when. The logs, filtered to ERROR around 14:02, show which requests and why. Monitoring finds the needle's haystack; logs are what you search once you know which haystack to search.

5 · Health checks: a metric an orchestrator asks for directly

One specific, very common check deserves its own shape: an HTTP endpoint whose only job is answering "are you okay?" — polled automatically, often every few seconds, by whatever manages your instances (Lesson 8's load balancer, or a container orchestrator). It's worth splitting into two genuinely different questions:

// liveness: is this PROCESS alive at all? No dependency checks —
// if this ever fails to respond, the orchestrator kills and restarts it.
func (h *Health) livenessHandler(w http.ResponseWriter, r *http.Request) {
    w.WriteHeader(http.StatusOK)
}

// readiness: can this INSTANCE actually serve a real request right now?
func (h *Health) readinessHandler(w http.ResponseWriter, r *http.Request) {
    ctx, cancel := context.WithTimeout(r.Context(), 2*time.Second)
    defer cancel()
    if err := h.db.Ping(ctx); err != nil {
        w.WriteHeader(http.StatusServiceUnavailable) // tell the LB: skip me for now
        return
    }
    w.WriteHeader(http.StatusOK)
}

Conflating them is a common mistake: if liveness checked the database too, one slow Postgres connection would make the orchestrator kill and restart a perfectly healthy process, over and over, solving nothing. Liveness is about the process; readiness is about this instance's dependencies right now, and it's the load balancer's job to route around a not-ready instance temporarily, not restart it.

Q: "How would you know a production service is unhealthy before a customer tells you?" A strong answer, out loud: "Two separate layers. Metrics — request rate, error rate, p99 latency — get graphed continuously and alert on thresholds, so a spike shows up as a trend within seconds, not when someone files a ticket. Every request also writes structured logs with a request ID and a level, so once an alert fires I can filter straight to ERROR-level lines around that timestamp instead of reading everything. And the service exposes separate liveness and readiness endpoints, so the orchestrator only restarts a process that's actually dead, and only stops routing to one that's temporarily not ready — like its DB connection being down — rather than treating those as the same failure." The signal: distinguishing "detect a trend" (monitoring/metrics) from "explain one event" (logs), and not conflating liveness with readiness.
