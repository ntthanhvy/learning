Go Screening Kit · Part 1 · 4 questions · pick 2–3 per screen

Go fundamentals

These separate "writes Go daily" from "did the tour once". Every question here has a surface answer and a depth answer — a genuine 3–4 year dev gives the depth answer without prompting, because they've been bitten by each of these in production.

How to use each card: ask the question exactly as written, stay silent, and let them finish. Then check what they covered against the guideline. The follow-up probe is where the real signal lives — first answers can be memorized, second-level answers can't.

Question 1

"What's the difference between a slice and an array in Go? And what happens when you append to a slice that was passed into a function?"

Why ask: slices are the most-used data structure in Go, and their sharing semantics cause real bugs. Anyone who has debugged one remembers it.

Answer guideline — a strong answer covers

  • An array has a fixed size that is part of its type ([3]int and [4]int are different types); arrays are copied on assignment. Slices are what you actually use day-to-day.
  • A slice is a small header: pointer + length + capacity over a backing array. Passing a slice copies the header, but both copies point at the same backing array.
  • So mutating elements inside a function is visible to the caller, but append is trickier: if there's spare capacity it writes into the shared array (caller may see it); if capacity is exceeded it reallocates, and from then on the function works on a private copy the caller never sees.
  • Fixes: return the slice from the function, or pre-size with make([]T, 0, n), or copy explicitly when you need isolation.

Strong signals

  • Says "pointer, length, capacity" unprompted
  • Explains the reallocation-on-append behavior and why it makes bugs intermittent
  • Mentions s[a:b] sub-slices sharing memory (or the three-index slice s[a:b:c])

Weak / red flags

  • "Slices are just dynamic arrays" and nothing more
  • Claims append always / never affects the caller (both wrong — it depends on capacity)
  • Can't say what capacity is

Follow-up probe

"You take a sub-slice of a 100 MB buffer and keep it around — what's the memory consequence?" (Good: the whole 100 MB backing array stays alive; copy the small piece out if you keep it long-term.)

3/5 = header semantics correct, append fuzzy. 5/5 = capacity/reallocation nailed plus the memory-retention follow-up.

Question 2

"When do you choose a pointer receiver vs a value receiver for a method?"

Why ask: it's an everyday decision that reveals whether they understand Go's copy semantics, not just its syntax.

Answer guideline — a strong answer covers

  • Pointer receiver when the method mutates the receiver, when the struct is large (avoid copying), or when any method on the type needs a pointer — the method set should be consistent, not mixed.
  • Value receiver for small immutable types (e.g. time.Time-style values).
  • Types containing a sync.Mutex (or any sync primitive, or a map/channel guard) must not be copied — a copied mutex is a different lock guarding nothing. go vet catches this.
  • Bonus: the interaction with interfaces — a value of type T doesn't satisfy an interface whose methods are on *T.

Strong signals

  • Mentions the mutex-copy trap without prompting
  • Talks about method-set consistency, not just per-method choice
  • Knows the *T-methods-vs-interface rule

Weak / red flags

  • "Always use pointers, they're faster" (cargo cult; small values often aren't)
  • Doesn't know a value receiver gets a copy
  • Never heard of go vet

Follow-up probe

"A method with a value receiver assigns to a struct field. What does the caller observe?" (Good: nothing — the method mutated its own copy. Instant, confident answer expected at this level.)

3/5 = mutation/copy reasoning correct. 5/5 = adds mutex-copy or interface method-set nuance.

Question 3

"Walk me through error handling in Go. How do you wrap errors, and how do you inspect a wrapped error later?"

Why ask: error handling is ~30% of any Go codebase. How someone talks about it tells you how much real Go they've read and reviewed.

Answer guideline — a strong answer covers

  • Errors are values returned, not exceptions thrown; the caller decides what to do at each level.
  • Wrapping with fmt.Errorf("fetch user %d: %w", id, err) — the %w verb preserves the chain and adds context at each layer.
  • Inspecting with errors.Is(err, sql.ErrNoRows) for sentinel errors and errors.As(err, &target) for typed errors carrying data.
  • Judgment: wrap with context at boundaries, don't double-log-and-return, translate internal errors to API errors at the handler layer (don't leak sql.ErrNoRows to HTTP clients).

Strong signals

  • Uses %w / errors.Is / errors.As fluently, distinguishing Is vs As
  • Has opinions: where to wrap, where to handle, log once
  • Mentions sentinel vs custom error types trade-off

Weak / red flags

  • Only knows if err != nil { return err } with no context-adding
  • Confuses %w with %v or has never used errors.Is
  • Reaches for panic/recover as a control-flow strategy

Follow-up probe

"When is panic acceptable?" (Good: programmer errors / impossible states, init-time failures; never for expected runtime errors like validation or a missing row. Bonus: a recover middleware at the HTTP layer as a last-resort crash barrier.)

3/5 = knows wrap + Is/As mechanically. 5/5 = layering judgment: where context is added, where errors are translated, log-once discipline.

Question 4

"Can an interface variable holding a nil pointer still be non-nil? What's going on there?"

Why ask: the classic "typed nil" trap. Nearly everyone with real Go mileage has lost an afternoon to it — the reaction to this question is itself the signal.

Answer guideline — a strong answer covers

  • Yes. An interface value is a pair of (concrete type, value). It's only == nil when both are nil.
  • Assigning a nil *MyError to an error variable gives an interface with type *MyError and value nil — so err != nil is true, and the nil check upstream "mysteriously" fails.
  • Typical source: a function with return type error that returns a possibly-nil concrete pointer type. Fix: return the error interface type and literal nil explicitly.
func doWork() error {
    var e *MyError            // nil pointer
    return e                  // non-nil interface! (type=*MyError, value=nil)
}
// caller: if err := doWork(); err != nil { ... }  ← always true

Strong signals

  • Immediate recognition — often with a war story
  • Explains the (type, value) pair model, not just the symptom
  • States the fix: return the interface type, literal nil

Weak / red flags

  • "No, nil is nil" — confidence without the model
  • Has the model explained and still can't reason to the fix

3/5 = recognizes it once you sketch the code. 5/5 = explains the interface internals unprompted. It's OK to skip this one if Q1–Q3 already went deep — it's a depth probe, not a gate.
