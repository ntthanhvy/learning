Recap, in one line

Lesson 1's whole point: pandas operations run on entire columns at once — vectorized, same instinct as a SQL WHERE — and a boolean mask is how you filter without a loop. Today assumes just that idea, nothing else — the DataFrame you built by hand yesterday is now one you'll load from disk, messy on arrival.

1 · read_csv: your SQL SELECT * from a file

In SQL, a table already has declared column types before you query it. A CSV has none of that — it's just text, and pandas has to guess a dtype per column by looking at the values:

df = pd.read_csv("orders_raw.csv")

That single call is doing real work: splitting rows, inferring one dtype per column, and turning recognizable blanks into NaN automatically. It usually gets numbers and clean dates right. "Usually" is doing a lot of work in that sentence — which is the whole lesson.

2 · The dtype that lies

Load a file with one dirty value and check what pandas actually inferred:

>>> df.dtypes
order_id       int64
customer         str
amount           str    -- should be numbers. one bad row poisoned the WHOLE column.
order_date       str    -- dates are just text until YOU say otherwise
One non-numeric value forces the whole column to text. pandas 3.0 (Sept 2025) names this the str dtype — a real, honestly-named text type. (Older books and pre-3.0 pandas call the same situation dtype('O')/"object" — same problem, older name.) Either way, the fix is identical: one bad row anywhere in a column, and every number in it becomes text, silently. df["amount"].sum() won't crash — it'll concatenate strings or raise, depending on version, neither of which is the number you wanted.

The fix is an explicit, coercing conversion — "convert what you can, and tell me clearly what you can't" instead of guessing:

amount_clean = pd.to_numeric(df["amount"], errors="coerce")
-- every parseable value becomes a float; "unknown" becomes NaN, not a crash

order_date_clean = pd.to_datetime(df["order_date"], errors="coerce")
-- every parseable date becomes a real datetime64; the blank field becomes NaT

errors="coerce" is the move you'll reuse constantly: it's the SQL equivalent of a TRY_CAST that returns NULL on failure instead of blowing up the whole query. A missing timestamp specifically gets its own value, NaT ("not a time"), so you can still ask "which rows lack a date" without special-casing strings.

3 · The Index — pandas' extra column you didn't ask for

A SQL table's rows have no inherent order; a DataFrame's rows always carry an Index — a label per row, separate from the data columns. Fresh from read_csv it's a plain RangeIndex (0, 1, 2…) — meaningless positions. If a column is really your row's identity (here, order_id), promote it:

df = df.set_index("order_id")   -- now df.loc[3] finds the row FOR order 3, not position 3

This matters because two different lookups exist and only one respects a custom index: .loc["label"] looks up by index label; .iloc[2] always means "the 3rd row physically," ignoring the index entirely. Mixing them up is a classic bug the moment a DataFrame gets sorted or filtered and the RangeIndex stops matching row position.

4 · The first-look ritual

Handed an unknown file, run these before writing a single transform — this is Lesson 1's teaser, answered:

df.shape          # (rows, cols) — the SELECT COUNT(*) reflex
df.dtypes         # one line per column — catches the lying dtypes above
df.head()          # first 5 rows — eyeball it
df.info()          # dtypes + non-null counts + memory, in one call
df.isna().sum()    # missing values PER COLUMN — your GROUP BY, one-liner style

df.isna().sum() is worth pausing on: isna() vectorizes a True/False mask over the whole frame (section 2's mask idea, applied to missingness), and .sum() adds each column's Trues — vectorized end to end, zero loops, and it's the fastest way to answer "which columns actually need cleaning."

Q: "You're handed a CSV you've never seen. Walk me through the first five minutes." A strong answer, out loud: "First shape and head() to see scale and eyeball the content. Then dtypes and info() — I'm specifically hunting for numeric or date columns that got read as text, because that means dirty rows are hiding in there. Then isna().sum() per column to quantify missingness before deciding whether to drop, fill, or flag it. Only after that do I write any transform — cleaning blind is how silent bugs get shipped." The subtext interviewers want: you don't trust the file, you verify it.
