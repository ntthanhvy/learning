Connection pool sizing & exhaustion

Lesson 1's request-lifecycle diagram named the connection pool in passing, and Lesson 17 left "going deeper on one track" open once every MISSION.md criterion had been touched — connection pool sizing was the specific example it named. This lesson is that: what the pool is actually saving you from, what happens when it runs out, and why "just make it bigger" isn't the fix it sounds like.

1 · What the pool is saving you from

Opening a database connection isn't free. Postgres runs one backend process per connection — not a thread, a whole OS process, using a few megabytes of memory and CPU time just to exist, before it ever runs a query. Getting there costs a TCP handshake plus an authentication round trip, tens of milliseconds that dwarf many queries' own execution time. Opening one fresh per request, then throwing it away, pays that cost on every single request. The connection pool exists to pay it once: open a small set of connections up front, keep them alive, and have every handler borrow one, use it, and return it.

That reframes the sizing question. A pool isn't "how many connections can I afford" — Postgres's default max_connections is 100 total, shared across every client that talks to that database, and each one costs real server memory whether it's doing anything or not. A pool is "how many connections does this instance actually need in flight at once," and for most services that number is much smaller than intuition suggests — see section 3.

2 · Configuring the pool

In Go, pgxpool (the pool wrapper around pgx, the driver this course has used since Lesson 5) exposes the knobs that decide the trade-off:

func newPool(ctx context.Context, dsn string) (*pgxpool.Pool, error) {
    cfg, err := pgxpool.ParseConfig(dsn)
    if err != nil {
        return nil, err
    }
    cfg.MaxConns = 20
    cfg.MinConns = 2
    cfg.MaxConnLifetime = 30 * time.Minute
    cfg.MaxConnIdleTime = 5 * time.Minute
    return pgxpool.NewWithConfig(ctx, cfg)
}

MaxConns caps how many connections this one instance will ever hold at once — if left unset, pgxpool defaults to the greater of 4 or the machine's CPU count, a number picked for a generic workload, not yours. MinConns keeps a small floor open even when idle, trading a little standing memory for skipping the connect handshake on the next burst of traffic. MaxConnLifetime and MaxConnIdleTime force periodically-recycled connections — useful when a load balancer or the database itself expects connections to rotate, and a cheap safety net against a connection that's quietly gone stale.

3 · Pool exhaustion: what running out actually looks like

Every connection in the pool is either idle (available to borrow) or acquired (currently in use). pool.Acquire(ctx) hands one out; the caller is responsible for giving it back:

func leaky(ctx context.Context, pool *pgxpool.Pool) error {
    conn, err := pool.Acquire(ctx)
    if err != nil {
        return err
    }
    var id int
    return conn.QueryRow(ctx, "SELECT 1").Scan(&id)
    // conn.Release() never runs — this slot is gone for good
}

func fixed(ctx context.Context, pool *pgxpool.Pool) error {
    conn, err := pool.Acquire(ctx)
    if err != nil {
        return err
    }
    defer conn.Release()
    var id int
    return conn.QueryRow(ctx, "SELECT 1").Scan(&id)
}

leaky never calls Release(), on either the success or the error path — that connection is permanently removed from the pool's available set until the process restarts. One leaked call is invisible. A thousand, from a handler that runs on every request, quietly drains MaxConns down to zero. The fix is the same discipline as Go's other acquire-then-release resources (a mutex's Unlock, a file's Close): defer conn.Release() immediately after a successful Acquire, unconditionally.

This is pool exhaustion, and it doesn't look like a pool problem from the outside. When every connection is acquired (leaked or genuinely busy), the next Acquire(ctx) call just blocks until one frees up or ctx's deadline fires — requests pile up and start timing out, which looks identical to a slow network or a downed database. CPU and memory graphs can look completely calm while this happens; the actual signal is pool.Stat(), which reports AcquiredConns(), IdleConns(), and MaxConns() — acquired sitting at max with idle at zero, under load that used to be fine, is the fingerprint of exhaustion.

4 · The multiplication problem: pool size × instance count

Lesson 8 established that a "stateless" server still runs as more than one instance in production. Each instance owns its own pool — a pool is never shared across processes. So the number that actually matters against Postgres's max_connections isn't one instance's MaxConns, it's MaxConns × instance count. Twenty connections per instance feels modest; twenty connections across fifteen instances is 300, well past a default max_connections of 100 (itself reduced further by connections Postgres reserves for superusers and replication) — the database starts rejecting new connections outright, and every instance sees it at once, right as traffic (and instance count, if autoscaling is involved) is highest.

Two real levers close that gap: size pools smaller than instinct suggests — most services need far fewer concurrent connections than requests-per-second, since each connection is only held for a query's duration, not a whole request — and, once instance count grows enough that per-instance pools alone can't fit under max_connections, put an external pooler like PgBouncer in front of Postgres, so every instance's pool talks to PgBouncer instead of the database directly, and PgBouncer multiplexes a much larger number of app-side connections onto a small, fixed number of real Postgres ones.

Q: "Your API starts timing out under load, but CPU and memory both look fine. What would you check?" A strong answer, out loud: "I'd suspect connection pool exhaustion before anything else, since it produces exactly this signature — requests hang waiting for a resource, but the machine itself looks idle. I'd check the pool's stats: acquired connections sitting at the max with idle at zero means every connection is either genuinely busy or leaked. If it's a leak, I'd look for an Acquire without a matching deferred Release on every code path, including errors. If it's not a leak, I'd check whether instance count times per-instance pool size has grown past the database's max_connections — that's easy to miss when autoscaling adds instances without anyone revisiting the pool size math." The signal: naming pool exhaustion specifically and the acquired-vs-idle diagnostic, not just "check the database."
