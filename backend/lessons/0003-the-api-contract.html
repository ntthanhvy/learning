The API contract

Resources, verbs, and who decides the status code — the other side of every fetch() you've ever written.

Lesson 2 ended with entities living in tables: customers, orders, order_items. Today's question: how does the outside world get at them? You already know half the answer from the consumer side — you've typed fetch("/api/orders/42") a thousand times. Today you sit on the other side and decide what that URL is, what the verbs mean, and what comes back when things go wrong. That set of decisions is the API contract.

Resources are nouns, not functions

The frontend instinct is to name endpoints after actions — /getOrder, /createOrder, /updateOrderStatus — one URL per function you wish you could call. Backend design flips it: expose resources (nouns), and let the HTTP verbs be the actions:

GET    /orders          -- the collection
POST   /orders          -- create one (server picks the id)
GET    /orders/42       -- one item
PATCH  /orders/42       -- change part of it
DELETE /orders/42       -- remove it
GET    /orders/42/items -- a nested collection, when it only makes sense inside its parent

Notice what happened: five "functions" collapsed into one noun. The entities you extracted in Lesson 2 usually are your resources — that's why schema design came first. When you catch yourself inventing /getX or /doY URLs, that's the same instinct Lesson 2 retrained: shaping the interface around one caller's convenience instead of around the things themselves.

Verbs come with promises

The verbs aren't decoration — each carries a promise the whole internet relies on:

VerbMeansThe promise
GETreadSafe: changes nothing, ever. Cacheable because of it.
POSTcreate / doNo promise — the one verb allowed to have side effects per call.
PUTreplace wholeIdempotent: replaying it lands in the same state.
PATCHchange partPartial update; idempotency depends on your design.
DELETEremoveIdempotent: deleting twice = deleted.

That word matters enough to be a whole interview topic: idempotent. You've met the problem from the frontend: the user double-clicks "Đặt hàng", two POSTs fire, two orders exist. The verb promises are why only POST needs special protection (dedup tokens, disabled buttons) — retrying a GET or DELETE was always safe. As the backend designer, you now own keeping those promises: a GET handler that mutates state breaks every cache and prefetcher between you and the user.

Status codes: who's at fault?

Your handler — the code — decides the status code, and the first digit is a blame assignment:

CodeSay it whenFrontend translation
200read/update worked, here's the bodyrender it
201POST created something (return it + its new URL)navigate to the new thing
204worked, nothing to say (typical for DELETE)just update local state
400request unreadable / fails validationshow the form errors
401 / 403who are you? / you specifically may notredirect to login / show "no access"
404that resource doesn't exist (your ErrNotFound!)show the empty state
409conflicts with current state (duplicate, stale edit)ask user to reload/rename
500we broke; the client did nothing wronggeneric "try again later"

The instinct to build: 4xx is information, 5xx is an alarm. A spike of 404s might be a bad link; a spike of 500s pages someone at 3am. This maps exactly to the error-handling layer you built in the Go course — errors.Is(err, ErrNotFound) at the top of the handler is precisely the code that chooses 404 over 500. One decides what happened, the other tells the world whose problem it is.

The error contract

Success bodies get all the design attention; error bodies get an afterthought — and every frontend dev has suffered the result: sometimes a string, sometimes HTML, sometimes {message}, sometimes {error: {msg}}. Pick ONE error contract and use it everywhere:

{
  "error": {
    "code": "order_not_found",      -- machine-readable, stable, for code to branch on
    "message": "Order 42 does not exist"  -- human-readable, for developers (not end users)
  }
}

The code field is the frontend's errors.Is — string comparison instead of sentinel values, same idea: something stable to branch on that survives message rewording.
