Method chaining & the ETL pipeline shape

Lesson 7 ended with a teaser: stringing everything so far — cleaning, grouping, ranking — into one readable pipeline instead of a wall of reassigned variables. Every exercise through Lesson 7 wrote clean = clean.assign(...), by_date = by_date.assign(...) — a fresh name for every step. Today: the same operations, written as one chain, and where that chain sits in the ETL/ELT picture.

1 · The problem with reassignment soup

Reassigning a variable at every step works, but it costs you two things: a growing pile of near-identical names (df, clean, by_date, ranked...) that all mean roughly the same table at different points, and the risk of accidentally reusing a stale one two cells later. pandas methods that transform a DataFrame return a new DataFrame rather than editing in place — which means every one of those calls is already chainable, you just haven't been chaining them.

2 · Method chaining: one expression, wrapped in parentheses

Wrap the whole expression in parentheses and drop one method call per line — Python doesn't need the usual backslash line-continuation inside parens, and the shape reads top to bottom like a recipe:

top_per_customer = (
    df
    .assign(amount=lambda d: pd.to_numeric(d["amount"], errors="coerce"))
    .dropna(subset=["amount", "order_date"])
    .assign(amount_rank=lambda d: d.groupby("customer")["amount"].rank(
        method="first", ascending=False))
    .query("amount_rank == 1")
)

That's Lessons 3 and 7's logic — coerce, drop the unparseable rows, rank per customer, keep rank 1 — as a single expression, no intermediate name required. Compare it to a SQL query built from a chain of WITH clauses (CTEs): each line is one transformation step feeding the next, except pandas doesn't make you name every step the way WITH cleaned AS (...) does.

3 · The one new piece of syntax: lambda d: ... inside assign()

Outside a chain, you can write clean["amount_rank"] = clean.groupby(...)... because clean already exists as a variable. Inside a chain, there is no variable yet for "the DataFrame as it looks right after dropna" — it only exists mid-expression. A lambda solves exactly that: .assign(amount_rank=lambda d: ...) passes the DataFrame at that point in the chain in as d, so the rank computation can reference it before it has any variable name at all. Read it as "whatever comes out of the line above, call it d for this one step."

4 · .pipe(): custom logic without breaking the chain

Not every step fits neatly in .assign(lambda d: ...) — sometimes the transformation is multi-line, or you want to reuse it across several chains. .pipe(func) calls func(current_dataframe) and keeps the result in the chain:

def add_amount_rank(d):
    return d.assign(
        amount_rank=d.groupby("customer")["amount"].rank(method="first", ascending=False)
    )

top_per_customer = (
    clean
    .pipe(add_amount_rank)
    .query("amount_rank == 1")
)

add_amount_rank is an ordinary, independently testable function — .pipe() is just the plumbing that lets it slot into a chain instead of forcing step = add_amount_rank(step) as its own reassigned line.

Trade-off, honestly: a long chain is easy to read but harder to debug — you can't set a breakpoint mid-expression the way you can inspect clean after a standalone reassignment. The common fix is temporary: comment out the tail of the chain and print what's left, or briefly split it back into named steps while debugging, then re-chain once it works.

5 · Where this fits: ETL vs ELT

This whole course has quietly been the ETL shape: read_csv is Extract, everything through Lesson 7 is Transform, and writing the result out (to_csv, to_sql) would be Load. A chain like today's makes that shape visible in the code itself — one pipeline, top to bottom. The alternative, ELT, loads the raw data first and transforms it afterward inside the warehouse, usually as SQL/dbt models — the industry's modern default for analytics pipelines, precisely because warehouses got cheap and fast enough that "load first, decide the transform later" beat "decide the transform up front." Pandas chains like today's still matter even in an ELT-first world: they're exactly what a one-off script, a scheduled job upstream of the warehouse, or an interview live-coding exercise still looks like.

Q: "Walk me through how you'd structure a pandas cleaning script." A strong answer, out loud: "I'd write it as one chain: read_csv for extract, then a sequence of assign/dropna/query calls for the transform, each one operating on the result of the last — so the code reads top to bottom like the pipeline it represents, instead of a pile of reassigned variables named clean, clean2, clean3. Inside the chain, assign(lambda d: ...) lets me reference the in-progress DataFrame before it has a name; if a step needs more than one line, I'd pull it into its own function and slot it in with .pipe() instead of breaking the chain. This is the same ETL shape SQL/dbt does as ELT in the warehouse — extract, transform, load — just expressed in pandas instead of SQL." The subtext: naming that this is deliberate pipeline structure, not just "chaining because it looks nice," is what interviewers are listening for.
