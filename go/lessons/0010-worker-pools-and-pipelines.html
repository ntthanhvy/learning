package link

import (
    "context"
    "sync"
)

var _ Store = (*MemStore)(nil)

type MemStore struct {
    mu    sync.RWMutex
    links map[string]Link
}

func NewMemStore() *MemStore {
    return &MemStore{links: make(map[string]Link)}
}

func (s *MemStore) Save(ctx context.Context, l Link) error {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.links[l.Code] = l
    return nil
}

func (s *MemStore) Get(ctx context.Context, code string) (Link, error) {
    s.mu.RLock()
    defer s.mu.RUnlock()
    l, ok := s.links[code]
    if !ok {
        return Link{}, ErrNotFound
    }
    return l, nil
}

New file link/counter.go:

package link

import "sync"

// ClickCounter is a concurrency-safe tally of redirects per code.
type ClickCounter struct {
    mu     sync.Mutex
    counts map[string]int
}

func NewClickCounter() *ClickCounter {
    return &ClickCounter{counts: make(map[string]int)}
}

func (c *ClickCounter) Inc(code string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.counts[code]++
}

func (c *ClickCounter) Count(code string) int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.counts[code]
}

func (c *ClickCounter) Snapshot() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    out := make(map[string]int, len(c.counts))
    for k, v := range c.counts {
        out[k] = v
    }
    return out
}

1 · The problem with "one goroutine per item"

Day 3's checkAll does this:

for _, l := range links {
    wg.Add(1)
    go func(l link.Link) {
        defer wg.Done()
        results <- checkOne(l)
    }(l)
}

For 3 links: fine. For 10,000 links (a real link-shortener's health sweep): 10,000 goroutines exist at once, each holding its own stack, all hammering whatever checkOne hits — a downstream server, a rate limiter, a database. You didn't mean to launch a thundering herd; the for loop did it for you. This is unbounded concurrency, and it's the most common reason a Go service that handles 10 requests/sec fine falls over at 10,000.

2 · The fix: a worker pool

A worker pool flips the shape: launch a fixed number of workers once, and let them pull from a shared channel of jobs. Ten links or ten million, the same 4 workers handle it — just for longer.

const maxWorkers = 4

func checkAll(links []link.Link, timeout time.Duration) map[string]bool {
    jobs := make(chan link.Link)      // unbuffered: a job is "handed off", not queued in bulk
    results := make(chan result)

    var wg sync.WaitGroup
    for w := 0; w < maxWorkers; w++ {   // TODO 1: launch maxWorkers long-lived workers
        wg.Add(1)                        //         each: defer wg.Done(), then `for l := range jobs`
        go func() {                     //         processing each l with checkOne and sending to results
            defer wg.Done()
            for l := range jobs {
                results <- checkOne(l)
            }
        }()
    }
    // ...feeder and closer below...
}

Notice what changed and what didn't. The result-collecting select loop from Day 3 — the one with the timeout — is untouched; it doesn't care whether 3 goroutines or 3,000,000 jobs feed results. What's new is the producing side: instead of one goroutine per link, a small fixed crew shares one jobs channel.

3 · Feeding the pool and knowing when it's done

Two more goroutines complete the picture — a feeder, and a closer:

// feeder: sends every link into jobs, then closes it. Closing jobs is the
// signal each worker's `for l := range jobs` uses to know there's no more work.
go func() {                        // TODO 2: for _, l := range links { jobs <- l }; close(jobs)
    for _, l := range links {
        jobs <- l
    }
    close(jobs)
}()

// closer: once every worker has drained jobs and returned, close results —
// same pattern as Day 3, just waiting on N workers instead of N per-link goroutines.
go func() {
    wg.Wait()
    close(results)
}()

Three goroutines, three jobs: workers process, one at a time each, forever until jobs closes; the feeder hands off work then signals "no more"; the closer waits for every worker to finish, then signals the same to whoever reads results. This is a pipeline: fan-out from the feeder to the workers, fan-in from the workers to results.

4 · Semaphore channels: bounding without restructuring

Sometimes a full worker-pool rewrite is more than you need — you already have Day 3's one-goroutine-per-item code and just want to cap how many run at once. A semaphore channel does exactly that, with a two-line change:

sem := make(chan struct{}, maxWorkers)   // capacity = how many may run at once

for _, l := range links {
    wg.Add(1)
    go func(l link.Link) {
        defer wg.Done()
        sem <- struct{}{}        // take a slot — blocks once maxWorkers are in flight
        defer func() { <-sem }()  // release it on the way out
        results <- checkOne(l)
    }(l)
}

struct{}{} is an empty struct: a signal with no payload, zero bytes. The goroutine count is still one-per-link — that part didn't change — but only maxWorkers of them can hold a semaphore slot simultaneously, so at most maxWorkers ever run checkOne concurrently. Use a worker pool when you're building the pipeline from scratch (today's practice); reach for a semaphore when you have existing one-goroutine-per-item code and just need a quick cap.

5 · Backpressure and draining safely

Backpressure is what an unbuffered jobs channel gives you for free: the feeder's jobs <- l blocks the instant all maxWorkers are busy, so the feeder itself slows to the pool's pace instead of piling up 10,000 pending links in memory. This is why the pool used an unbuffered channel above — a large buffer would let the feeder race ahead and build an unbounded backlog, defeating the whole point of bounding concurrency.

Draining matters at the other end: when the Day 3 select's timeout fires, the function returns immediately with partial results — but the workers and feeder are still running, potentially still blocked trying to send into channels nobody's reading anymore. For today's practice that's an acceptable, documented tradeoff (the goroutines exit once jobs/results stop being read and everything gets garbage collected at process exit); Day 11 (debugging concurrency) comes back to close this gap properly with context cancellation.

Step 2 — prove the bound is real. Add to checker_test.go (given): 20 links, 4 workers, 50ms each — that's ~250ms of real work if bounded, ~50ms if it's secretly still unbounded.

func TestCheckAllBoundsConcurrency(t *testing.T) {
    links := make([]link.Link, 20)
    for i := range links {
        links[i] = link.Link{Code: fmt.Sprintf("l%d", i), URL: "https://x.dev"}
    }
    start := time.Now()
    got := checkAll(links, 2*time.Second)
    elapsed := time.Since(start)
    if len(got) != 20 {
        t.Fatalf("checked %d, want 20", len(got))
    }
    if elapsed < 200*time.Millisecond {
        t.Fatalf("elapsed %v looks unbounded, not worker-pooled", elapsed)
    }
}
