Think in tables, not loops

One idea powers everything pandas does — and it's an idea you already use every day in SQL.

Where this sits in a pipeline

Data teams talk about pipelines in two shapes. In ETL, data is transformed before it lands in the warehouse; in ELT, raw data lands first and is transformed inside the warehouse (dbt: ETL vs ELT). Either way, the T is where the real work is — and when that work happens in Python, it happens in pandas. This whole activity — cleaning, reshaping, combining messy data until it's usable — is what job posts call data wrangling.

You already know the data model

pandas has one central object: the DataFrame. It is a table. Each column of it is a Series, and every value in a column shares one dtype — exactly like a typed column in Postgres. Under the hood each column is a NumPy array. So your SQL instincts transfer almost line for line (official SQL ↔ pandas comparison):

The one idea: vectorization

Here is the mental shift. Coming from application code, your reflex for "compute unit price for every order" is a loop. In pandas, operations are vectorized: you write the expression as if columns were single values, and pandas applies it to every row at once.

✗ App-code reflex (slow, un-idiomatic)
# row-by-row, in the Python interpreter
prices = []
for _, row in df.iterrows():
    prices.append(row["amount"] / row["qty"])
df["unit_price"] = prices
✓ Table thinking (fast, idiomatic)
# whole columns at once, in compiled C
df["unit_price"] = df["amount"] / df["qty"]

Filtering works the same way. df["amount"] > 100 doesn't return True or False — it compares the whole column at once and returns a boolean mask. Putting that mask inside df[...] keeps the rows where it's True. That's your WHERE clause: build a mask, index with it.

Same thinking as SQL. You never write WHERE as a loop over rows — you declare a condition and let the engine apply it to the whole table. Vectorization is that exact habit, moved into Python. When you catch yourself reaching for for over rows, ask: "how would I say this in SQL?" — the pandas spelling is usually one expression away (McKinney ch. 5).

Q: "Why is vectorized pandas so much faster than a Python loop?" — a classic warm-up question. A strong answer, out loud: "A DataFrame column is a NumPy array — contiguous memory with one dtype. A vectorized operation hands the whole array to compiled C code in one call. A Python loop instead pays interpreter overhead per row: unboxing each value into a Python object, dispatching each operation dynamically. Same work, but one path runs in C over packed memory and the other re-enters the interpreter millions of times." Bonus point that interviewers like: iterrows() in review code is a smell — the fix is almost always a column expression, a mask, or (later) a groupby.
