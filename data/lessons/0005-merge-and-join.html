Merge & join

Recap, in one line

Lesson 4 answered questions about rows sliced into groups within one table. Today's questions need a second table before "group" even means anything — same table instincts, one level wider.

1 · pd.merge() is JOIN, how= is the join type

A merge is pandas' JOIN, and the how= argument is exactly the join type — no new concept, only new spelling:

SQLpandas
SELECT * FROM orders JOIN customers ON orders.customer = customers.customerorders.merge(customers, on="customer", how="inner")
SELECT * FROM orders LEFT JOIN customers ON ...orders.merge(customers, on="customer", how="left")
SELECT * FROM orders RIGHT JOIN customers ON ...orders.merge(customers, on="customer", how="right")
SELECT * FROM orders FULL OUTER JOIN customers ON ...orders.merge(customers, on="customer", how="outer")

on="customer" is the join condition — both frames need a column with that name (use left_on=/right_on= if the key is spelled differently in each). inner (the default) keeps only rows whose key exists on both sides; left keeps every row from the left frame, filling missing right-side columns with NaN where nothing matched — the shape you reach for constantly, because "keep everything I started with, plus whatever matched" is the far more common real question than "only keep rows that matched."

2 · Finding the gap: indicator=True

The single most common real-world join question isn't "combine these tables" — it's "which rows on one side had no match on the other." indicator=True adds a _merge column naming exactly that, per row:

merged = customers.merge(orders, on="customer", how="left", indicator=True)
merged[merged["_merge"] == "left_only"]   # customers with NO matching order — never ordered

_merge takes one of three values: "both" (matched on both sides), "left_only" (only in the left frame), "right_only" (only in the right — only possible with how="outer", since how="left" drops unmatched right-side rows entirely). This is the pandas way to answer "customers who never ordered," "orders with no matching customer record," or any other one-sided-gap question — filter _merge, don't reach for a manual set-difference.

3 · Two gotchas that break the 1:1 mental model

Row multiplication. SQL warns about this too, but it's easy to forget in pandas specifically because nothing raises an error. If the join key isn't unique on one side — one customer, three orders — merging produces one row per match, not one row per input row. Merging a 3-row customers table against a 6-row orders table (where one customer alone has three orders) can come back with more rows than either table started with. Not a bug — that's what "one row per match" means — but a number that looks wrong at a glance if you expected row counts to be preserved.

Column name collisions. If both frames have a same-named column that isn't the join key (both have a date column, say), pandas doesn't error and doesn't silently pick one — it renames both automatically, date_x (from the left frame) and date_y (from the right), and leaves you to figure out which is which unless you pass suffixes=("_orders", "_customers") yourself to name them clearly.

SQL-habit trap: in SQL, an ambiguous column name is a compile-time error you must resolve with a table prefix before the query even runs. pandas resolves the same ambiguity silently, after the fact, with _x/_y — nothing stops you from shipping code that reads the wrong one.

Q: "Given an orders table and a customers table, find every customer who has never placed an order." A strong answer, out loud: "That's a left join from customers to orders on the customer key, with indicator=True, then filter for _merge equal to left_only — those are the customers with no matching row on the orders side. I wouldn't use how='inner' here, since inner join is exactly the rows I want to exclude." The subtext: naming indicator/_merge by name, and explicitly ruling out inner, reads as someone who has hit this exact gap-finding problem before, not someone guessing at merge syntax live.
