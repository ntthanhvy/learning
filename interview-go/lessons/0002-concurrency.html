Go Screening Kit · Part 2 · 5 questions · pick 2–3 per screen

Concurrency — the core differentiator

Concurrency is why companies choose Go, so this is the highest-signal section of the screen. The trap to avoid as interviewer: accepting textbook theory. Every question here should be pushed toward "tell me when this actually happened to you."

Budget: if you only have time to go deep on one section, make it this one. A candidate who is strong here and average elsewhere is usually a better Go hire than the reverse.

Question 5

"What's the difference between a goroutine and an OS thread? Roughly, how does Go schedule goroutines?"

Why ask: baseline mental-model check. They don't need scheduler internals — they need to know why spawning 100k goroutines is normal and 100k threads is not.

Answer guideline — a strong answer covers

  • Goroutines are runtime-managed, not OS-managed: tiny growable stacks (~KB vs ~MB), cheap to create, cheap to switch.
  • The runtime multiplexes many goroutines onto few OS threads (M:N). Naming G/M/P is a bonus, not a requirement — the multiplexing idea is the requirement.
  • GOMAXPROCS bounds how many goroutines run in parallel; concurrency vs parallelism distinction.
  • Blocking behavior: a goroutine blocked on a channel or network I/O just gets parked — it doesn't burn a thread. (Bonus: blocking syscalls/cgo do occupy a thread.)

Strong signals

  • "Cheap because small stacks + user-space scheduling," in their own words
  • Distinguishes concurrency from parallelism cleanly
  • Knows blocked-on-channel goroutines cost almost nothing

Weak / red flags

  • "Goroutines are just lightweight threads" with nothing behind it
  • Thinks each goroutine gets its own OS thread
  • Recites GMP letters but can't apply them ("what happens when a goroutine blocks on I/O?" → blank)

3/5 = multiplexing model correct. 5/5 = can reason about blocking, parallelism limits, and why goroutine-per-request is a sane server model.

Question 6

"When do you reach for a channel, and when for a mutex? Give me a concrete example of each from your own work."

Why ask: this is a judgment question disguised as a knowledge question. The "from your own work" clause is load-bearing — keep them there.

Answer guideline — a strong answer covers

  • Mutex: protecting long-lived shared state that many goroutines read/write in place — a counter, an in-memory cache, a config map. Small critical sections, cheap, direct.
  • Channel: transferring ownership of a piece of work between goroutines — pipelines, fan-out/fan-in, worker pools, signaling (done/close semantics).
  • The proverb "share memory by communicating" is a guideline, not a law — a strong candidate explicitly says a mutex-guarded counter is simpler and more correct than forcing it through a channel.
  • Bonus: sync.RWMutex when reads dominate (and that it's slower than Mutex under write-heavy load — measure); sync/atomic for single counters.

Strong signals

  • Frames it as ownership-transfer vs in-place protection
  • Gives a real, specific example of each (named system, named problem)
  • Pushes back on channel dogma

Weak / red flags

  • "Channels are more idiomatic, I always use channels"
  • Examples stay hypothetical after you ask for real ones
  • Doesn't mention defer mu.Unlock() or critical-section size when describing mutex use

Follow-up probe

"What can go wrong holding a mutex while making a database call?" (Good: every other goroutine needing that lock queues behind a slow I/O — you've serialized the server. Keep critical sections tiny; never do I/O under a lock.)

3/5 = correct split, generic examples. 5/5 = real examples + the I/O-under-lock hazard + "measure, don't assume" instinct.

Question 7

"What's a goroutine leak? How does one happen, and how would you find one in a running service?"

Why ask: leaks are the most common production concurrency failure in Go services. Theory-only candidates can define it; practitioners can find it.

Answer guideline — a strong answer covers

  • A goroutine blocked forever — usually on a channel send with no receiver, a receive with no sender, or waiting on a context that's never cancelled. Memory and scheduler load grow monotonically.
  • Classic causes: forgetting to drain a channel after an early return; select missing a ctx.Done() case; unbuffered channel send in a goroutine whose consumer errored out first; time.After in a hot loop (pre-1.23 timers).
  • Finding it: pprof — /debug/pprof/goroutine?debug=2 shows every goroutine's stack; leaks show up as thousands of identical stacks parked on the same line. Also runtime.NumGoroutine() as a metric on a dashboard, trending up and to the right.
  • Prevention: every goroutine you start should have a known exit path — context cancellation, closed channel, or bounded work. Bonus: goleak in tests, errgroup for structured lifetimes.

Strong signals

  • Names pprof unprompted, ideally the goroutine profile specifically
  • Gives a concrete leak pattern with the fix (add ctx.Done() case, buffer of 1, close on sender side)
  • "Every goroutine needs an exit plan" phrased in any form

Weak / red flags

  • Defines the leak but has no idea how to detect one
  • "The GC cleans up goroutines" (it doesn't — a blocked goroutine is a GC root)
  • Never used pprof at all in 3–4 years of Go

3/5 = correct definition + one real cause. 5/5 = detection story with pprof/metrics from an actual incident.

Question 8

"How does context.Context work, and how do you use it in an HTTP service?"

Why ask: context is ubiquitous in real Go codebases. A candidate who hand-waves here has probably not built services under load.

Answer guideline — a strong answer covers

  • A context carries cancellation, a deadline, and request-scoped values down a call tree; cancelling a parent cancels all children.
  • In HTTP: r.Context() is cancelled when the client disconnects — pass it into every DB query, outbound HTTP call, and goroutine spawned for the request, so abandoned requests stop consuming resources.
  • Mechanics: context.WithTimeout/WithCancel, always defer cancel() (leaks a timer/goroutine otherwise), check ctx.Err() / select on ctx.Done() in long loops.
  • Judgment: context.Value is for request-scoped metadata (trace ID, auth principal), not for passing business parameters or dependencies. First parameter, named ctx, never stored in a struct (with rare documented exceptions).

Strong signals

  • Connects client-disconnect → query cancellation as the practical payoff
  • Mentions defer cancel() and why
  • Has the context.Value restraint opinion

Weak / red flags

  • "It's for passing values around" as the primary use
  • Uses context.Background() inside handlers out of habit, severing cancellation
  • Doesn't know what happens to in-flight DB queries when a client disconnects

3/5 = cancellation + timeout mechanics. 5/5 = full propagation story plus the Value discipline.

Question 9

"What is a data race, exactly? How do you detect one, and what are the limits of that detection?"

Why ask: precise definition + tooling + limits is a three-step ladder; each step up is a year or two of real experience.

Answer guideline — a strong answer covers

  • Definition: two goroutines access the same memory concurrently, at least one is a write, with no synchronization ordering them. Result is undefined — not "sometimes wrong values" but genuinely undefined behavior (torn reads, vanished writes; concurrent map writes crash the runtime on purpose).
  • Detection: the race detector — go test -race, or a -race build in staging. Reports the two conflicting stacks.
  • Limits: it only catches races that actually execute during the run — it proves presence, never absence. It costs ~5–10× CPU/memory, so it's for tests and canaries, not usually full production.
  • Bonus: the memory-model point — without a synchronizing operation (mutex, channel op, atomic) there is no guarantee a write in one goroutine is ever visible to another; "it's just an int" is still a race.

Strong signals

  • Definition includes "at least one write" and "unsynchronized"
  • Knows -race only catches races that happen at runtime
  • Mentions running race-enabled builds in CI/staging as practice

Weak / red flags

  • Thinks races are only about maps, or only about "big" values
  • Believes a clean -race run proves the code race-free
  • "Reading while writing is fine if you don't mind stale data" — the memory model says otherwise

3/5 = definition + -race. 5/5 = limits of the detector + memory-model visibility argument.
