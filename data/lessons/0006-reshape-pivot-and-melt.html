Reshape: pivot & melt

Lesson 5 ended with a teaser: pivoting rows into columns and back, the pandas answer to a spreadsheet-style "wide" report built from a SQL-style "long" table. Every table you've touched Lessons 1–5 has been long format: one row per order. Today builds the other shape people actually ask for in a report — one row per customer, one column per date — and, just as often, undoes it.

Recap, in one line

Lesson 4's groupby answered "one number per group." Lesson 5's merge answered "combine two tables." Today answers a different question entirely: same data, same values, just laid out differently on the page.

1 · Long vs. wide, and why SQL never needed this word

A SQL query result is always long: every `GROUP BY customer` you've ever written comes back as one row per customer, full stop. A wide report — customers down the side, dates across the top, amounts in the grid — is a presentation choice, not a query result; SQL people build it with a wall of SUM(CASE WHEN order_date = '...' THEN amount END) columns, one per date, because SQL has no single keyword for "spread this column out." pandas does: pivot_table.

2 · pivot_table: index, columns, values, aggfunc

Four arguments answer four questions, and mapping them to the SQL wall of CASE WHENs makes each one obvious:

ArgumentAnswersSQL equivalent
index=what goes down the rows?GROUP BY column
columns=what gets spread across the top?one CASE WHEN col = '...' per distinct value
values=what fills each cell?the column inside SUM(...)
aggfunc=how to combine if more than one row lands in the same cell?the aggregate itself — SUM, COUNT, MEAN
wide = clean.pivot_table(
    index="customer", columns="order_date",
    values="amount", aggfunc="sum", fill_value=0,
)

fill_value=0 is the piece with no SQL equivalent to lean on: a customer with no order on a given date has no row to aggregate at all, so that cell would otherwise be NaN — "no data," not "zero." fill_value decides what goes there instead. 0 reads naturally for a money total; it would be the wrong choice for, say, an average price, where "no orders" and "average order was $0" are different facts worth keeping apart as NaN.

Gotcha: pivot_table's default dropna=True silently drops any row whose index or columns key is itself NaN — a customer with a missing order_date wouldn't get a row at all, not even a zero-filled one. That's a second, quieter reason (beyond Lesson 3's cleaning) to resolve missing keys before pivoting, not after.

3 · melt: wide back to long

The reverse operation, melt, un-spreads columns back into rows — needed constantly, because a wide report is a dead end for further analysis (try grouping by "which date column" when the date is a column name, not a value).

long = wide.reset_index().melt(
    id_vars="customer", var_name="order_date", value_name="amount",
)

reset_index() comes first for a reason worth noticing: pivot_table moved customer out of the columns and into the index (visible in section 2's output as its own row above the data, not a normal column). melt only reshapes ordinary columns, so customer has to move back into one before melting can see it — id_vars names the column(s) that stay put; every other column becomes two new columns: var_name (which original column a row came from) and value_name (that column's value).

SQL-habit trap: melting a pivoted table does not hand you back the exact original table — it hands you back every cell of the grid you built, fill_value padding included. Pivoting An/Binh's 4 real orders into a 2×4 grid and melting it back produces 8 rows, not 4; the 4 extra rows are the zero-filled combinations that never existed. Filtering them back out (amount != 0, or whatever fill_value was) is a manual step, not something melt knows to do for you.

Q: "You have a long orders table and need a report with one row per customer and one column per month. How do you build it?" A strong answer, out loud: "pivot_table with index=customer, columns=month, values=amount, aggfunc=sum, and fill_value=0 so customers with no orders that month show zero instead of NaN. If I need to reshape it back to long for further analysis — say, to join it with something else — reset_index and melt undo it, with id_vars=customer. The one thing I'd double check is whether melting gets me back the original rows exactly, or the padded grid — those aren't the same table if fill_value introduced zeros that weren't really there." The subtext: knowing pivot and melt are inverses, but not perfect inverses once fill_value is involved, is the detail that separates "knows the two function names" from "has actually hit this."
