1 · Three rules, no garbage collector

Go answers "when is this memory freed?" with a garbage collector that runs behind your back. Rust answers it with three compile-time rules — ownership:

  1. Every value has exactly one owner.
  2. Assigning or passing the value moves ownership.
  3. When the owner goes out of scope, the value is freed. Deterministic, immediate, no GC pause.

Rule 2 is the one that bites. Watch the memory:

let a = String::from("go");
//  STACK              HEAP
//  a ─ptr──────────►  "go"        one value, owner: a

let b = a;             // MOVE — the pointer is copied, ownership transfers
//  a ─✗ (dead)
//  b ─ptr──────────►  "go"        STILL one value, owner: b

println!("{}", a);    // ✗ compile error — a no longer owns anything

In Go, b := a leaves both usable and the GC figures out lifetimes later. In Rust the old binding is dead — so there's never a question of who frees the heap data: exactly one owner does, exactly once (Book ch. 4.1).

Exception: small fixed-size types — i32, bool, char, f64 — implement Copy: assignment duplicates them and both variables stay alive. That's why yesterday's E0382 message said "…because code has type String, which does not implement the Copy trait." Heap-owning types move; stack-trivial types copy.

2 · Feel it in your own code

Open linkshort-rs/src/main.rs. In the Command::Save arm, add one line right after the push:

Command::Save { code, url } => {
    links.push(Link { code, url });        // code and url MOVE into the new Link…
    println!("saved {}", code);            // …so this line uses a dead variable

cargo check:

error[E0382]: borrow of moved value: `code`
   |
   |  Command::Save { code, url } => {
   |                  ---- move occurs because `code` has type `String`,
   |                       which does not implement the `Copy` trait

Read it like a story: the match arm bound code, the struct literal moved it into the new Link, the println! arrived too late. Fix: print before the push (or borrow — next section). Keep the fix; this is part of today's practice.

And now yesterday's mystery dissolves: parse_command receives args as a borrow (it doesn't own the strings), but Command::Save needs to own its fields — so .clone() manufactures an owned copy. It wasn't ceremony; it was the type system telling the truth about who keeps what.

3 · Borrowing: use without taking

Moving everything everywhere would be unbearable — usually you just want to look at a value. That's a borrow:

fn print_link(links: &[Link], code: &str) { … }   // borrows: reads only

print_link(&links, &code);   // links and code still fully usable after the call

The borrow checker enforces one law:

Many readers or one writer — never both. Any number of &T at once, OR exactly one &mut T. This single rule kills data races at compile time — it's why yesterday's glossary called Rust concurrency "fearless".

Feel this one too. Inside your Get arm's loop, try pushing while iterating:

for l in &links {                         // shared borrow of links (reader)
    links.push(Link { … });               // &mut borrow (writer) — while readers exist!
error[E0502]: cannot borrow `links` as mutable because it is also borrowed as immutable
   |
   |  for l in &links {
   |           ------ immutable borrow occurs here

Go would compile this and let the slice reallocate under your feet. Rust names both borrows and refuses. Delete the line after reading the error — the point was meeting it.

4 · String vs &str

The pair you'll see in every signature from now on:

TypeWhat it isUse it for
StringOwned, growable, heap-allocated textstruct fields, building/keeping text
&strA borrowed view into text someone else ownsfunction parameters — accept broadly

Two automatic conversions make this ergonomic: &String coerces to &str, and &Vec<Link> coerces to &[Link] (a slice — a borrowed view into a sequence). So print_link(&links, &code) just works even though the parameters say slice and str. Rule of thumb, same spirit as Go's "accept interfaces": parameters borrow (&str, &[T]), fields own (String, Vec<T>).
