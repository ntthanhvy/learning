1 · Option<T>: no null, ever

Rust has no nil/null value that can secretly hide inside any type. "Maybe nothing" is instead a real, ordinary enum from the standard library:

enum Option<T> {
    Some(T),
    None,
}

Option<T> shows up the moment "not found" is a normal, expected outcome — like Vec::iter().find(...), which returns Option<&T>. You handle it with the match you already know, or a shortcut when you only care about one branch:

if let Some(l) = links.iter().find(|l| l.code == code) {
    println!("{}", l.describe());
} else {
    println!("not found");
}

if let is a one-armed match — reach for it when you only need the "yes" case and nothing happens on "no".

2 · Result<T, E>: your Go instincts, enforced

You already think this way — Go's (Link, error) return and ErrNotFound sentinel are this idea. Rust just makes it one generic type instead of a convention:

enum Result<T, E> {
    Ok(T),
    Err(E),
}

The difference that matters: Result<T, E> is marked #[must_use] — ignore one and the compiler warns you, in a way Go's if err != nil convention can't enforce (nothing stops a Go dev from skipping the check). And E is a real type parameter: instead of one shared error interface, you write a small enum naming exactly your failure modes:

use thiserror::Error;

// #[derive(Error)] writes Display + std::error::Error for you — you only
// write the shape and the message. Same job as Go's ErrNotFound, but the
// compiler knows the TYPE, not just a sentinel value it has to trust you compare right.
#[derive(Debug, Error)]
enum LinkError {
    #[error("no link with code '{0}'")]
    NotFound(String),
}

thiserror is the standard pick when you're naming specific error shapes — the library-code side of Rust error handling.

3 · ?: Go's if err != nil in one character

Every layer in yesterday's Go project wrapped and returned on error — that's exactly what ? automates:

let l = find_link(&links, &code)?;   // Ok(l) -> l, continue. Err(e) -> return Err(e) from THIS function, now.

Side by side:

GoRust
l, err := store.Get(ctx, code)
if err != nil { return err }		let l = store.get(code)?;

Same idea, same "handle once at the top" discipline from your Go Day 2 — ? just removes the four lines of ceremony every layer used to repeat. It only works in a function that itself returns Result (or Option, for ? on an Option) — which is why today's practice changes fn main()'s signature.

4 · anyhow vs thiserror: which one, when

CrateUse it forTrade-off
thiserrorLibrary code: callers need to match on specific variantsYou name every variant up front
anyhowApplication code (like main): "something failed, show it and stop"One opaque anyhow::Error holds any error; convenient, not match-able

anyhow is why fn main() -> anyhow::Result<()> can use ? on any error type without you writing a conversion — and if main returns Err, the process prints Error: {the message} and exits non-zero, automatically. Today: thiserror at the library boundary (LinkError), anyhow at main — the two almost always travel together.

#[test]
fn find_link_returns_err_for_missing_code() {
    let links = vec![Link { code: "go".to_string(), url: "https://go.dev".to_string() }];
    let err = find_link(&links, "missing").unwrap_err();
    assert_eq!(err.to_string(), "no link with code 'missing'");
}

#[test]
fn find_link_finds_existing_code() {
    let links = vec![Link { code: "go".to_string(), url: "https://go.dev".to_string() }];
    assert_eq!(find_link(&links, "go").unwrap().url, "https://go.dev");
}
