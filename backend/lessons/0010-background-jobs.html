Background jobs: work a handler shouldn't do itself

Lesson 9 closed with a teaser: some work shouldn't run inside the request/response cycle at all. Sending a receipt email, resizing an uploaded image, generating a PDF report — none of that needs to finish before the client gets a response. Today: how a handler hands that work off instead of doing it inline.

1 · Why "just do it in the handler" breaks down

The naive version is tempting: the handler receives the request, does the slow thing right there — call the email provider, resize the image, write the report — and only then responds. Three problems compound: the client sits waiting for work that has nothing to do with what they actually need back; the connection (and, depending on the server, a whole goroutine/thread) is held open the entire time; and if the process crashes or restarts mid-way, the work is simply lost with no record it was ever supposed to happen.

A background job is exactly that work, deliberately pulled out of the request path. The handler's job changes from "do the work" to "make sure the work is remembered," then respond immediately.

2 · The pattern: enqueue now, process later

The handler writes a small record describing the work — not the result, just "do this" — to a durable place, and returns right away. A separate, independent process reads that place continuously and does the actual work. That durable place is a job queue; writing a job into it is enqueueing it. The separate process that reads jobs out and executes them is a worker.

// handler: enqueue and return immediately — no email sending here
func createOrder(w http.ResponseWriter, r *http.Request) {
    order, err := saveOrder(r.Context(), parseOrder(r))
    if err != nil {
        writeError(w, err)
        return
    }
    // durable row in a "jobs" table — survives even if this process dies next
    enqueueJob(r.Context(), "send_receipt_email", order.ID)
    w.WriteHeader(http.StatusAccepted) // 202: accepted, not necessarily done yet
}

// worker: a totally separate process, polling the same table
func runWorker(ctx context.Context) {
    for {
        job, ok := dequeueJob(ctx) // claim one pending row
        if ok {
            processJob(ctx, job) // the actual slow work happens HERE, off the request path
        }
        time.Sleep(time.Second)
    }
}

The response code matters: 202 Accepted tells the client "I've recorded this, it's happening, but not necessarily done" — a deliberately different promise than 200 OK's "done, here's the result."

Frontend-habit trap: this is the same move as pushing expensive work off a browser's main thread into a Web Worker so the UI doesn't jank — the request thread is your "main thread" here, and a background job is the backend's Web Worker. Same instinct, same reason: don't block the thing the user is actually waiting on with work they aren't.

3 · Jobs get retried — so they must be safe to run twice

Workers crash too — mid-job, after the email actually sent but before the queue is told the job is done. Most systems resolve that ambiguity by retrying: better to risk running a job again than to silently drop it. That guarantee has a name: at-least-once delivery: a job might run more than once, but it is never silently dropped.

That single fact drives the most important design rule for job handlers: they must be safe to run twice. Lesson 3's idempotent is the same idea, applied here instead of to an HTTP endpoint. "Send an email" isn't naturally idempotent — sending it twice means two emails — so the job checks first ("has a receipt already been sent for this order?") before acting, the same way a payment job would check for an existing charge before charging again.

Q: "A signup handler needs to send a welcome email. How would you build that?" A strong answer, out loud: "I wouldn't call the email provider inside the handler — that ties the response time to a third-party API and loses the work if the process dies mid-request. I'd write a job row (or push to a queue) describing 'send welcome email for user X' and return 202 immediately; a separate worker process dequeues and sends it. Because queues are usually at-least-once — a worker can crash after sending but before marking the job done — I'd make the job idempotent, checking a welcome_email_sent_at column before sending, so a retry doesn't double-send." The subtext: naming that retries are expected, not an edge case, and designing the handler to tolerate them, is what separates "used a queue" from "actually understands why."
