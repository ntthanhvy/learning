package link

import (
    "context"
    "errors"

    "github.com/jackc/pgx/v5"
    "github.com/jackc/pgx/v5/pgxpool"
)

var _ Store = (*PostgresStore)(nil)

type PostgresStore struct {
    pool *pgxpool.Pool
}

func NewPostgresStore(ctx context.Context, connString string) (*PostgresStore, error) {
    pool, err := pgxpool.New(ctx, connString)
    if err != nil {
        return nil, err
    }
    return &PostgresStore{pool: pool}, nil
}

func (s *PostgresStore) Save(ctx context.Context, l Link) error {
    _, err := s.pool.Exec(ctx,
        `INSERT INTO links (code, url) VALUES ($1, $2)
         ON CONFLICT (code) DO UPDATE SET url = EXCLUDED.url`,
        l.Code, l.URL,
    )
    return err
}

func (s *PostgresStore) Get(ctx context.Context, code string) (Link, error) {
    var l Link
    err := s.pool.QueryRow(ctx,
        `SELECT code, url, created_at FROM links WHERE code = $1`, code,
    ).Scan(&l.Code, &l.URL, &l.CreatedAt)
    if errors.Is(err, pgx.ErrNoRows) {
        return Link{}, ErrNotFound
    }
    if err != nil {
        return Link{}, err
    }
    return l, nil
}

1 · pgx v5: three ways to run a query

*pgxpool.Pool — a connection pool, not a single connection; Save and Get already use two of its three query methods. Exec runs a statement that returns no rows (INSERT/UPDATE/DELETE) and gives you back a row count. QueryRow expects exactly one row back and .Scans it directly into variables — Get above. Query is for anything returning zero-to-many rows: it hands back a pgx.Rows you iterate, one column set per call to .Scan. That manual iteration — for rows.Next() { rows.Scan(&a, &b) }, checking rows.Err() after the loop — is correct but repetitive: the same five lines, in every function, for every query shape. Section 2 is the fix.

2 · Generics in the wild: pgx.CollectRows

A generic function is written once, parameterized over a type T, and works for any T that fits — instead of a hand-copied Scan loop per query. pgx.CollectRows[T any](rows pgx.Rows, fn RowToFunc[T]) ([]T, error) is exactly that: it drives the Next/Scan/Err loop internally and returns a plain []T. The fn argument is itself generic — pgx.RowToStructByName[T] — and does the column-to-field matching for you:

type LinkStats struct {
    Code   string
    Clicks int64
}

func ClicksPerLink(ctx context.Context, pool *pgxpool.Pool) ([]LinkStats, error) {
    rows, err := pool.Query(ctx, `
        SELECT link_code AS code, count(*) AS clicks
        FROM clicks
        GROUP BY link_code
        ORDER BY clicks DESC`)
    if err != nil {
        return nil, err
    }
    return pgx.CollectRows(rows, pgx.RowToStructByName[LinkStats])
}

RowToStructByName matches a returned column to a struct field by name, case-insensitively, ignoring underscores — clicked_at matches a field named ClickedAt with no tag needed; count(*) AS clicks is aliased to line up with the Clicks field. This is the aggregate query the Day 5 index earns its keep on: how many clicks per link, most-clicked first. Clicks is int64, not int — Postgres's count(*) returns bigint, and matching the width avoids a silent truncation on a link popular enough to overflow a 32-bit count.

Compare this to Day 3's goroutines or Day 4's context.Context: those were generic-shaped problems Go solved with interfaces and channels instead. This is genuinely the case generics were added for — one Scan loop, reused for LinkStats, Click, or any future row shape, with the compiler (not a reflection-heavy ORM) checking that T lines up.

3 · Keyset pagination: never OFFSET

Offset pagination — LIMIT 20 OFFSET 4000 — asks Postgres to walk and discard four thousand rows on every request for page 201, then throw them away. Keyset pagination instead filters on the last row you already saw — no counting, no discarding, and it's exactly the query shape the Day 5 composite index was built to serve:

4 · Transactions: when one statement isn't enough

A transaction matters the moment more than one statement has to succeed or fail together. A single INSERT ... ON CONFLICT is already atomic on its own — no transaction needed. But seeding demo data needs two statements — insert the link, then insert its clicks — and they must not be allowed to half-succeed: a crash between them would either leave clicks pointing at a link that was never saved (impossible here, the foreign key blocks it and the second statement would simply fail) or leave a link with no clicks when the caller expected both. Wrapping both in one transaction makes "both or neither" true regardless:

func SeedDemoClicks(ctx context.Context, pool *pgxpool.Pool, code, url string, n int) error {
    tx, err := pool.Begin(ctx)
    if err != nil {
        return err

    defer tx.Rollback(ctx) // no-op once Commit succeeds

defer tx.Rollback(ctx) right after a successful Begin is the idiomatic pgx shape: if the function returns early on any error, the deferred rollback undoes whatever partial work happened; if Commit already ran, the deferred Rollback is a documented no-op. Notice what did not need a transaction: ClicksPerLink and RecentClicks are single SELECTs, already atomic by themselves. Reach for a transaction when a caller-visible "unit of work" spans more than one statement — not as a default wrapper around every query.

5 · Reading EXPLAIN: proving the index earns its keep

An EXPLAIN shows what Postgres actually decided to do, not what you hope it did. Run the exact shape from section 3 against your Neon database, with and without the Day 5 index (you built it in the migration — this just confirms it's being used):

psql "$LINKSHORT_DB_URL" -c "EXPLAIN ANALYZE
  SELECT clicked_at FROM clicks
  WHERE link_code = 'go' AND clicked_at < now()
  ORDER BY clicked_at DESC LIMIT 20;"

Look for Index Scan using clicks_link_code_clicked_at_idx in the output — that's the composite index from Day 5 doing exactly the job section 4's docstring said it would. If you instead see Seq Scan on clicks, the planner decided a full table scan was cheaper — normal on a tiny seeded table (Postgres estimates cost from row counts; an index lookup plus a table fetch can lose to "just read the whole table" when the table is a few dozen rows). SeedDemoClicks from section 4 exists partly to make this demo repeatable: seed a few hundred clicks for one code, a handful for others, and re-run EXPLAIN ANALYZE — the skew is enough to tip the planner toward the index.
