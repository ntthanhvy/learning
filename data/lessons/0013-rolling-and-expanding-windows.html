1 · Back to Lesson 7's window family: two tools, not yet covered

Lesson 7 taught two "window" tools: cumsum() (running total over the whole history so far) and rank() (position within a group). Both are members of a wider family pandas calls window operations — pandas' docs page for it was already linked as a "go deeper" resource back then, but only cumsum and rank themselves got taught. Today fills in the two most interview-common members left: rolling window (a fixed-size sliding window — "the last N rows") and expanding window (a growing window — "everything so far," which is exactly what cumsum() already does, generalized to any aggregation, not just sum).

2 · .rolling(window=N).mean(): a fixed-size moving window

rolling(window=N) looks back at only the last N rows (including the current one) and slides that window forward one row at a time. A "3-order moving average" or "2-order moving average" is exactly this. On the clean order fixture, sorted by date within customer — rolling() trusts row order with no ORDER BY of its own, the same caveat Lesson 11's pct_change() carried:

ordered = clean.sort_values(["customer", "order_date"])
ordered["amount_roll2_mean"] = ordered.groupby("customer")["amount"].rolling(window=2).mean().values
# An:   120.0 -> NaN,  42.0 -> 81.0    ((120.0 + 42.0) / 2)
# Binh:  35.5 -> NaN, 180.0 -> 107.75  ((35.5 + 180.0) / 2)

The first window - 1 rows of each group come back NaN — with window=2, that's just the very first order per customer, since there's only 1 row available and the window needs 2. That's the same "genuinely doesn't exist yet" reasoning as Lessons 3 and 11's NaN rules, not a bug: a 2-order moving average has nothing to average until a second order shows up.

3 · .expanding().mean(): a window that only grows

expanding() takes the same idea and drops the fixed size — instead of "the last N rows," it's "every row from the start of the group through the current one," growing by one row each step. Swap rolling(window=2) for expanding() and nothing else changes:

ordered["amount_expand_mean"] = ordered.groupby("customer")["amount"].expanding().mean().values
# An:   120.0 -> 120.0,  42.0 -> 81.0     (mean of just row 1, then mean of rows 1-2)
# Binh:  35.5 -> 35.5,  180.0 -> 107.75   (mean of just row 1, then mean of rows 1-2)

No leading NaN here — the first row's expanding window is just itself, a valid 1-row mean. Swap .mean() for .sum() and expanding().sum() becomes identical to cumsum(): both accumulate everything seen so far. cumsum() is really just expanding().sum() with a shorter name — expanding() is the general form that works with any aggregation (mean, max, std, …), not only sum. rolling(), by contrast, forgets anything older than N rows — that's the whole difference: fixed-size sliding window vs. ever-growing window.

4 · The SQL bridge

Both map directly onto SQL window frames, same as Lesson 7's RANK()/SUM() OVER mapping:

-- rolling(window=2).mean() ==
AVG(amount) OVER (
  PARTITION BY customer ORDER BY order_date
  ROWS BETWEEN 1 PRECEDING AND CURRENT ROW
)

-- expanding().mean() ==
AVG(amount) OVER (
  PARTITION BY customer ORDER BY order_date
  ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
)

The frame clause is the whole story: ROWS BETWEEN 1 PRECEDING AND CURRENT ROW is a 2-row fixed frame — rolling(window=2). ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW is "everything before, no limit" — expanding(). PARTITION BY is groupby(); ORDER BY is the sort_values() that must come first in pandas since neither method takes an ordering argument of its own.

Q: "Compute a 3-day moving average of daily sales per store, then separately a running average since the store opened — how do these differ in pandas?" A strong answer, out loud: "Same shape both times — sort by date within store first, since neither method has its own ORDER BY, then groupby('store')['sales']. For the moving average it's .rolling(window=3).mean() — a fixed 3-row frame that slides forward, forgetting anything older. For the running average it's .expanding().mean() — the frame never forgets, it just keeps growing from day one. The first two rows of the rolling version come back NaN, since there aren't 3 days yet to average; the expanding version has no such gap, a 1-row mean is still a valid mean. In SQL both are just different ROWS BETWEEN frames on the same AVG() OVER (PARTITION BY store ORDER BY date) — rolling is '1 PRECEDING AND CURRENT ROW' with N-1 preceding, expanding is 'UNBOUNDED PRECEDING AND CURRENT ROW'." The signal: connecting expanding().sum() back to cumsum() unprompted shows the window-function family is understood as one idea with several frame shapes, not four unrelated method names to memorize.
