SQL injection & input validation

Lesson 12 covered authorization (is this user allowed to touch this row) and Lesson 7 covered information disclosure (what a response is allowed to reveal). This lesson covers the piece that sits before both: can untrusted input change what your code even does. It's MISSION.md's last uncovered security item — the OWASP-flavored one.

1 · A query that trusts its input

Every lesson so far has built queries assuming the values plugged into them are just data. Here's a login lookup written the way string formatting makes tempting:

func findUserVulnerable(ctx context.Context, db DB, username string) Row {
    query := fmt.Sprintf("SELECT id, email FROM users WHERE username = '%s'", username)
    return db.QueryRow(ctx, query)
}

This works for every normal username. It also works, disastrously, for a username of ' OR '1'='1 — the resulting query text becomes WHERE username = '' OR '1'='1', a condition that's true for every row, returning the first user in the table (often an admin) with no matching credentials at all. A username of x'; DROP TABLE users; -- is worse: it closes the string early, appends a second statement, and comments out whatever the original query intended to say next. Neither payload is exotic — both are the literal top two examples in every SQL injection tutorial that exists, and the code above is vulnerable to both.

2 · Why "no real user would type that" is not a defense

The input never comes from a real user typing carefully in a browser form. It comes from whatever can reach the handler: a scripted HTTP request, a malformed field in a JSON body, a query parameter edited by hand. fmt.Sprintf has no idea it's building SQL — it just interpolates a string, exactly as asked, and the database has no way to tell "text the developer wrote" apart from "text a user's input happened to produce" once they're concatenated into one string. That's the entire bug, in one sentence: the query's code and the query's data end up in the same channel, so anything that looks like SQL syntax inside the data is read as SQL syntax.

Frontend-habit trap: validating input in the browser (a form field that rejects quotes) stops nothing — an attacker who skips the browser entirely and calls the API directly never runs that JavaScript. Client-side validation is a UX nicety; it is not a security boundary. The only place this bug can actually be fixed is server-side, at the query itself.

3 · The fix: keep code and data in separate channels

The fix already showed up quietly in every query snippet since Lesson 1 — this lesson just names why it's non-negotiable. A parameterized query sends the SQL text and the values as two separate messages instead of one merged string:

func findUser(ctx context.Context, db DB, username string) Row {
    return db.QueryRow(ctx, "SELECT id, email FROM users WHERE username = $1", username)
}

Here $1 is a placeholder, not a format slot — pgx sends the literal text SELECT id, email FROM users WHERE username = $1 to Postgres first, and username separately as a value to bind into it. Postgres parses the query's shape before it ever sees the value, so ' OR '1'='1 arrives as a single string being compared for equality — not as SQL syntax — and simply matches no username, which is exactly the correct behavior for a bogus login attempt. This isn't "sanitizing" or "escaping" the input; it's never letting the value enter the code channel in the first place. Every $1/$2 placeholder used in Lessons 6, 11, and 15's snippets was already doing this — the rule made explicit here is: never fmt.Sprintf (or +) a value into a SQL string, ever, even for values that "can't" contain quotes, because the cost of being wrong once is total.

4 · Input validation is a different, complementary layer

Parameterized queries close the injection hole completely — but a query can still do something wrong with input that's syntactically safe. input validation is a separate question asked before the query even runs: is this string actually an email, is this page size a positive integer under some cap, is this enum value one of the three the handler understands. The two defenses solve different problems and neither substitutes for the other:

SQL injection defenseInput validation
Question it answersCan this value change what the query does?Is this value the shape the handler expects?
Where it livesInside the query — parameterized placeholdersBefore the query — handler-level checks
What it stops' OR '1'='1 changing query logicA negative page size, a 10MB "name" field, an unknown enum string
Skipping it meansAttacker rewrites your SQLGarbage or oversized data reaches business logic / storage

The most reliable validation strategy is an allowlist: define exactly what's acceptable (an enum of three status strings, an integer range 1–100) and reject everything outside it — rather than a blocklist that tries to guess every bad value in advance (block quotes, block semicolons, block DROP...) and reliably misses one, the same losing game as trying to string-match your way out of injection instead of parameterizing.

Q: "How would you explain SQL injection to a junior who just wrote fmt.Sprintf into a query?" A strong answer, out loud: "I'd show them that the database can't tell the difference between SQL the developer wrote and SQL that showed up inside a user's input, once both are concatenated into one string — a username like ' OR '1'='1 turns a WHERE clause that's supposed to match one row into one that matches every row. The fix isn't escaping quotes or blocking suspicious characters — it's parameterized queries, placeholders like $1 that make the driver send the query text and the values as two separate messages, so a value physically cannot be interpreted as syntax no matter what it contains. I'd also point out this is a different problem from input validation — checking that a page size is a positive number is about correctness, not injection, and you need both." The signal: naming parameterization specifically, not "sanitize the input" — sanitizing is the blocklist approach that keeps losing.
