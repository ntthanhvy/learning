Rate limiting & backpressure: protecting your service from itself

Lesson 10 closed with a teaser: what protects your own service — and its queues — from being overwhelmed faster than it can keep up? A worker can only drain a queue so fast; a handler can only hold open so many connections. Today: the two mechanisms that keep "too much traffic" from turning into "the whole service falls over."

1 · Unbounded acceptance is the actual bug

A naive server accepts every request that arrives and does its best. That "best effort" model works fine until load exceeds capacity — then requests queue up in memory, database connections run out, response times climb, and eventually the process runs out of memory or every request times out anyway, including ones from well-behaved clients who did nothing wrong. The failure isn't "too much traffic" itself; it's a server with no mechanism to say no before that happens.

Two related tools fix this from different ends. Rate limiting caps how much a single client can demand. backpressure is the more general idea: any point in a system that's falling behind pushes that fact back upstream, instead of absorbing unlimited work and hoping.

2 · Rate limiting: the token bucket

The standard algorithm is the token bucket: a bucket holds up to N tokens and refills at a steady rate (say, 10/second). Every request must take one token to proceed; an empty bucket means the request gets rejected. This allows short bursts (spend the whole bucket at once) while still enforcing a long-run average rate — friendlier than a hard "N requests per fixed minute" cutoff, which punishes a client unlucky enough to burst right at a window boundary.

type bucket struct {
    mu           sync.Mutex
    tokens       float64
    capacity     float64
    refillPerSec float64
    last         time.Time
}

func (b *bucket) allow() bool {
    b.mu.Lock()
    defer b.mu.Unlock()

    now := time.Now()
    elapsed := now.Sub(b.last).Seconds()
    b.tokens = math.Min(b.capacity, b.tokens+elapsed*b.refillPerSec) // refill since last check
    b.last = now

    if b.tokens < 1 {
        return false // bucket empty — reject
    }
    b.tokens--
    return true
}

// middleware: one bucket per API key, wraps any handler
func rateLimit(next http.Handler, buckets *sync.Map) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        key := r.Header.Get("X-API-Key")
        b, _ := buckets.LoadOrStore(key, &bucket{tokens: 20, capacity: 20, refillPerSec: 10, last: time.Now()})
        if !b.(*bucket).allow() {
            w.Header().Set("Retry-After", "1")
            w.WriteHeader(http.StatusTooManyRequests) // 429
            return
        }
        next.ServeHTTP(w, r)
    })
}

The response code matters as much as the algorithm: 429 Too Many Requests tells a well-behaved client exactly what happened and, via Retry-After, when it's worth trying again — a client SDK can back off automatically instead of hammering harder.

3 · Backpressure: pushing "slow down" further upstream

Rate limiting protects a service from any one client. Backpressure protects a service from itself, at every internal handoff — the clearest case is Lesson 10's job queue. A worker that processes 50 jobs/second can't magically drain a queue that's filling at 200 jobs/second; if the queue has no limit, it grows until the process runs out of memory. A bounded queue rejects (or the producer checks queue depth before enqueueing and returns 503 Service Unavailable) once it's full, instead of accepting work it has no way to ever finish.

Frontend-habit trap: this is the same shape as a frontend debounce on a search box: the UI protects itself from firing a network request on every keystroke by dropping the excess and only acting on the settled result. A token bucket and a bounded queue are doing the same thing server-side — refusing to do more work than the system can actually sustain, rather than accepting everything and falling behind.

Q: "How would you protect an API from a single client hammering it with requests?" A strong answer, out loud: "I'd rate-limit per client — typically a token bucket keyed by API key or user id, since it allows short bursts while still capping the long-run average, unlike a hard fixed-window count. Once the bucket's empty I'd return 429 with a Retry-After header so well-behaved clients back off automatically instead of retrying immediately. And I'd apply the same idea internally, not just at the edge — if a downstream queue or worker pool is the bottleneck, it needs its own backpressure, like a bounded queue that returns 503 once full, so one slow client or client group doesn't fill memory or starve everyone else." The subtext: naming that rate limiting is about the client-facing edge and backpressure is the same idea applied to every internal hop is what shows systems thinking, not just "I know what 429 means."
