1 · Modules: main.rs stops being one file

Go's file layout is a convenience, not a boundary: every .go file in a directory shares the same package, and a capitalized name is visible to the whole package regardless of which file it's in. Rust's module system draws the boundary on purpose: mod link; in main.rs tells the compiler "there's a module named link, go find it at src/link.rs," and everything inside is private to that module unless explicitly marked otherwise.

That "otherwise" is pub: without it, an item is only visible inside its own module (and modules nested under it). Rust's default is private; Go's capitalization convention defaults to "exported unless lowercase." Same underlying idea — draw a line between a package's public surface and its internals — opposite defaults, and Rust's is enforced per-item, not per-file.

// src/link.rs
pub struct Link {
    pub code: String,
    pub url: String,
}

impl Link {
    pub fn describe(&self) -> String {
        format!("{} -> {}", self.code, self.url)
    }
}

Every one of those pubs matters independently: dropping it from code alone would still let main.rs see the Link type and call .describe(), but not read or construct l.code directly. Fields, structs, functions, and modules each carry their own visibility — nothing is transitively exported just because its container is.

2 · HashMap: the lookup Vec was never good at

A Vec<Link> answers "give me link number 3" instantly, but "give me the link whose code is go" only by scanning every element — exactly the found-flag loop (or Day 4's .find()) that's been sitting in the Get arm since Day 1. A HashMap is the fix, and it's the same data structure as Go's built-in map — a hash table, average O(1) insert and lookup by key, unspecified iteration order:

use std::collections::HashMap;

let mut links: HashMap<String, Link> = HashMap::new();
links.insert("go".to_string(), Link { code: "go".to_string(), url: "https://go.dev".to_string() });

match links.get("go") {
    Some(l) => println!("{}", l.describe()),
    None => println!("not found"),
}

.get(&key) already returns Option<&Link> — exactly the shape Day 4's .find() refactor produced, but for a different reason: .find() still scans every element checking a predicate, while .get() jumps straight to the bucket a hashed key lives in. Same Option-shaped answer, one is O(n), the other is O(1). Use Vec when order matters or you're keeping everything and walking it end to end (Day 4's print_all); reach for HashMap the moment "find the one with this key" is the actual question, which for linkshort's code lookup, it always was.

One rough edge, unrelated to Go's map: .insert(k, v) on a key that already exists silently overwrites the old value and returns it as Option<V> — no error, no panic. That's actually the right behavior for Save (re-saving a code should update its URL), but worth knowing by name for when it isn't: it's a decision, not a default to trip over.

3 · serde: structs in and out of JSON

serde is the standard way a Rust struct becomes JSON and back — a compile-time derive macro, not a runtime reflection library the way many other languages' JSON libraries work:

use serde::{Deserialize, Serialize};

#[derive(Serialize, Deserialize)]
pub struct Link {
    pub code: String,
    pub url: String,
}

That one attribute is the entire cost: serde_json::to_string_pretty(&link) now works, producing {"code": "go", "url": "https://go.dev"} with field names taken straight from the struct — no schema written separately, no reflection at runtime. The derive expands at compile time into real, checked Rust code; a typo'd field name is a compile error, not a JSON key silently missing at runtime.
