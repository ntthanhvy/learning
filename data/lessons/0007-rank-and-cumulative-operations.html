Rank & cumulative operations

Lesson 6 ended with a teaser: rank and cumulative operations, answering "top N per group" and running totals without a single Python loop. You already know the SQL side of this — window basics were called out as comfortable ground in this course's baseline. Today is the pandas spelling of exactly that: RANK()/ROW_NUMBER() OVER (PARTITION BY ... ORDER BY ...) and SUM() OVER (...), no new concept, new syntax only.

Recap, in one line

Lesson 4's groupby answered "one number per group" (a single aggregate, rows collapse). Lesson 6's reshape answered "same data, different layout." Today's question keeps every row but adds a per-row answer computed within its group — a rank, or a running total — which is exactly what a SQL window function does: compute per-group, but don't collapse the rows.

1 · rank(): the pandas RANK() OVER (PARTITION BY ...)

rank() assigns each value its position within an ordering. Alone, on a whole column, it ranks globally — chained after groupby(), it ranks within each group independently, the direct pandas equivalent of adding PARTITION BY customer to a SQL window function:

# SQL: RANK() OVER (PARTITION BY customer ORDER BY amount DESC)
clean["amount_rank"] = clean.groupby("customer")["amount"].rank(
    method="first", ascending=False
)

ascending=False makes rank 1 the biggest value, matching "top N" intuition. The tie-breaking method matters more than it looks: method="first" breaks ties by row order — the ROW_NUMBER() flavor, always assigning 1, 2, 3 with no repeats. method="min" or "dense" give tied values the same rank — the RANK()/DENSE_RANK() flavor. Same SQL distinction, same reason to pick one over the other: use "first" when you need exactly N rows out no matter what; use "min"/"dense" when ties should visibly share a place.

2 · Top N per group: filter the rank, don't sort-and-slice by hand

Once every row has a per-group rank, top N per group is just a filter — the same move as wrapping a SQL window function in an outer WHERE rn <= N:

top_1_per_customer = clean[clean["amount_rank"] == 1]

A second, equally common way reaches the same answer without rank() at all: clean.sort_values("amount", ascending=False).groupby("customer").head(1) — sort first, then take the first N rows of each group. Both are idiomatic; rank() wins when you need the rank number itself kept as data (e.g. to show "this is your #2 order"), sort_values().groupby().head(N) wins when you only need the rows and never care what number they'd have been.

3 · cumsum(): the pandas SUM() OVER (ORDER BY ...)

cumsum() is a running total: each row gets the sum of itself plus everything before it, in whatever order the rows currently are. That last clause is the one gotcha — cumsum() has no ORDER BY argument the way SQL's window frame does; it just walks the DataFrame top to bottom. So the order has to already be right before you call it:

# SQL: SUM(amount) OVER (PARTITION BY customer ORDER BY order_date)
by_date = clean.sort_values("order_date")  # cumsum trusts row order — sort FIRST
by_date["running_total"] = by_date.groupby("customer")["amount"].cumsum()
Gotcha: skip the sort_values and cumsum() still runs — silently, with no error — it just accumulates in whatever order the rows happened to arrive from read_csv, which is rarely the order you meant. Unlike SQL, where ORDER BY inside the window is mandatory syntax you can't forget, pandas' cumsum() has nothing to force you to get this right — the sort has to happen as a deliberate step before it, every time.

Q: "Find each customer's top 2 orders by amount, and their running total spend over time." A strong answer, out loud: "For the top-2, I'd groupby customer, rank amount descending with method='first' so ties don't produce ambiguous duplicates, then filter to rank <= 2 — same idea as ROW_NUMBER() OVER PARTITION BY customer wrapped in a WHERE. For the running total, cumsum only accumulates in row order, not by any column I name, so I sort by order_date within customer first, then groupby customer and cumsum the amount — that ordering step is the one difference from SQL's SUM() OVER, which takes ORDER BY as part of the window itself." The subtext: naming that cumsum() has no built-in ordering clause — and catching that gap before it produces a wrong answer — is what separates "knows the function name" from "has been burned by this once."
