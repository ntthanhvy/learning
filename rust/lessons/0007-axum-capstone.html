1 · What an HTTP framework is actually for

Your Go project's server (net/http) and today's Rust service solve the identical problem: turn incoming bytes on a socket into a Rust value your code can work with, and turn your return value back into bytes to send. axum is the de-facto standard Rust web framework, built on tokio (Day 6) — and its central idea is one you already half-know from Go's http.HandlerFunc: a handler is just an async function. The difference is what a handler's *signature* is allowed to say.

In Go, every handler has the identical signature — func(w http.ResponseWriter, r *http.Request) — and you manually parse the URL, the body, the query string out of r yourself, by hand, every time. axum handlers instead declare exactly what they need as typed parameters, and axum extracts it for you before your code ever runs:

async fn get_link(Path(code): Path<String>) -> ... { /* code is already a String */ }

Each parameter type here is called an extractor. Path<String> pulls a URL segment; Json<T> parses (and, via T: Deserialize, type-checks) the request body; State<S> hands back whatever shared state you registered with the router. Get the type wrong — ask for a Path<u32> on a segment that isn't numeric — and axum rejects the request with a 400 automatically, before your handler body runs at all.

2 · Shared state: the HashMap needs a lock now

Day 5 put your links in a HashMap<String, Link>. Day 6 spawned concurrent tasks that each only read data they owned outright — no conflict possible. An HTTP server is different: tokio may run many request-handling tasks at the same time, and now they all need to read and write the same HashMap — the one holding every saved link. That's exactly the shared-mutable-state problem Day 6's Send/Sync discussion was building toward:

type Db = Arc<Mutex<HashMap<String, Link>>>;

Arc lets every request handler hold its own cheap, clonable pointer to the *same* underlying map — like a Go pointer shared across goroutines, but the compiler enforces that sharing is safe (that's what makes Arc<T>'s T require Send + Sync in the first place, straight from Day 6). Mutex is the actual safety mechanism: .lock() blocks until no one else is holding the lock, guaranteeing two concurrent requests can never corrupt the map by writing at the same instant — Go's sync.Mutex, same idea, except the borrow checker won't let you touch the data inside without going through the lock at all (there's no way to accidentally forget to lock — the HashMap isn't reachable except through the Mutex).

3 · Errors become real HTTP responses

Day 3 made errors a value you match on or propagate with ?. In a web service, an error has to become bytes on the wire — some status code, some body — and axum's answer is one trait: IntoResponse. Implement it once for your own error enum, and ? inside a handler works exactly like Day 3 — except now "returning early" means axum sends the matching HTTP response for you:

enum LinkError {
    NotFound(String),
}

impl IntoResponse for LinkError {
    fn into_response(self) -> Response {
        match self {
            LinkError::NotFound(code) =>
                (StatusCode::NOT_FOUND, format!("no link for code: {code}")).into_response(),
        }
    }
}

A handler returning Result<Json<Link>, LinkError> can now use ? exactly like any other Rust function this week — db.get(&code).cloned().ok_or(LinkError::NotFound(code))? — and a missing link becomes a real 404 response, without a single if err != nil-style manual status-code call anywhere in the handler body.

4 · Practice: build the service

Step 1 — dependencies. Edit Cargo.toml:

Step 2 — GIVEN: types, state, and the error type. Replace the top of main.rs with:

use axum::{
    extract::{Path, State},
    http::StatusCode,
    response::{IntoResponse, Response},
    routing::get,
    Json, Router,
};
use serde::{Deserialize, Serialize};
use std::collections::HashMap;
use std::sync::{Arc, Mutex};

#[derive(Clone, Serialize, Deserialize)]
struct Link {
    code: String,
    url: String,
}

#[derive(Deserialize)]
struct NewLink {
    code: String,
    url: String,
}

type Db = Arc<Mutex<HashMap<String, Link>>>;

#[derive(Clone)]
struct AppState {
    db: Db,
}

enum LinkError {
    NotFound(String),
}

impl IntoResponse for LinkError {
    fn into_response(self) -> Response {
        match self {
            LinkError::NotFound(code) =>
                (StatusCode::NOT_FOUND, format!("no link for code: {code}")).into_response(),
        }
    }
}

fn seed() -> Db {
    let mut m = HashMap::new();
    m.insert("go".to_string(), Link { code: "go".to_string(), url: "https://go.dev".to_string() });
    m.insert("rust".to_string(), Link { code: "rust".to_string(), url: "https://www.rust-lang.org".to_string() });
    Arc::new(Mutex::new(m))
}

Step 5 — GIVEN: list_links and main, wiring it all together:

async fn list_links(State(state): State<AppState>) -> Json<Vec<Link>> {
    let db = state.db.lock().unwrap();
    Json(db.values().cloned().collect())
}

#[tokio::main]
async fn main() {
    let state = AppState { db: seed() };

    let app = Router::new()
        .route("/links", get(list_links).post(create_link))
        .route("/links/{code}", get(get_link))
        .with_state(state);

    let listener = tokio::net::TcpListener::bind("127.0.0.1:3000").await.unwrap();
    println!("listening on {}", listener.local_addr().unwrap());
    axum::serve(listener, app).await.unwrap();
}

Note what disappeared: the old Command enum and hand-rolled parse_command from Day 1 are gone entirely — Router::new().route(path, method(handler)) IS the command dispatch now, just addressed by URL + HTTP verb instead of CLI words. Same shape (match an input against known cases, run the matching handler), different address space.

Step 6 — TODO 3: tests, calling handlers directly (no server needed). An axum handler is just an async function — you can call it directly in a test with hand-built extractor values, no HTTP round trip required:

#[cfg(test)]
mod tests {
    use super::*;

    #[tokio::test]
    async fn get_link_found_returns_ok() {
        let state = AppState { db: seed() };
        let result = get_link(State(state), Path("go".to_string())).await;
        assert!(result.is_ok());
    }

    #[tokio::test]
    async fn get_link_missing_returns_not_found() {
        let state = AppState { db: seed() };
        let result = get_link(State(state), Path("missing".to_string())).await;
        assert!(result.is_err());
    }

    // TODO 3: write create_link_then_get_it_back — call create_link with a

5 · What this week actually built

Line up the whole week: enums + exhaustive match (Day 1) → ownership, why Arc needs to exist at all (Day 2) → Result/?, now a real HTTP error path (Day 3) → traits, IntoResponse/Serialize/Deserialize being exactly that (Day 4) → HashMap + serde, today's actual storage and wire format (Day 5) → Send/Sync/tokio, today's concurrency safety net (Day 6) → today, all six wired into one running service. Every concept this week was, without exception, in service of this one file.

Q: "Pitch Rust for backend work, compared to Go." A strong answer, out loud: "Both give you real concurrency and a fast compiled binary. The difference shows up in what the compiler catches before it ships: Rust's ownership model rules out data races at compile time rather than catching them with the race detector at runtime, exhaustive match on enums means a new variant is a compile error everywhere it's unhandled, and Result forces every error path to be handled somewhere. It's a steeper first week — the borrow checker is real — but a lot of a Go on-call's 'someone forgot to check this' bugs simply don't compile in Rust." The subtext: contrasting *when* each language catches a bug (compile time vs. runtime vs. code review) is what separates this from "Rust is just faster."
