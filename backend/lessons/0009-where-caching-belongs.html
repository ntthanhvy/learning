Where caching belongs (and what invalidation really means)

Lesson 8 left off here: once instances scale freely, they all keep hitting Postgres for the same unchanging data on every single request. A cache is the fix — but only if you know exactly where it sits and exactly when it goes stale. Get either wrong and you've traded a slow bug for a wrong one.

1 · The question a cache always answers

Every cache exists to answer one question: "did I already do this expensive thing, and can I reuse that answer instead of doing it again?" The "expensive thing" is usually a database query, but it can be anything repeatable — a slow computation, a call to a third-party API you're rate-limited on. A cache trades a little staleness for a lot of speed: it hands back an answer computed earlier, betting that the underlying data hasn't changed since then.

The most common shape, and the one to reach for first, is cache-aside: on every read, check the cache first. If it's there — a cache hit — return it, no database touched at all. If it's not — a cache miss — read the real source (Postgres), then write that value into the cache before returning it, so the next request for the same thing is a hit.

// cache-aside, in miniature — the handler's own logic, not a framework feature
func getProduct(ctx context.Context, id string) (*Product, error) {
    if p, ok := redisGet(ctx, "product:"+id); ok {
        return p, nil // hit — Postgres never touched
    }
    p, err := db.QueryProduct(ctx, id) // miss — pay the real cost once
    if err != nil {
        return nil, err
    }
    redisSet(ctx, "product:"+id, p, 10*time.Minute)
    return p, nil
}

2 · Where a cache actually lives — three different layers

"Add caching" is not one decision — it's a choice of layer, and each layer trades off differently:

  • HTTP response caching. The Cache-Control header tells the browser and any proxy in between to reuse a response for N seconds — the request never even reaches your server again. Cheapest possible cache, but only works for responses that are truly the same for everyone (public data, static assets) — never for anything personalized or authenticated.
  • Application-level cache (Redis/Memcached). Your server code explicitly checks and writes a shared cache, as in the snippet above. This is the workhorse layer: it can hold per-user data, it's shared across every instance (Lesson 8's shared-nothing requirement), and it's fully under your control.
  • Database-level caching. Postgres already caches recently-read pages in memory (its buffer cache) — you get this for free, no code required, but you don't control what stays cached or for how long, and it does nothing to reduce query load the way an application cache does.

Most backend caching work happens at the middle layer — it's the only one you fully control and the only one that helps with per-user, authenticated data. HTTP caching is a free win when it applies; DB buffer caching is background behavior you benefit from without touching it.

3 · The part everyone underestimates: invalidation

A cache that never updates is just a bug with a delay on it. Two different strategies decide when a cached value stops being trusted, and most real systems use both together:

  • TTL (time-based expiry). Set an expiry when writing the entry (10*time.Minute above) and stop thinking about it — after that window, the entry is gone and the next read pays the real cost again. Simple, but for up to that whole window a write elsewhere can leave every cached reader looking at old data.
  • Explicit invalidation on write. The moment a product's price changes, the handler that updates it in Postgres also deletes (or overwrites) product:{id} in the cache — the very next read is a guaranteed miss, which correctly re-reads the fresh row. This closes TTL's staleness window, at the cost of remembering to do it at every single write path that touches that data.

The gap between those two is exactly what a stale read is: whenever an update happens and the corresponding cache entry hasn't been touched yet (TTL not expired, invalidation forgotten, or simply not there yet), some reader sees the old value. No cache eliminates this risk entirely — the real design decision is how much staleness a given piece of data can tolerate, and picking a TTL (or invalidation discipline) that matches it.

Frontend-habit trap: this is the exact same trade-off as a stale value sitting in a frontend cache (React Query, SWR, a service worker) after the underlying API data changed — "why does the UI still show the old value" is the same bug, just one layer further from the browser. The fix instinct transfers directly: either shorten the TTL, or explicitly invalidate the cached key the moment the write that changes it happens.

Q: "How would you add caching to this endpoint?" A strong answer, out loud: "First I'd check whether the response is the same for everyone — if so, an HTTP Cache-Control header might be enough and never even hits my server. If it's per-user or needs explicit control, I'd reach for cache-aside with Redis: check the cache, miss falls through to Postgres and writes the result back with a TTL. The part I'd think hardest about is invalidation — does this data need to be correct the instant it changes, or can it tolerate being up to N minutes stale? That answer picks the TTL, and tells me whether I also need to explicitly delete the cache key on every write path that touches this data." The subtext: naming the staleness tolerance as a deliberate choice — not an accident — is what separates "knows what Redis is" from "has actually designed a cache."
