Transform: broadcasting a group value back to every row

Lesson 15's teaser named transform() — a per-row group-relative value — as the next fresh pattern once a drill-outcome signal didn't surface. Every reshape and aggregation so far has either collapsed rows (groupby().sum()) or kept every row's own value untouched. Today's tool does neither: it keeps every row, but replaces each one with something computed from its group.

Recap, in one line

Lesson 4's groupby().agg() answers "one number per group" — the output is shorter than the input. Today's transform() answers "one number per group, but written back onto every row of that group" — the output is exactly as long as the input.

1 · The problem plain agg() can't solve directly

Say the task is: for every order, how far above or below its customer's own average is this order's amount? groupby("customer")["amount"].mean() gives the group averages — but that Series has one row per customer, not one row per order, so subtracting it straight from clean["amount"] doesn't line up. The textbook fix is a merge: compute the group means, then merge them back onto the original rows on customer, the same join mechanics as Lesson 5. That works, but it's two steps and a join key to get exactly back to where you started — every original row, no fewer, no more.

2 · transform() skips the merge entirely

transform() computes the exact same aggregation as agg(), but hands back a result already aligned row-for-row with the input it was called on:

group_mean = clean.groupby("customer")["amount"].transform("mean")
-- 0    81.00   (An's rows both get An's mean)
-- 1   107.75   (Binh's rows both get Binh's mean)
-- 2   107.75
-- 3    81.00

clean["amount"] - group_mean
-- 0    39.00   (120.0 is 39 above An's average)
-- 1   -72.25   (35.5 is 72.25 below Binh's average)
-- 2    72.25
-- 3   -39.00

No merge, no join key, no risk of a many-to-many blow-up if a merge key isn't unique — group_mean already has the same index as clean, so plain Series subtraction lines every row up correctly. This is the SQL equivalent of AVG(amount) OVER (PARTITION BY customer): a window function (Lesson 7's term) computed per partition without collapsing the rows — transform() is pandas' general-purpose member of that same family, alongside rank(), cumsum(), and Lesson 13's rolling()/expanding().

3 · transform() isn't limited to "mean"

Any aggregation name that works with agg() works with transform() — the only difference is the shape it hands back. Swap in "sum" to get each row's share of its customer's total instead of its distance from the average:

group_total = clean.groupby("customer")["amount"].transform("sum")
pct_of_group_total = clean["amount"] / group_total
-- 0    0.7407   (An's 120.0 is 74% of An's 162.0 total)
-- 1    0.1647   (Binh's 35.5 is 16% of Binh's 215.5 total)
-- 2    0.8353
-- 3    0.2593

Each customer's own percentages sum to exactly 1.0 — every row still carries its customer's full total, just divided differently per row. That's the whole pattern: pick the aggregation, call transform(), use the result exactly like any other same-length column.

transform() vs agg(), the one-sentence version: agg()/sum()/mean() collapse — fewer rows out than in. transform() broadcasts — exactly as many rows out as in, every row replaced by its group's value. Same aggregation vocabulary, opposite output shape.

Q: "How would you add a column showing each row's deviation from its group's average, without a separate merge step?" A strong answer, out loud: "I'd use groupby().transform('mean') instead of groupby().mean() — transform computes the same per-group average, but returns it already broadcast back to every original row, aligned by index, instead of collapsing to one row per group. That means I can subtract it directly from the original column with no merge and no join key. It's the same idea as a SQL window function like AVG(amount) OVER (PARTITION BY customer) — the aggregate is computed per partition, but every row keeps its place." The signal: naming the row-count-preserving property specifically, not just "it computes an average per group" — that part alone doesn't distinguish it from agg().
