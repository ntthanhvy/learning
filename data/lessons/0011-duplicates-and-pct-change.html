1 · Finding exact repeats: duplicated()

SQL answers "are there repeat rows" with GROUP BY ... HAVING COUNT(*) > 1, or a window trick with ROW_NUMBER(). pandas has a direct, row-level equivalent: duplicated(), which flags each row True the moment an identical row has already appeared above it:

is_dupe = raw_export.duplicated()
# False, False, True, False, True — the 1st occurrence of each row stays False,
# repeats after it flip True. Same idea as a running "have I seen this before?" check.

The default keeps the first occurrence unflagged (keep="first"). keep=False flips every copy to True, including the first — useful when the question is "show me every row involved in a duplicate," not just "which ones are extras." Passing a column list, e.g. duplicated(subset=["customer"]), narrows what "identical" means, exactly like grouping by a subset of columns instead of the whole row.

2 · Removing them: drop_duplicates()

Once rows are flagged, dropping them is one call — the twin of duplicated(), same keep/subset keywords, but returning the cleaned DataFrame directly instead of a boolean mask:

deduped = raw_export.drop_duplicates()
# equivalent to raw_export[~raw_export.duplicated()] — drop_duplicates() is just
# the named shortcut for "keep everything duplicated() marked False"

SQL's nearest cousin is SELECT DISTINCT (whole-row) or a dedupe-by-key pattern using ROW_NUMBER() OVER (PARTITION BY key ORDER BY ...) = 1. That ORDER BY matters in both worlds: if "duplicate" really means "same customer, keep the most recent," sort first, then drop_duplicates(subset=["customer"], keep="last") — reaching for the plain default silently keeps whichever copy happened to load first, which is rarely the intent.

3 · Change from the previous row: pct_change()

Lesson 7's cumsum() answered "running total so far." A different, equally common interview question is "how much did this grow compared to last time" — order-over-order, month-over-month. pct_change() is built exactly for that, and — like rank() and cumsum() — becomes per-group the instant it's chained after groupby():

ordered = clean.sort_values(["customer", "order_date"])
ordered["amount_pct_change"] = ordered.groupby("customer")["amount"].pct_change()
# An: 120.0 -> 42.0  is -0.65   (a 65% drop)
# Binh: 35.5 -> 180.0 is  4.07  (a 407% increase, i.e. ~5.07x)
# first order per customer has nothing before it -> NaN, same reasoning as
# Lesson 3's NaN-not-zero for "value genuinely doesn't exist yet"

Two things trip people up under time pressure. First, pct_change() trusts row order the same way cumsum() does — no ORDER BY argument of its own, so sort_values() first or the numbers are meaningless. Second, it returns a fraction, not a percentage — multiply by 100 (or format as a percent) before showing it to anyone; SQL has no single built-in for this at all, it's normally hand-written as (curr - LAG(curr) OVER (...)) / LAG(curr) OVER (...), which is exactly what pct_change() is doing under the hood.

Q: "This CSV export has some duplicate rows from a retry — walk me through cleaning it, and separately, how would you flag which customers grew fastest order-over-order?" A strong answer, out loud: "Two different tools. For the duplicates, I'd check duplicated() first to see how many and which ones before blindly dropping — if 'duplicate' means the whole row, plain drop_duplicates() is fine, but if it really means 'same customer, same day, keep the latest,' I'd sort by timestamp first and pass subset and keep='last' so I'm not keeping an arbitrary copy. For order-over-order growth, that's sort_values by date within customer, then groupby('customer')['amount'].pct_change() — same per-group idea as rank() and cumsum() from earlier, just answering 'change since last' instead of 'position' or 'running total.' Each customer's very first order comes back NaN, which is correct — there's nothing before it to compare to, not a zero-percent change." The subtext: naming why a default (keep="first", no sort) might be wrong for the specific question is what separates "knows the method" from "knows when to trust the default."
