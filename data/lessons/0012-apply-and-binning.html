1 · apply(axis=1): pandas' escape hatch back to per-row Python

Lesson 1 opened this whole course on "think in tables, not loops" — every operation so far (groupby, merge, rank, pct_change) has been a vectorized, column-wise call with no explicit Python loop anywhere. apply() is the deliberate exception: it hands each row (or column) to an ordinary Python function, one at a time. It exists for real cases where the per-row logic genuinely can't be expressed as a column-wise operation — combining several columns into one formatted string is a clean example, since there's no single vectorized "format these three columns together" call:

def make_label(row):
    return f"{row['customer']} ({row['region']}): ${row['amount']:.2f}"

merged["label"] = merged.apply(make_label, axis=1)
# "An (North): $120.00", "Binh (South): $35.50", ...

axis=1 is the part worth burning in: it means "one call per row," with each row handed in as a little Series you index by column name (row["customer"]). axis=0 (the default) instead calls the function once per column, handing in a whole column as a Series — a different, less common shape. SQL has no real equivalent for row-wise `apply()`; the nearest cousin is a scalar user-defined function called once per row, which most engines flag as a performance smell for exactly the same reason pandas does.

The catch, and why it matters in an interview: apply() is still a Python-level loop under the hood — one Python function call per row, no NumPy vectorization underneath. On a small DataFrame the difference is invisible; on a few million rows it can be 10-100x slower than an equivalent vectorized call. The rule worth saying out loud: reach for apply() only after confirming no vectorized operation covers the logic — and Section 2's tool is exactly the vectorized alternative for the specific case people reach for apply() for most often: turning a number into a category.

2 · pd.cut(): binning a number into labeled ranges, vectorized

"Turn this amount into Low/Mid/High" looks at first like an apply() job — a small if/elif/else per row — but it's actually one call, no Python loop, no per-row function at all: pd.cut().

merged["tier"] = pd.cut(merged["amount"], bins=[0, 40, 100, 200], labels=["Low", "Mid", "High"])
# 35.5  -> "Low"   (0, 40]
# 42.0  -> "Mid"   (40, 100]
# 120.0 -> "High"  (100, 200]
# 180.0 -> "High"  (100, 200]

bins gives the edges (here: 4 edges make 3 ranges), each interval is (previous_edge, this_edge] — left-exclusive, right-inclusive by default, so a value exactly on an edge falls into the lower bucket, not the upper one. labels names each range in order; leave it off and pandas shows the raw interval instead (e.g. (100, 200]), which is honest but not what you'd ship to a report. SQL's nearest cousin is a CASE WHEN amount <= 40 THEN 'Low' WHEN amount <= 100 THEN 'Mid' ELSE 'High' END — same idea, and also the exact per-row logic apply() would have needed if cut() didn't exist. The two-tool pairing today is the point: know both, and know which one a given "categorize this column" question actually calls for.

Q: "You need to bucket a numeric column into a few labeled ranges — walk me through it, and would you use apply()?" A strong answer, out loud: "For pure numeric binning, pd.cut() — one vectorized call, you give it bin edges and labels, no Python loop involved. I'd reach for apply(axis=1) only if the logic genuinely needs more than one column combined in a way that isn't just numeric ranges — building a formatted string from a few fields, say. And even then I'd double-check there isn't a vectorized way to do it first, because apply() is still a per-row Python function call under the hood — fine on a small table, a real cost on a few million rows." The signal: knowing cut() exists at all, and reflexively checking for a vectorized option before defaulting to apply() — the same "why, not just how" instinct Lesson 1 opened the whole course on.
