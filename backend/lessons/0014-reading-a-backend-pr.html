Reading a backend PR: substance over style

Lesson 13 closed on this teaser: putting the runtime and security tracks together into what a PR review actually looks like. Every lesson so far taught one concept at a time; today's job is different — given a diff, which of those thirteen lenses actually applies, and what does a comment that catches a real bug look like versus one that just moves commas around?

1 · A nit is free; it's also not the job

The frontend review habit that transfers cleanest is exactly the one that matters least here: catching a missing semicolon, a variable named x, an inconsistent quote style. Those are real, worth flagging, and conventionally prefixed nit: so the author knows they're optional. But a PR can pass every nit and still corrupt data, leak another user's records, or take the whole service down under load — and a review that only caught the nits missed the actual job. The instinct to build today: read a diff and ask "what does this code let happen at runtime" before "does this look tidy."

2 · Five questions, one per track already taught

Every lesson so far maps to one question worth asking of almost any backend diff. Not a checklist to run mechanically top to bottom — a set of lenses to hold up against whatever the diff actually touches:

  • Data model (Lessons 2, 5): does this touch the schema? Is the migration safe to run against a live table, and can it be rolled back?
  • API contract (Lesson 3): does this change a response shape, a status code, or a URL that some client already depends on? Is it actually backward compatible, or just untested against one?
  • Auth (Lessons 4, 12): does every new endpoint check both that the caller is logged in and that this specific record belongs to them? A missing ownership check is Lesson 12's IDOR, and it's the single most common bug this lens catches.
  • Correctness under concurrency (Lesson 6): if this handler writes to more than one table, or reads-then-writes the same row, is it inside a transaction? Two instances of this handler running at once — does anything race?
  • Operations (Lessons 7–11, 13): what does a failure look like to the caller (information disclosure), to another instance (shared state), to the cache (invalidation), to a retry (idempotency), and to whoever's on call (logs, metrics)?

3 · Reading a real diff

A small PR: "add PATCH /orders/{id} to let a customer update their shipping address." The diff, as submitted:

func (s *Server) patchOrderHandler(w http.ResponseWriter, r *http.Request) {
    orderID := r.PathValue("id")
    userID := auth.UserIDFromContext(r.Context()) // Lesson 4: who is calling
    if userID == "" {
        http.Error(w, "unauthorized", http.StatusUnauthorized)
        return
    }

    var body struct{ ShippingAddress string `json:"shipping_address"` }
    json.NewDecoder(r.Body).Decode(&body)

    s.db.Exec(r.Context(),
        "UPDATE orders SET shipping_address = $1 WHERE id = $2", body.ShippingAddress, orderID)
    s.db.Exec(r.Context(),
        "INSERT INTO order_events (order_id, event) VALUES ($1, 'address_updated')", orderID)

    w.WriteHeader(http.StatusOK)
}

Running the five lenses: data model — no schema change, skip. API contract — a new endpoint, shape looks reasonable, skip. Auth — stop here. The handler checks that someone is logged in, never that orderID belongs to userID. Any authenticated user can PATCH any order by guessing or incrementing an id in the URL — Lesson 12's IDOR, verbatim, in production code that "works" in every manual test a developer runs against their own account. Correctness under concurrency — also stop here. The address update and the event-log insert are two separate statements with no transaction around them; if the process crashes or the second call errors between them, the order's address changes with no audit trail of it — Lesson 6's exact failure shape.

Neither bug shows up in a quick manual test, neither is a style issue, and neither would be caught by only reading for naming and formatting. Both show up the moment the five lenses are actually applied.

4 · What the comment itself should say

"This is wrong" or "fix this" tells the author a location, not a reason — they still have to reconstruct the failure themselves, and might reconstruct a different, easier-to-dismiss one. A substantive comment names the concrete scenario a reviewer would need to explain out loud in an interview:

// Weak: "this needs an ownership check"

// Substantive:
// blocking: this handler never verifies orderID belongs to userID — any
// logged-in user can PATCH another user's order by changing the id in the
// URL (IDOR, Lesson 12). Add `WHERE id = $2 AND user_id = $3` to the UPDATE,
// or a separate SELECT ... FOR UPDATE ownership check before it, and return
// 404 (not 403 — don't confirm the order exists to a non-owner) if it fails.

The shape worth copying: name the failure scenario concretely (who, doing what, causing what), point at which lesson/pattern it violates, and propose a specific fix — not just a direction to think harder in. The optional blocking: prefix (mirroring nit:) tells the author this one has to be resolved before merge, not just considered.

Q: "Walk me through how you'd review a backend PR." A strong answer, out loud: "First pass for the nits — naming, formatting — and I'll mark those non-blocking. The real review is running the diff through a handful of questions: does it touch the data model, and is any migration safe and reversible? Does it change the API contract in a way some client already depends on? Does every new endpoint check not just that the caller is authenticated, but that they own the specific record — that's the most common bug I catch. If it writes to more than one place, is it in a transaction, and what happens if two instances run it at once? And operationally — what does a failure look like to the caller, to the cache, to whoever's on call reading the logs. A comment that just says 'this is wrong' isn't useful; I try to name the actual scenario that breaks and propose a concrete fix." The signal: a repeatable set of lenses, not "I read it carefully," and comments that name failure scenarios instead of vague objections.
