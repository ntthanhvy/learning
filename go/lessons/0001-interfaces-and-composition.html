Interfaces & composition

The one design choice that shapes every idiomatic Go codebase — and the foundation for everything else this week.

You already write Go, so today isn't "what is an interface." It's the three idioms that separate working Go from Go that passes review — and interviews: implicit satisfaction, small consumer-defined interfaces, and composition by embedding.

Terms: click any dotted term to get a popup — the software meaning in English first, then Vietnamese. Every term is also in the glossary.

1. Satisfaction is implicit — and that changes where interfaces live

A type satisfies an interface simply by having the methods. No implements keyword, no declaration, no import of the interface's package (Effective Go). This is not a syntax nicety — it flips who owns the abstraction: because the implementer doesn't need to know an interface exists, the consumer gets to define it.

The rule: define interfaces in the package that uses them, not the package that implements them. Return concrete types; let consumers abstract as they need (Go Code Review Comments). A central interfaces package is a Java habit, not a Go one.

2. Small interfaces: the bigger the interface, the weaker the abstraction

That's a Go proverb, and the stdlib lives it: io.Reader is one method, and half the ecosystem composes around it. A one- or two-method interface is trivially satisfied, trivially faked in tests, and combines freely (io.ReadWriter is just two embedded interfaces).

✗ Fat, implementer-defined
type UserService interface {
    Create(u User) error
    Get(id string) (User, error)
    Update(u User) error
    Delete(id string) error
    List() ([]User, error)
    Validate(u User) error
}
✓ Small, consumer-defined
// in the handler package,
// which only needs lookups:
type UserGetter interface {
    Get(id string) (User, error)
}

func ShowUser(g UserGetter, id string)

3. Accept interfaces, return structs

Parameters name the behavior you need; return values give callers the full concrete type. If your function takes *os.File but only calls Read, it should take io.Reader — now it works with buffers, network connections, and test fixtures for free.

4. Composition by embedding

Go has no inheritance; it has embedding. Embed a type and its methods are promoted — the outer type satisfies whatever the inner one satisfies, and you override by redeclaring (Effective Go — Embedding):

type LoggingStore struct {
    Store              // embedded interface — promotes all its methods
    log *slog.Logger
}

func (s LoggingStore) Save(ctx context.Context, l Link) error {
    s.log.Info("saving", "code", l.Code)
    return s.Store.Save(ctx, l)  // delegate to the wrapped value
}

This wrapper pattern changes just one method and delegates the rest: LoggingStore logs, then hands the actual saving to the wrapped Store. This is how middleware, logging, and caching layers are built all over production Go.

"Why doesn't Go have inheritance?" and "where do you define interfaces?" are stock questions. Strong answer in one breath: implicit satisfaction lets consumers define minimal interfaces at the point of use, so packages stay decoupled without a shared type hierarchy; reuse comes from embedding, which is composition, not inheritance.

Step 2 — link/link.go (given, copy it)

package link

import (
    "context"
    "errors"
    "time"
)

// Link is one shortened URL.
type Link struct {
    Code      string    // the short code, e.g. "go"
    URL       string    // the destination
    CreatedAt time.Time
}

// ErrNotFound is returned when a code has no link. (Day 2 names this pattern: sentinel error.)
var ErrNotFound = errors.New("link: not found")

// Store is what consumers of this package need from storage — nothing more.
type Store interface {
    Save(ctx context.Context, l Link) error
    Get(ctx context.Context, code string) (Link, error)
}

Step 5 — main.go (given, copy it)

package main

import (
    "context"
    "fmt"
    "log/slog"
    "time"

    "linkshort/link"  // module name from go.mod + package directory
)

func main() {
    store := link.NewLoggingStore(link.NewMemStore(), slog.Default())
    ctx := context.Background()

    if err := store.Save(ctx, link.Link{Code: "go", URL: "https://go.dev", CreatedAt: time.Now()}); err != nil {
        panic(err)
    }

    got, err := store.Get(ctx, "go")  // promoted — you never wrote Get on LoggingStore
    if err != nil {
        panic(err)
    }
    fmt.Println(got.Code, "→", got.URL)
}

Now go run . — you should see a log line for the save (from your code) followed by go → https://go.dev.

func TestLoggingStoreDelegates(t *testing.T) {

    s := NewLoggingStore(fake, slog.Default())

    if err := s.Save(context.Background(), Link{Code: "x", URL: "https://go.dev"}); err != nil {
        t.Fatal(err)

    if len(fake.saved) != 1 || fake.saved[0].Code != "x" {
        t.Fatalf("save was not delegated to the inner store: %+v", fake.saved)

}
