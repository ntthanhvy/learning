Variables, pointers, and &T{}

A refresher on what the declaration forms actually mean — then a re-read of your own scaffold code, symbol by symbol.

1. The three ways to create a variable

var count int          // ① declare with a type → gets the ZERO VALUE (0 here)
var name = "go"        // ② declare with a value → type is inferred (string)
code := "abc"          // ③ short form of ② — only allowed INSIDE functions

①–③ all create the same kind of thing; they differ only in what you spell out. The important concept is the zero value: in Go a variable is never uninitialized — var x int is 0, var s string is "", and var m map[string]Link is nil (Tour of Go).

The nil-map trap: a nil map can be read (returns zero values) but writing to it panics. That's exactly why NewMemStore exists in your scaffold — it replaces the nil map with a real, empty one before anyone writes.

2. Constructing values: T{}, the composite literal

A composite literal creates a value and fills it in one expression:

l := Link{Code: "go", URL: "https://go.dev"}  // struct: named fields (unset ones → zero value)
m := map[string]Link{}                        // empty-but-real map (writable, unlike nil)
s := []string{"a", "b"}                       // slice with two elements

For maps, slices, and channels you can also use make: make(map[string]Link) is the same as map[string]Link{}; make earns its keep when you pre-size, e.g. make([]Link, 0, 100) (Effective Go — allocation).

3. & and * — three jobs, two symbols

This is the usual blur point, because * has two different jobs depending on where it appears:

You seeWhereIt means
*MemStorein a type position"pointer to MemStore" — a type, like int or string
*pin an expressiondereference: the value p points at
&vin an expressionaddress-of: a pointer to v

So &MemStore{links: …} is just two things you now know, stacked: a composite literal (MemStore{…} — create and fill the struct) wrapped in address-of (& — and give me a pointer to it). One expression, one new struct, one pointer to it.

Why pointers at all?

Go copies values when you assign or pass them. A pointer is how two places share one value instead of each holding copies:

func reset(n *int) { *n = 0 }   // writes through the pointer → caller sees it

x := 5
reset(&x)                        // pass x's address, not a copy of 5
fmt.Println(x)                   // 0

The same logic drives receivers: func (s *MemStore) Save(…) means "Save runs against the original store," so every caller sees the same map. Rule of thumb: if any method needs the pointer, give all methods of that type pointer receivers (Code Review Comments). And no pointer arithmetic exists in Go — a pointer is a reference, not a number.

4. Now re-read your own scaffold

Every mystery symbol from yesterday's practice, decoded:

func NewMemStore() *MemStore {          // returns POINTER-TO-MemStore: all users share one store
    return &MemStore{                   // composite literal + address-of: build it, point at it
        links: map[string]Link{},       // real empty map, not nil — writable
    }
}

func (s *MemStore) Save(…) error {      // pointer receiver: Save touches the original, not a copy
    s.links[l.Code] = l                 // s.links auto-dereferences — no (*s).links needed
    …
}

var _ Store = (*MemStore)(nil)          // hardest line: "(*MemStore)(nil)" = a nil pointer that has
                                        // the TYPE *MemStore; assigning it to a Store variable makes
                                        // the compiler verify *MemStore satisfies Store. The "_"
                                        // means: keep no variable, I only wanted the check.
