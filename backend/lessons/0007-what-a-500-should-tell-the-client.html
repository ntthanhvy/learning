What a 500 should (and shouldn't) tell the client

Same runtime/security track as Lesson 6 — no dependency on it. Lesson 3 gave your API an error contract: a stable shape every failure returns. Today is the other half of that promise — what's actually allowed to go inside it when the failure is your server's own bug, not the client's mistake.

1 · A 500 has two audiences, and they want opposite things

A frontend habit carries over here in a way that quietly turns dangerous: console.error(err) in the browser is fine to print in full — the stack trace, the file paths, the library versions — because only the one developer looking at that console ever sees it. A server's error response goes over the network, to whoever sent the request, which might be your own frontend, or might be anyone with a URL and curl.

The two people who care about a 500 want different things from it. The engineer debugging it wants everything: the full stack trace, the exact query that failed, which internal service timed out. The client code that receives the response wants almost nothing: just enough to decide what to do next — retry, show a generic error, or give up. Every backend framework's default error page (a raw stack trace rendered as HTML or JSON) is built for the first audience and shipped, by default, to the second one too — that's the actual bug.

2 · What never belongs in the response body

OWASP calls this class of bug information disclosure: none of the following should ever reach a client, no matter how convenient it is while you're debugging locally.

  • Stack traces. Every frame names a file path, a function, sometimes a line of your actual source — free reconnaissance for an attacker mapping your stack.
  • Raw exception/driver messages. A Postgres error like duplicate key value violates unique constraint "users_email_key" just told the caller your table and column names, and that the row already exists — useful for enumerating valid accounts.
  • Library and version strings. "Express 4.17.1" or a specific ORM version tells an attacker exactly which known CVEs to try.
  • Internal file paths. /app/src/services/payment/charge.go:142 reveals your project layout for no benefit to a legitimate client.

The tell that something's wrong: if turning on a debug flag would change what a production response body contains, the non-debug version is already leaking too much reasoning about internals — the safe response shouldn't need a flag to become safe.

3 · What replaces it: a generic message plus a correlation ID

The fix isn't "tell the client nothing" — an API that returns {"error": "something went wrong"} with no way to reference it is nearly as bad for support and debugging as leaking a stack trace is for security. The fix is separating the two audiences at the response boundary: the client gets a safe, stable message from Lesson 3's error contract; the engineer gets everything, but from the server's own logs, connected by one shared value.

// what the client actually receives — nothing here can be used to attack anything
{
  "error": {
    "code": "internal_error",
    "message": "Something went wrong on our end. Please try again.",
    "request_id": "req_8f3a1c9e"
  }
}

That request_id is a correlation ID (also called a request ID): meaningless by itself, safe to hand to anyone, and useless to an attacker — but paired with your own logs, it's the exact thread that unspools into the full stack trace, the failing query, everything the engineer wanted in section 1. On the server side, that detail lives in structured logs keyed by that same request_id, queryable the moment a user reports "it broke, here's the ID from the error screen."

Frontend-habit trap: "I'll just forward the caught exception's message to the client, it's more helpful than a generic string." Helpful to whom — the same person who could also just read the server logs, or an anonymous caller who now knows your schema? If a message is genuinely useful to a legitimate client (e.g. "email already registered"), that's a 4xx with its own stable error code from Lesson 3 — not a 500's internals leaking through.

Q: "How do you avoid leaking internal details in error responses while still being able to debug production issues?" A strong answer, out loud: "Every response follows one error contract — a stable code and a safe, generic message for anything that isn't the client's own mistake. I attach a request ID to every incoming request, log it alongside the full stack trace and query details server-side, and return that same ID in the error body. So the client and any support ticket get something to reference, but nothing that helps an attacker map the system — the actual detail only exists in logs I control." The subtext: naming where the detail goes (structured server logs, not the wire) is the part that separates this from "I'd just be more careful about what I send back."
