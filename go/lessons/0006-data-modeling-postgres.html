1 · Schema design as invariant design

linkshort needs two tables. links you already know — it's Link{Code, URL, CreatedAt}, one row per short code. The new one is clicks: one row every time handleRedirect fires. The design question worth pausing on: should a link's hit count live as a plain integer column on links (like Day 4's in-memory hits map[string]int), or as a separate log of individual click events?

A single counter column is denormalized: fast to read, but it's a fact derived from something else (the clicks themselves), stored redundantly, and it can drift if a write path forgets to increment it. A clicks table with one row per hit is normalized: the count becomes a query (COUNT(*) ... GROUP BY link_code, Day 6's territory) instead of a number that can lie. It also gives you things a counter never could — when each click happened, which Day 6's pagination and aggregation both need. Start normalized; denormalize later only with a measured reason (a cache column, refreshed deliberately) — never as the first draft.

Either way, the schema is really a set of invariants you're choosing to make impossible to violate: every click must point at a link that exists; every link must have a non-empty URL; a code names at most one link. Section 2 turns each of those into a line of DDL.

2 · Constraints: the first line of correctness

A constraint is an invariant the database enforces on every write, from every caller, forever — including code you haven't written yet. That beats remembering "always check the URL isn't empty" in every handler:

CREATE TABLE links (
    code       text PRIMARY KEY,
    url        text NOT NULL CHECK (url <> ''),
    created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE clicks (
    id         bigint GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    link_code  text NOT NULL REFERENCES links(code) ON DELETE CASCADE,
    clicked_at timestamptz NOT NULL DEFAULT now()
);

Four constraints, four invariants: PRIMARY KEY on code says "at most one link per code" (Go's map already assumed this — now the database enforces it too). NOT NULL + CHECK (url <> '') says "a link without a destination isn't a link." REFERENCES links(code) is a foreign key: no click can ever point at a nonexistent link, checked on every insert, not just the ones your code remembers to validate. ON DELETE CASCADE decides what happens when a link is deleted — its clicks go with it. That's the right call for linkshort's domain (an orphaned click means nothing); in a different domain (say, an audit log) you'd pick RESTRICT instead and refuse the delete. The cascade behavior is itself a design decision, not a default to accept blindly.

3 · Choosing keys: natural vs. surrogate

Two different key decisions sit in that DDL, on purpose. links.code is a natural key: the short code is already unique by the nature of the problem (two links can't share a redirect target path), so it doubles as its own primary key — matching the Go side, where MemStore was already keyed by code in a map. clicks.id is the opposite: a click has no natural identity — "the click that happened at 14:02:03 for code X" isn't guaranteed unique (two people can click the same link the same millisecond) — so it gets a surrogate key: GENERATED ALWAYS AS IDENTITY, Postgres's modern auto-incrementing integer (the SQL-standard replacement for the older SERIAL). Rule of thumb for interviews: reach for a natural key when one field is already guaranteed unique and stable; reach for a surrogate the moment that's not true, or the "natural" field could ever need to change.

4 · Indexes: paying for reads with writes

Every PRIMARY KEY and UNIQUE constraint above silently created an B-tree index — Postgres's default, and the right choice for the vast majority of lookups (equality, ranges, sorting). Indexes aren't free: every write updates every index on that table too, so an index is a bet that this column gets searched on far more often than the table gets written to. For linkshort, the bet that matters is Day 6's aggregate query — "how many clicks per link, most recent first" — so today you add the index that query will need before you've even written it:

CREATE INDEX clicks_link_code_clicked_at_idx
    ON clicks (link_code, clicked_at DESC);

That's a composite index: it serves "all clicks for link X" alone, and "all clicks for link X, newest first" even faster, because the second column is already sorted within each first-column group — exactly the shape Day 6's keyset pagination wants. Column order matters: this index is useless for "all clicks at time T across every link," because link_code isn't fixed in that query.

One more shape worth knowing by name, even though linkshort doesn't need it yet: a partial index covers only rows matching a condition, e.g. CREATE INDEX ... ON links (code) WHERE created_at > now() - interval '30 days' for a table where only recent rows are ever looked up hot-path. Smaller index, same B-tree, a fraction of the maintenance cost — the tool for "this table is huge but 99% of queries only touch a sliver of it."

5 · Migrations: schema changes as versioned artifacts

A migration is just a SQL file with a number in front of its name, committed to the repo like any other code: 0001_init.sql, 0002_add_something.sql. Running them in order, in any environment — your laptop, a teammate's, a fresh Neon branch for Day 7's integration tests — reproduces the exact same schema, and the migration file itself is the reviewable record of why the schema looks the way it does. Today's practice writes exactly one: 0001_init.sql.

package link

import (
    "context"
    "errors"

    "github.com/jackc/pgx/v5"
    "github.com/jackc/pgx/v5/pgxpool"
)

var _ Store = (*PostgresStore)(nil)

type PostgresStore struct {
    pool *pgxpool.Pool
}

func NewPostgresStore(ctx context.Context, connString string) (*PostgresStore, error) {
    pool, err := pgxpool.New(ctx, connString)
    if err != nil {
        return nil, err
    }
    return &PostgresStore{pool: pool}, nil
}

// TODO: Save upserts a link — INSERT ... ON CONFLICT (code) DO UPDATE SET
// url = EXCLUDED.url, using s.pool.Exec(ctx, query, l.Code, l.URL, l.CreatedAt).
// Return the Exec error directly; pgx reports constraint violations as a
// typed error, same job as Go's own error values.
func (s *PostgresStore) Save(ctx context.Context, l Link) error {

}

// Get: given. Note the shape — this is your comma-ok/ErrNotFound pattern
// from Day 2, translated to SQL: pgx.ErrNoRows means "no row", mapped to
// the SAME sentinel MemStore already returns. Callers never know which
// Store they're talking to — that's the whole point of Day 1's interface.
func (s *PostgresStore) Get(ctx context.Context, code string) (Link, error) {
    var l Link
    err := s.pool.QueryRow(ctx,
        `SELECT code, url, created_at FROM links WHERE code = $1`, code,
    ).Scan(&l.Code, &l.URL, &l.CreatedAt)
    if errors.Is(err, pgx.ErrNoRows) {
        return Link{}, ErrNotFound
    }
    if err != nil {
        return Link{}, err
    }
    return l, nil
}
