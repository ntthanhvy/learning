API versioning: what changes safely, what doesn't

Lesson 3 designed the contract; Lesson 15 made it scale. This is the last thing that contract needs: a rule for changing it later without breaking every client who already trusts it.

1 · The contract is a promise, not a snapshot

Lesson 3 called the API contract a promise — the shape of every request and response, agreed on before the frontend and backend teams stop talking to each other. A promise made once still has to survive change: a new field, a renamed field, a field that needs to disappear. The frontend code calling GET /orders/42 today was written against the contract as it exists today — it has no way to know the contract changed unless something in this scheme tells it. API versioning is that scheme.

2 · Two kinds of change, and only one is free

Not every change needs a version bump. The dividing line is whether existing client code — code that only reads the fields it already knows about — keeps working unmodified.

ChangeBreaks existing clients?
Adding a new optional field to a responseno
Adding a new optional request parameter with a defaultno
Adding a new endpointno
Renaming or removing a fieldyes
Changing a field's type or meaning (string id → int id)yes
Changing a status code an endpoint returns for the same conditionyes
Making a previously-optional request field requiredyes

The first group is a backward-compatible change — ship it any day, no version bump, no announcement. A client that ignores fields it doesn't recognize (the normal, correct way to write a client) never notices. The second group is a breaking change — this is what versioning exists to manage, because "just deploy it" means every client depending on the old shape breaks at the same moment, with no warning.

3 · Where the version lives

Three common places to put it, each with a real trade-off:

SchemeLooks likeTrade-off
URL path/v1/orders, /v2/ordersMost visible and cacheable; but the resource's identity now includes a version number, which feels wrong for something that's still "the same order"
HeaderAccept: application/vnd.api+json;version=2Keeps URLs clean and stable; but versions are invisible in logs, browser address bars, and copy-pasted curl commands
Query parameter/orders?version=2Easy to add later without a routing change; easy to omit by accident, silently falling back to a default

Stripe and GitHub's public APIs both use a variant of the header approach (a single header or a date-based version string) precisely because it decouples "which resource" from "which shape of that resource" — the URL /orders/42 keeps meaning the same thing forever. The URL-path scheme is more common in practice anyway, mostly because it's obvious from a browser bar or a log line which version a request hit, and that legibility tends to win for internal and mid-size APIs even though it's the theoretically messier option.

4 · The real cost isn't picking a scheme — it's supporting two versions at once

The hard part of versioning was never where to put the number. It's that shipping v2 doesn't retire v1 — every existing client on v1 keeps calling it, so the service now runs two contracts simultaneously: two sets of validation rules, maybe two response-serialization paths, and every bug fix has to be decided against both ("does this fix apply to v1 too, or is v1 frozen as-is?"). This is why the honest advice from every team that has actually done this at scale is the same: avoid the breaking change if there is any way to. A few reframes that dodge a version bump entirely:

  • Renaming a field → add the new name alongside the old one, deprecate the old one in docs, remove it only after telemetry shows nothing calls it anymore.
  • Changing what a field means → add a new field with the new meaning instead of repurposing the old one.
  • A required field that wasn't required before → give it a server-side default instead of forcing every client to send it.

When a breaking change is genuinely unavoidable — a field that returns wrong data, not just old-shaped data, for instance — that's when a new version earns its cost, and even then, a deprecated version with a stated sunset date (not an indefinite "eventually") is what turns "supporting two versions forever" into "supporting two versions for six months."

5 · A response shape that survives a breaking change without a version bump

Some breaking changes really are just additive if you design the response with room to grow. Compare two ways of returning a total:

// Fragile: if "total" ever needs to become an object (say, {amount, currency}),
// this is a breaking change — every client reading total as a number now gets an object.
{ "total": 4200 }

// Sturdier: the number can grow a sibling field without displacing anything
// a client already reads.
{ "total_cents": 4200, "currency": "USD" }

Naming the unit in the field itself (total_cents, not total) and giving currency its own field up front means "we now support multiple currencies" never becomes a breaking change — it was always going to need a currency field, so ship it on day one even with a single hardcoded value. This is the same instinct as Lesson 15's opaque cursor: design the contract so the obvious next requirement is additive, not a rename.

Q: "Your team wants to change a field's type on a widely-used endpoint. How do you ship that?" A strong answer, out loud: "First I'd check whether it's actually a breaking change — if I can add a new field alongside the old one instead of repurposing it, that's backward-compatible and needs no version bump at all, so that's always my first move. If it's genuinely unavoidable — the old field is actively wrong, not just old-shaped — then I'd stand up a new version rather than change v1 in place, because existing clients are depending on the current shape right now and a silent change breaks them with no warning. I'd pick whichever versioning scheme the rest of the API already uses for consistency, and I'd explicitly deprecate the old version with a stated sunset date rather than leave it open-ended — the real cost here isn't the version number, it's that the service now has to maintain two contracts at once, and an open-ended deprecation means maintaining them forever instead of for a fixed window." The signal: leading with "avoid the breaking change if possible," and naming the ongoing-maintenance cost as the actual reason versioning is expensive — not the mechanics of picking v1 vs v2.
