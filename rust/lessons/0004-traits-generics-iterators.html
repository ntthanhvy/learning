1 · trait: Go's interface, said out loud

Go's Store interface (Day 1 of your other course) works by implicit satisfaction — any type with the right methods qualifies, with no declaration anywhere. Rust's version, a trait, requires the opposite: an explicit impl Trait for Type block, always, even if the type already happens to have a method of the right shape by coincidence:

trait Summary {
    fn summary(&self) -> String;
}

impl Summary for Link {
    fn summary(&self) -> String {
        format!("[{}] {}", self.code, self.url)
    }
}

Nothing here is new mechanically — it's the same impl block you've used since Day 1 for inherent methods. What's new is that this impl is now for a trait, which means any code that only knows "something implementing Summary" can call .summary() on it, without knowing it's a Link at all. That's the whole pitch: swap implicit-by-accident (Go) for explicit-by-declaration (Rust) — a type can never accidentally satisfy an interface it wasn't meant to.

2 · Generics + trait bounds: one function, many types

A function written over a type parameter T, constrained by what T can do, is generic:

fn print_all<T: Summary>(items: &[T]) {
    for item in items {
        println!("{}", item.summary());
    }
}

T: Summary is a trait bound: "any T works here, as long as it implements Summary." The compiler generates a separate, specialized copy of print_all for every concrete type you actually call it with — a process called monomorphization — so there's no runtime cost at all, unlike Go's generics (which the runtime handles more dynamically in some cases) or any language doing this via boxing.

3 · dyn Trait: when you need it decided at runtime

Sometimes you genuinely don't know the concrete type until runtime — a list holding several different types that all implement the same trait. That's dyn Trait, dynamic dispatch via a vtable, closer to how Go's interface values always work under the hood:

fn print_one(item: &dyn Summary) {
    println!("{}", item.summary());
}
T: Summary (static)dyn Summary (dynamic)
ResolvedCompile time, monomorphizedRuntime, via a vtable pointer
CostZero — same as calling the concrete type directlyOne indirect call, small but nonzero
Use whenThe type is known per call siteYou need a mixed collection, or the type genuinely varies at runtime

Default to T: Trait (or impl Trait in argument position, its shorthand) unless you specifically need a heterogeneous collection or a trait object stored behind a pointer — that's the idiomatic Rust default, and the interview-ready answer for "why does Rust have two ways to do this."

4 · Iterator adapters: the loop you don't write

Your Get arm right now is a found-flag loop — the exact shape a iterator adapter chain replaces:

✗ Found-flag loop (Day 1, still in your file)
let mut found = false;
for l in &links {
    if l.code == code {
        println!("Found: {}", l.describe());
        found = true;
    }
}
if !found {
    println!("No link found for code: {}", code);
}
✓ One adapter, no flag
match links.iter().find(|l| l.code == code) {
    Some(l) => println!("Found: {}", l.describe()),
    None => println!("No link found for code: {}", code),
}

|l| l.code == code is a closure: an anonymous function, here capturing code from the enclosing scope. .find(...) returns Option<&Link> directly — the "did we find it" bookkeeping the found-flag loop hand-rolled is now the adapter's job, and there's no way to forget to reset the flag because there's no flag. This is the same instinct as SQL's declarative WHERE, or your Go course's own vectorized-thinking lesson: describe what you want, let the engine do the looping.

Adapters chain. .iter().map(|l| l.code.clone()).collect::<Vec<String>>() turns a slice of links into a Vec<String> of just their codes — one line replacing a manual loop with a fresh Vec and a manual push. Nothing runs until .collect() (or a for loop) actually consumes the chain — each adapter just describes one more step, lazily.
