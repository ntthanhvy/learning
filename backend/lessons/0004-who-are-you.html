Who are you?

Lesson 1's stateless server forgets everything between requests — so how does it recognize the same logged-in user, request after request?

Lesson 1's punchline was the backend is stateless; the database is the state. That raises an obvious problem: if the server keeps nothing in memory, how does request #2 know it's the same person as request #1? The browser doesn't re-send a password on every click. Something has to travel with each request that proves "this is Lan, still logged in" — and deciding what that something is, is authentication. Two designs dominate; today you learn both and when each earns its place.

Design 1: sessions — a claim ticket for server-side state

The login handler checks the password once, then hands back a claim ticket instead of making the browser re-prove itself every time:

1. POST /login {email, password}  ──▶  handler verifies password
2. handler creates a row: sessions(id="a1b2c3...", user_id=42, expires_at=...)
3. response: Set-Cookie: session_id=a1b2c3...; HttpOnly; Secure
4. every later request ──▶  browser auto-attaches the cookie
5. middleware: SELECT user_id FROM sessions WHERE id = 'a1b2c3...' ──▶  42, attach to request

Name the pieces: the cookie carries only a random, meaningless session token — a lookup key, not the user's identity itself. The identity — which user, what role, when it expires — lives in a session store: a database table (or Redis) row per logged-in session. Note what did not break: the handler still starts from zero every request (Lesson 1's rule intact) — it just does one extra lookup, in the database, same as looking up an order.

Design 2: tokens (JWT) — a self-contained claim

A session lookup is one more query per request. What if the claim ticket could prove itself, with no database at all?

header:  {"alg": "HS256"}
payload: {"user_id": 42, "role": "admin", "exp": 1799999999}
signature: HMAC(header + payload, server's secret key)   -- the part that can't be forged

-- the three parts, base64-joined with dots, IS the token:
eyJhbGciOiJIUzI1NiJ9.eyJ1c2VyX2lkIjo0Mn0.k3f9a2...

A JWT carries the user's claims directly, signed so that tampering (changing role to "admin") invalidates the signature and gets rejected. Verification is math, not a query — that's the entire appeal: any of your five server copies behind the load balancer can verify it alone, no shared session store required. The bill comes due on revocation: a session row can be deleted the instant you want someone logged out; a JWT is valid until it expires, full stop, unless you build a second system (a denylist) to check against — at which point you've reinvented a database lookup and lost the point of tokens.

SessionJWT
Where identity livesServer-side store, looked up by tokenInside the token itself
Per-request costOne DB/cache lookupSignature check only (no lookup)
Instant revoke ("log this user out now")Delete the row — doneNot until expiry, without extra machinery
Scales across servers with no shared storeNeeds a shared session storeYes — that's the whole selling point

Rule of thumb going into an interview: sessions when you control the backend and want easy revocation (most first-party web apps); tokens when multiple independent services must verify identity without calling each other back (microservices, third-party APIs). Neither is "more secure" by default — they trade off differently.

Where this sits in the request journey

This is Lesson 1's middleware layer, finally given a job: before the handler runs, middleware reads the cookie or Authorization: Bearer <token> header, resolves it to a user (a lookup for sessions, a signature check for JWTs), and attaches that identity to the request. The handler never parses auth itself — it just asks "who is this request from," already answered.

The frontend habit this retrains

You've stored a token in localStorage before because it's easy to read in JS. That ease is the risk: any script running on your page — including an injected one from an XSS bug — can read localStorage and steal the token. An HttpOnly cookie (the session example above) is invisible to JavaScript entirely, closing that hole — at the cost of opening a different one: the browser attaches cookies automatically to any request to your domain, including ones a malicious page tricks the browser into firing, called CSRF. Neither storage choice is free of trade-offs; "where do I put the token" is itself a security decision, not a implementation detail.
