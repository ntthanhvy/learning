Today is the toolbox tour: sync.Once, sync/atomic, sync.Map, sync.Pool, and singleflight — five narrower tools that each beat a plain mutex in one specific shape, and are worse than a plain mutex in every other shape. link/memstore.go (replace the whole file — Day 8's mutex-guarded store):

package link

import (
    "context"
    "sync"
)

var _ Store = (*MemStore)(nil)

type MemStore struct {
    mu    sync.RWMutex
    links map[string]Link
}

func NewMemStore() *MemStore {
    return &MemStore{links: make(map[string]Link)}
}

func (s *MemStore) Save(ctx context.Context, l Link) error {
    s.mu.Lock()
    defer s.mu.Unlock()
    s.links[l.Code] = l
    return nil
}

func (s *MemStore) Get(ctx context.Context, code string) (Link, error) {
    s.mu.RLock()
    defer s.mu.RUnlock()
    l, ok := s.links[code]
    if !ok {
        return Link{}, ErrNotFound
    }
    return l, nil
}

link/counter.go (Day 8's click counter — today's Section 6 adds an atomic sibling next to it):

package link

import "sync"

// ClickCounter is a concurrency-safe tally of redirects per code.
type ClickCounter struct {
    mu     sync.Mutex
    counts map[string]int
}

func NewClickCounter() *ClickCounter {
    return &ClickCounter{counts: make(map[string]int)}
}

func (c *ClickCounter) Inc(code string) {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.counts[code]++
}

func (c *ClickCounter) Count(code string) int {
    c.mu.Lock()
    defer c.mu.Unlock()
    return c.counts[code]
}

func (c *ClickCounter) Snapshot() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    out := make(map[string]int, len(c.counts))
    for k, v := range c.counts {
        out[k] = v
    }
    return out
}

checker.go (replace the whole file — Day 9's worker pool + Day 10's ctx/errgroup rewrite, exactly as Day 11 used it):

package main

import (
    "context"
    "strings"
    "time"

    "linkshort/link"

    "golang.org/x/sync/errgroup"
)

type result struct {
    code string
    ok   bool
}

func checkOne(l link.Link) result {
    time.Sleep(50 * time.Millisecond)
    return result{code: l.Code, ok: strings.HasPrefix(l.URL, "https://")}
}

const maxWorkers = 4

func checkAll(ctx context.Context, links []link.Link) map[string]bool {
    jobs := make(chan link.Link)
    results := make(chan result)

    g, ctx := errgroup.WithContext(ctx)
    for w := 0; w < maxWorkers; w++ {
        g.Go(func() error {
            for {
                select {
                case l, open := <-jobs:
                    if !open {
                        return nil
                    }
                    select {
                    case results <- checkOne(l):
                    case <-ctx.Done():
                        return ctx.Err()
                    }
                case <-ctx.Done():
                    return ctx.Err()
                }
            }
        })
    }

    go func() {
        defer close(jobs)
        for _, l := range links {
            select {
            case jobs <- l:
            case <-ctx.Done():
                return
            }
        }
    }()

    go func() {
        g.Wait()
        close(results)
    }()

    out := make(map[string]bool, len(links))
    for {
        select {
        case r, open := <-results:
            if !open {
                return out
            }
            out[r.code] = r.ok
        case <-ctx.Done():
            return out
        }
    }
}

1 · sync.Once — exactly once, thread-safe, no matter who asks first

A sync.Once answers one narrow question: "run this exactly once, and make every concurrent caller wait for that one run instead of racing to do it themselves." The naive version — a plain if cfg == nil { cfg = load() } — is a textbook data race: two goroutines can both see nil, both call load(), and one of them can even observe the other's half-written struct.

var (
    cfg     *Config
    cfgOnce sync.Once
)

func GetConfig() *Config {
    cfgOnce.Do(func() {
        cfg = loadConfig() // runs exactly once, ever, across all goroutines
    })
    return cfg
}

Every goroutine calls Do with the same closure; Once guarantees exactly one of those closures actually executes, and — this is the part a hand-rolled flag-plus-mutex gets wrong if you're not careful — every other caller's Do doesn't return until that one execution has completed, so nobody ever observes a half-initialized cfg. This is Go's answer to the "lazy singleton" pattern, and it's also exactly the tool yesterday's "close exactly once" rule was missing: wrap a channel close in a sync.Once and a second, third, hundredth call to close it is simply a no-op instead of a panic.

type SafeCloser struct {
    ch   chan struct{}
    once sync.Once
}

func (c *SafeCloser) Close() {
    c.once.Do(func() { close(c.ch) }) // safe from any number of callers, any number of times
}

2 · sync/atomic — lock-free updates to a single value

An atomic operation updates one machine word — an int, a pointer, a bool — without a lock at all, using a CPU instruction that can't be interrupted mid-update. Go 1.19+ wraps this in typed values so you stop juggling raw int64s and offsets:

var total atomic.Int64

func (c *AtomicTotal) Inc() { total.Add(1) }
func (c *AtomicTotal) Load() int64 { return total.Load() }

No Lock/Unlock, no struct to define beyond the atomic type itself — and for exactly this shape (one counter, one flag, one pointer swap) it's both simpler and measurably cheaper than a mutex, because there's no OS-level contention path to fall into under load. The catch is the scope: an atomic protects exactly the one value it wraps, nothing else. The moment you need to update two related fields together — say, a total count AND a last-updated timestamp, and they must never be observed out of sync — one atomic per field does NOT give you that guarantee; two goroutines can interleave between the two independent atomic writes. That's still a mutex's job. Rule of thumb: reach for atomic for one bare counter or flag, reach for a mutex the instant more than one field needs to change together.

3 · sync.Map — not a faster map, a differently-shaped one

The obvious-looking upgrade from "map + mutex" is sync.Map — and it's the most misapplied tool in this package. Its own doc comment is blunt about it: use a plain map plus a separate Mutex/RWMutex "for most use cases," and reach for sync.Map only in two specific shapes — (1) a key is written once and read many times afterward (a cache that only grows), or (2) many goroutines each touch a disjoint set of keys with little overlap. ClickCounter's map fails both tests — every code's count is written repeatedly, by any goroutine, forever — so it correctly stays a plain map[string]int behind a Mutex, not a sync.Map.

var m sync.Map
m.Store("go", 42)
v, ok := m.Load("go")          // v is `any` — a type assertion is on you
m.Range(func(k, v any) bool { return true }) // keep iterating

Two more costs worth knowing for an interview: values come back as any, so every read needs a type assertion the compiler can't check for you — a plain typed map doesn't have that problem. And it has no Len() method at all; counting entries means a full Range.

4 · sync.Pool — reusing garbage instead of making more

A sync.Pool holds reusable scratch objects — the classic example is a []byte or bytes.Buffer reused across many requests instead of allocated fresh (and garbage-collected) every time:

var bufPool = sync.Pool{
    New: func() any { return new(bytes.Buffer) },
}

func encode(v any) []byte {
    buf := bufPool.Get().(*bytes.Buffer)
    defer func() { buf.Reset(); bufPool.Put(buf) }() // reset BEFORE returning it — the pool doesn't clear for you
    json.NewEncoder(buf).Encode(v)
    return append([]byte{}, buf.Bytes()...)
}

The one rule that trips people up: the pool may evict anything, at any time, for any reason — most aggressively, it drops every pooled item across a GC cycle. That makes it correct only for pure allocation-reduction, never for anything with state you need to survive (a connection, a cache entry, a counter). linkshort's stdlib-only HTTP handlers would reach for exactly this pattern to reuse response buffers under load; today's practice stays with Once and atomic, since Pool's payoff only shows up under real allocation pressure a course exercise can't easily simulate.

5 · singleflight — one flight in the air, everyone else waits for the same seat

singleflight solves a shape that shows up constantly in real backends and that neither Once (only ever runs once, period) nor a mutex (serializes, but every caller still re-does the work in turn) actually fixes: a cache stampede. Say linkshort's redirect handler has a cold cache and a hundred requests for the same code land in the same instant — without protection, all hundred hit Postgres at once for identical data.

var g singleflight.Group

func resolveWithDedup(ctx context.Context, code string) (link.Link, error) {
    v, err, _ := g.Do(code, func() (any, error) {
        return store.Get(ctx, code) // only ONE of the hundred concurrent callers actually runs this
    })
    if err != nil {
        return link.Link{}, err
    }
    return v.(link.Link), nil
}

Unlike Once, this runs again for the next stampede on the same key — it dedupes calls that overlap in time, not calls forever. That's the whole difference in one sentence: Once is "never again," singleflight is "not right now, while one's already in flight."

Step 2 — extend link/counter.go: add an atomic sibling next to the existing mutex-guarded ClickCounter, tracking one grand total (not per-code — that stays on ClickCounter, per Section 3's rule about sync.Map not fitting a repeatedly-written key set).

import "sync/atomic"

// AtomicTotal is a lock-free running total — one value, no map.
type AtomicTotal struct {
    n atomic.Int64
}

func (a *AtomicTotal) Inc()        { a.n.Add(1) }
func (a *AtomicTotal) Load() int64 { return a.n.Load() }

// MutexTotal is the same running total, guarded by a plain Mutex — the
// benchmark below measures the cost this indirection adds over AtomicTotal.
type MutexTotal struct {
    mu sync.Mutex
    n  int64
}

func (m *MutexTotal) Inc() {
    m.mu.Lock()
    defer m.mu.Unlock()
    m.n++
}

func (m *MutexTotal) Load() int64 {
    m.mu.Lock()
    defer m.mu.Unlock()
    return m.n
}

Step 3 — new file link/counter_bench_test.go: benchmark both under concurrent load with RunParallel, which spins up GOMAXPROCS goroutines all hammering Inc at once — the exact shape that makes lock contention visible.

package link

import "testing"

func BenchmarkAtomicTotal(b *testing.B) {
    a := &AtomicTotal{}
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            a.Inc()
        }
    })
}

func BenchmarkMutexTotal(b *testing.B) {
    m := &MutexTotal{}
    b.RunParallel(func(pb *testing.PB) {
        for pb.Next() {
            m.Inc()
        }
    })
}
