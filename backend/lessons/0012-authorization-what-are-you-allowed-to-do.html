Authorization: what are you allowed to do?

Lesson 4 answered "who are you?" — sessions and JWTs both end with middleware attaching an identity to the request. That was never the whole story, and the glossary said so at the time: authentication is "distinct from authorization, which decides what they're allowed to do once known." Today is that second half — the one most handlers quietly skip.

1 · Knowing who ≠ knowing what they can do

Once Lesson 4's middleware runs, every handler downstream knows who is asking. It's tempting to stop there — the request has a user attached, so it must be fine. It isn't. Authorization is a second, independent check: this specific user, doing this specific action, on this specific piece of data — allowed, or not? A logged-in user is a fact about identity. Whether they may delete order #4821 is a fact about permissions, and nothing about being logged in answers it.

2 · Two granularities: role-based, and record-level

Most systems need both, and they answer different questions:

  • RBAC answers "does this kind of user get to do this kind of thing?" — can any admin delete any order? Cheap to check (one field on the user, or one join to a roles table) and coarse by design.
  • Record-level (ownership) checks answer a narrower question RBAC can't: "does this user own this row?" A regular user isn't admin, but they still need to edit their own profile, their own orders — a permission that has nothing to do with roles at all.

Real systems layer both: "an admin, OR the record's owner, may proceed." Skipping the second one is where most real bugs live.

3 · The bug that skipping record-level checks creates

Here's a handler that looks completely reasonable — it's authenticated, it queries the database, it returns JSON:

func getOrderHandler(w http.ResponseWriter, r *http.Request) {
    orderID := r.PathValue("id")
    order, err := store.GetOrder(r.Context(), orderID)
    if err != nil {
        http.Error(w, "not found", http.StatusNotFound)
        return
    }
    json.NewEncoder(w).Encode(order) // BUG: returns ANY order to ANY logged-in user
}

Authentication middleware ran; the request definitely came from a real, logged-in user. Nobody ever asked whether this user is allowed to see this order — so any authenticated user can read anyone else's data just by changing the id in the URL. This exact pattern is common enough to have its own name: IDOR, and it's been the OWASP Top 10's "Broken Access Control" entry for years running — not an exotic attack, just a missing if.

func getOrderHandler(w http.ResponseWriter, r *http.Request) {
    user := UserFromContext(r.Context()) // attached by Lesson 4's auth middleware
    orderID := r.PathValue("id")
    order, err := store.GetOrder(r.Context(), orderID)
    if err != nil {
        http.Error(w, "not found", http.StatusNotFound)
        return
    }
    if order.UserID != user.ID && !user.HasRole("admin") {
        http.Error(w, "forbidden", http.StatusForbidden) // 403, not 401
        return
    }
    json.NewEncoder(w).Encode(order)
}

One line, both granularities at once: admin role OR ownership. That if is the entire fix — and it has to be repeated, deliberately, in every handler that touches a record someone might not own. There's no middleware that can do this one for you the way authentication's middleware could, because only the handler knows which record is actually being asked for.

4 · 401 vs 403: a status code that's also a security signal

The two failure codes say genuinely different things, and interviewers ask this specifically because backend devs mix them up constantly:

CodeMeansFix from the client's side
401 UnauthorizedAuthentication itself failed or is missing — the server doesn't know who you are.Log in / attach a valid token.
403 ForbiddenAuthentication succeeded — the server knows exactly who you are — and that identity isn't allowed to do this.Nothing to retry; a different user or role is required.

Getting this backwards is a small tell in an interview: 401 means "prove who you are"; 403 means "I know, and the answer is still no."

Frontend-habit trap: hiding a delete button when user.role !== "admin" is a UX nicety, not a security control. Anyone can open devtools, or just call the API directly with curl — the button was never what stopped them. Every authorization decision the UI reflects must also be enforced server-side in the handler; the frontend check is a convenience for well-behaved users, never the actual gate.

Q: "Walk me through how you'd add a 'delete order' feature safely." A strong answer, out loud: "First, the route sits behind Lesson 4's auth middleware, so I know who's asking. But that's not enough — inside the handler I load the order and check that either the caller owns it or has an admin role, and return 403 if neither is true, before touching anything else. I wouldn't rely on the frontend hiding the delete button — that's just UX, not a security boundary, since anyone can call the endpoint directly. And I'd make sure the check happens on every mutating route that touches a specific record, not just this one, because it's per-handler, not something middleware gives you for free." The signal: naming that authentication and authorization are two separate checks, and that the second one is the handler's job, record by record.
