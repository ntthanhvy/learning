What happens after fetch()?

The anatomy of one request through a backend — and the one mindset shift that changes how you read every backend file from now on.

You already know half of this conversation. From the frontend you send fetch("/api/orders/42") and you get JSON and a status code back. This lesson names everything that happens in between, so that when you open a backend codebase you know what each layer is for.

The journey of one request

browser ── GET /api/orders/42 ──▶ ┌───────────── server process ─────────────┐
                                  │ router ▶ middleware ▶ handler            │
                                  │                        │                 │
                                  │                 business logic           │
                                  │                        │                 │
                                  │                 connection pool          │
                                  └────────────────────────│─────────────────┘
                                                           ▼
                                                       PostgreSQL
◀── 200 OK + JSON ── (serialize) ◀─────────────── rows come back
  1. A server — one long-running program — is listening on a port. Your request arrives as plain text: a method, a path, headers, maybe a body. Nothing frontend about it survives — no components, no framework, just HTTP.
  2. The router matches GET /api/orders/{id} and picks one function to run.
  3. Before that function runs, middleware gets its turn: log the request, check the auth token, set a timeout. Cross-cutting jobs live here, not copy-pasted into every function.
  4. The handler runs: parse the input (id = "42"), call the business logic, choose a status code.
  5. The logic needs data, so it borrows a database connection from the connection pool, runs a query, gets rows back.
  6. The result is turned into JSON — serialization — and sent back with the status code. Done. The server forgets everything and waits for the next request.

In Go (your work stack), the whole skeleton is visible in a few lines:

mux := http.NewServeMux()
mux.HandleFunc("GET /api/orders/{id}", getOrder)  // route → handler

func getOrder(w http.ResponseWriter, r *http.Request) {
    id := r.PathValue("id")                    // parse input
    order, err := store.Find(r.Context(), id)  // logic + DB via the pool
    if err != nil {
        http.Error(w, "not found", http.StatusNotFound)
        return
    }
    json.NewEncoder(w).Encode(order)           // serialize the response
}

The mindset shift

The backend is stateless; the database is the state. Step 6 above is the heart of it: the server keeps nothing between requests. Being stateless is deliberate — it lets you run five copies of the server behind a load balancer, restart any of them anytime, and lose nothing. Anything that must survive is persisted.

This inverts your frontend instincts, where state living in memory (useState, a store) is the normal case. The mapping:

Frontend habitBackend equivalent
Component receives propsHandler receives a request
useState holds state between rendersNothing is held between requests — the database holds state
State change → re-renderEvery request re-runs the handler from zero
One user's browser, one user's stateOne database, every user's state — shared and concurrent
Handle the status code fetch() gives youYou choose the status code and error body

That fourth row is why backend design is its own discipline: the database is shared, concurrent, and permanent. Schema design (coming lessons, plus your Go week Days 5–6) is high-stakes for the same reason.
