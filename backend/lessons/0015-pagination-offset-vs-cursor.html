Pagination: offset vs. cursor

MISSION.md's API-design criterion lists resource naming, verbs, status codes, pagination, error contract, idempotency — Lesson 3 covered the first two, Lesson 7 the error contract, Lessons 3 and 10 idempotency. Pagination is the one word in that list this course never actually taught. A list endpoint that returns every row works fine in a demo with 50 rows and quietly becomes the slowest, most fragile part of the API once a table has 5 million.

1 · "Just return all of them" doesn't survive scale

GET /orders returning every row is the natural first version — no pagination logic, no edge cases, ships fast. It stays fine right up until the table stops being small: one response serializing millions of rows, one query holding a read lock or scanning the whole table, one client trying to parse a multi-gigabyte JSON body it never asked for. The fix isn't a bigger server; it's never sending all the rows in the first place. Pagination is the API-design answer to the same problem Lesson 5's indexes solved on the query side: don't do more work than the caller actually needs right now.

2 · Offset/limit: simple, and simple to break

The obvious approach mirrors SQL directly. The client asks for a page number; the handler translates it into LIMIT/OFFSET:

-- GET /orders?page=2&per_page=20
SELECT * FROM orders ORDER BY created_at LIMIT 20 OFFSET 20;

This is offset pagination, and it reads exactly like the frontend mental model of "page 2" — easy to reason about, easy to build page-number links for. It has two real problems, and both get worse as the table grows, not better.

Problem one: OFFSET doesn't skip work, it does the work and throws it away. OFFSET 100000 doesn't jump to row 100,000 the way an array index would — Postgres still has to walk through and discard the first 100,000 matching rows on every single request, even with an index on created_at. Page 2 is fast. Page 5,000 is scanning and discarding 100,000 rows just to return the next 20 — the deeper into the list a client pages, the slower every request gets.

Problem two: page 2 breaks under concurrent writes. Offset pagination silently assumes the list holds still between requests. It never does. Say a client fetches page 1 (rows 1–20, ordered by created_at), then before it fetches page 2, a new row is inserted at the front of that order. Every row shifts down by one — what was row 20 is now row 21, sitting in the range page 2 is about to fetch as row 21–40 collapses to a different actual set. The client ends up seeing row 20 twice (once on each page) or, if a row is deleted instead of inserted, skipping a row entirely — with no error, no warning, just quietly wrong data stitched across two "correct" responses.

3 · Cursor pagination: ask "after what?" instead of "skip how many?"

Cursor pagination (also called keyset pagination) replaces "skip N rows" with "give me rows after this specific point." The client doesn't send a page number — it sends back an opaque cursor, typically the sort key of the last row it received:

-- GET /orders?after=2026-07-20T09:00:00Z_42  (created_at, id of the last row seen)
SELECT * FROM orders
WHERE (created_at, id) > (:cursor_created_at, :cursor_id)
ORDER BY created_at, id
LIMIT 20;

The pair (created_at, id) rather than created_at alone matters: created_at isn't guaranteed unique (two orders can share a timestamp), so tie-breaking on id guarantees a strict, total order — no row is ever skipped or repeated because two rows landed on the same instant. This directly fixes both of offset pagination's problems. It's stable under writes: a new row inserted anywhere doesn't change what "after this cursor" means, because the cursor names an actual row, not a position that shifts when the list does. And it's fast at any depth: with an index on (created_at, id), that WHERE clause is an index seek straight to the right spot — page 5,000 costs the same as page 2, because there's no OFFSET to walk through and discard.

4 · The trade-off, named honestly

Cursor pagination isn't strictly better — it gives up something offset pagination has: random access. There's no way to ask a cursor-paginated endpoint for "page 7" directly; a cursor only ever means "after wherever I last stopped," so reaching page 7 means walking through pages 1–6 first. That's a non-issue for infinite-scroll UIs and API-to-API consumers, which only ever ask for "the next batch" anyway — exactly the shape most list endpoints actually have. It's a real problem for a UI that renders clickable numbered page links (1 2 3 ... 9), where a user expects to jump straight to page 7. That's the case where offset/limit still earns its place — just not as the default for a large or fast-growing table.

 Offset/limitCursor/keyset
Jump to arbitrary pageyesno — sequential only
Speed on a deep pagedegrades — scans and discardsconstant — index seek
Stable under concurrent writesno — rows shift between pagesyes — cursor names a row, not a position

5 · A cursor-paginated handler

type Order struct {
    ID        int64     `json:"id"`
    CreatedAt time.Time `json:"created_at"`
}

// ListOrdersAfter returns up to limit orders strictly after the given cursor,
// ordered oldest-first. A zero-value cursor (createdAt IsZero) returns the first page.
func (s *Store) ListOrdersAfter(ctx context.Context, createdAt time.Time, id int64, limit int) ([]Order, error) {
    rows, err := s.db.Query(ctx,
        "SELECT id, created_at FROM orders "+
        "WHERE (created_at, id) > ($1, $2) "+
        "ORDER BY created_at, id LIMIT $3",
        createdAt, id, limit)
    if err != nil {
        return nil, err
    }
    defer rows.Close()

    var out []Order
    for rows.Next() {
        var o Order
        if err := rows.Scan(&o.ID, &o.CreatedAt); err != nil {
            return nil, err
        }
        out = append(out, o)
    }
    return out, rows.Err()
}

func listOrdersHandler(w http.ResponseWriter, r *http.Request) {
    createdAt, id := decodeCursor(r.URL.Query().Get("after")) // zero value if absent
    orders, err := store.ListOrdersAfter(r.Context(), createdAt, id, 20)
    if err != nil {
        http.Error(w, "internal error", http.StatusInternalServerError)
        return
    }
    var next string
    if len(orders) == 20 { // full page: there may be more, hand back a cursor for it
        last := orders[len(orders)-1]
        next = encodeCursor(last.CreatedAt, last.ID)
    }
    json.NewEncoder(w).Encode(struct {
        Orders []Order `json:"orders"`
        Next   string  `json:"next_cursor,omitempty"`
    }{orders, next})
}

The client's job on each call is small: send back whatever next_cursor it was last given, or nothing for the first page. It never has to know what's inside the cursor or do arithmetic on a page number — that opacity is what makes it safe to change the encoding later without breaking clients.

Q: "How would you paginate a list endpoint backed by a table with millions of rows?" A strong answer, out loud: "I'd reach for cursor pagination, not offset/limit. Offset/limit looks simpler — LIMIT and OFFSET map directly to a page number — but it has two real problems at scale: OFFSET doesn't skip the earlier rows, it still scans and discards them, so a deep page gets slower the further in you go. And it's not stable under writes — if a row is inserted or deleted between two page requests, the rows after it shift, and the client can see a duplicate or silently miss a row. Cursor pagination fixes both: the client sends back the sort key of the last row it saw, typically (created_at, id) to break ties, and the query becomes a WHERE-greater-than with an index seek instead of a scan-and-discard — so page 5,000 costs the same as page 2, and a mid-list insert or delete never shifts what 'after this cursor' means. The trade-off I'd flag up front is that cursor pagination can't jump to an arbitrary page — no direct 'page 7' — which is fine for infinite scroll or an API consumer that only ever asks for the next batch, but wrong for a UI that needs numbered page links, where I'd keep offset/limit instead." The signal: naming the OFFSET-cost and write-stability problems specifically, not just "cursor pagination is more efficient," and volunteering the random-access trade-off unprompted.
