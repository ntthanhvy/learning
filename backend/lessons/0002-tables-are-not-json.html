Tables are not JSON

How backend devs decide what becomes an entity — before any SQL is written.

Recall Lesson 1's punchline: the backend is stateless; the database is the state. Today's question follows directly: if the database holds everything, what shape should it hold it in? Your frontend instinct says "the shape my screen needs" — one nested JSON tree per view. That instinct is exactly what this lesson retrains.

Two shapes for the same data

Here's the order screen you'd happily render from:

{
  "id": 42,
  "customer": { "name": "Lan Phạm", "email": "lan@example.com" },
  "items": [
    { "product": "Bàn phím cơ", "qty": 1, "price": 89.00 },
    { "product": "USB hub",    "qty": 2, "price": 15.50 }
  ],
  "total": 120.00
}

And here's what a backend dev stores instead — four tables, no nesting:

-- illustration, not a SQL lesson — you know the syntax
customers   (id, name, email)
products    (id, name, price)
orders      (id, customer_id, created_at)          -- customer_id → customers.id
order_items (order_id, product_id, qty, unit_price) -- one row per line item

Why does the stored shape differ so much from the rendered shape? Because they answer to different masters:

A JSON response is shaped for one screen. A schema is shaped for every feature — including the ones nobody has asked for yet. The response is a view, rebuilt per request; the schema is the truth it's built from. Confuse the two and you've designed a database for one screen.

The nested JSON above is fine — as output. The handler assembles it per request by joining tables. What you must never do is store that tree as-is: the same customer would be copied into every order they place, and the day Lan changes her email you'd be updating a thousand frozen copies — or missing some.

Finding the entities

An entity is a noun with identity: it exists on its own, changes over time, and other things point at it. The working test, applied to the JSON above:

  • Would two screens refer to the same one? The customer appears on the order screen, the profile screen, the admin list — same person everywhere. Entity.
  • Does it have a lifecycle of its own? A product is created, repriced, discontinued — independent of any order. Entity.
  • Or is it just a fact about something else? qty: 2 describes one line of one order. total is arithmetic over the items. Not entities — a column and a computed value.

Each entity gets a table and a primary key. The JSON's nesting then turns into pointers: orders.customer_id is a foreign key saying "this order belongs to that customer". And order_items is a join table: many orders contain many products, and each row of it ties one order to one product.

One fact, one place

The principle behind all that splitting is normalization. You don't need the textbook normal forms yet; you need the instinct: every fact lives in exactly one row, and everyone else points at it. Lan's email lives in customers, nowhere else. When it changes, one UPDATE, and every screen that joins to customers is instantly right. Duplicated facts drift apart — and a database that disagrees with itself is worse than a slow one.

Now the interesting judgment call. Look again at order_items.unit_price — the price is copied from products, not joined. Isn't that duplication? Deliberately, yes: this is denormalization — because it's a different fact. products.price is "what this costs now"; order_items.unit_price is "what the customer paid then". If you joined instead of copying, repricing the keyboard next week would silently rewrite last month's invoices. Same-looking number, two facts. Backend design interviews love this distinction, and so do auditors.

The instinct retrain

Frontend habitSchema thinking
Shape data for the component rendering itShape data for every feature, present and future
Nest what appears together into one treeSplit things with identity into tables, linked by keys
Copy a value into props wherever it's neededStore each fact once — copies drift and lie
Compute derived values (total) in renderCompute them in the query — rarely store them
