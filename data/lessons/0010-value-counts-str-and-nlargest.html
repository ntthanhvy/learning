1 · Counting without a groupby: value_counts()

Lesson 4's groupby("customer").size() answers "how many rows per customer" — correct, but three words longer than it needs to be for the single most common counting question in any interview. Series.value_counts() does the same split-count in one call, directly on a column, sorted by count descending by default:

orders_per_customer = clean["customer"].value_counts()
# same numbers as clean.groupby("customer").size() — value_counts() IS a groupby+count,
# just spelled for the one-column case that comes up constantly

Reach for value_counts() the instant the question is "how many of each," full stop — no other column involved. The moment a second aggregation joins in (total revenue and count, say), that's back to groupby(...).agg(...) from Lesson 4; value_counts() only ever does the one thing.

2 · Filtering text with the .str accessor

SQL's WHERE column LIKE '%th%' has a direct pandas equivalent: .str.contains(). It's not "call .contains() on the column" — a plain Python string has no such vectorized method — it's the .str accessor, a namespace that runs the equivalent per-character-string method across the whole column in one vectorized pass:

th_region_customers = customers[customers["region"].str.contains("th", case=False)]
# .str.contains, .str.upper, .str.startswith, .str.split — all vectorized,
# all the "obvious" string method, just reachable through .str first

The case=False keyword is the same idea as SQL's ILIKE vs LIKE — case-insensitive by explicit request, not by default. Forgetting .str and calling a string method directly on a column is one of the single most common pandas AttributeErrors — the column is a Series of strings, not a string itself.

3 · Top N overall vs. top N per group: nlargest()

Lesson 9's Drill 1 found each customer's own second-highest order — a per-group question, answered with rank() inside a groupby. A plainer, more common question skips the grouping entirely: "the two biggest orders, full stop, across everyone." That's what nlargest() is for — no groupby, no rank, one call:

top_two_orders = clean.nlargest(2, "amount")
# equivalent to clean.sort_values("amount", ascending=False).head(2),
# but nlargest names the intent directly and reads faster in review

The two are easy to conflate under time pressure — the fix is one question: does "top N" mean overall, or per something? Overall reaches for nlargest/nsmallest; per-group reaches for groupby(...).rank() (Lesson 9) or groupby(...).head(n) after sorting. Naming which one a question is actually asking, out loud, before writing any code, is exactly the fluency an interviewer is listening for.

Q: "Give me the top 3 orders by amount for each customer, in one line." A strong answer, out loud: "That's per-group, not overall, so nlargest by itself won't do it — I'd sort_values by amount descending, then groupby('customer').head(3), which keeps each group's first 3 rows post-sort. Or equivalently groupby('customer')['amount'].rank(ascending=False) filtered to rank <= 3, same idea as Lesson 9's Nth-highest drill, generalized past N=2. I'd reach for nlargest only if the question drops 'for each customer' and just wants the biggest 3 across the whole table." The subtext: catching "for each customer" as the word that rules out nlargest is the actual skill being tested, not the syntax.
