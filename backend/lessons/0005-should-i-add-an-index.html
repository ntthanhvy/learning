Should I add an index?

If Lesson 4's auth quiz still has a fuzzy answer, no problem — today is a different track (runtime behavior) and doesn't build on it. A PR adds a new query; someone comments "should this have an index?" This lesson is the reasoning you run before answering, not yet the SQL itself.

Lesson 2's punchline was tables are shared state, and the schema is a promise everyone codes against. An index is a promise of a different kind: "this specific access pattern will stay fast, at the cost of every write doing a little more work." The mistake frontend instincts make here is treating "add an index" like adding a cache — a free speedup with no downside. It isn't. Reasoning about whether one earns its place is itself the skill; the Go track (this week's Days 5–6) covers the Postgres mechanics — B-tree structure, composite columns, EXPLAIN — in depth. This lesson is the frame you bring before you open that toolbox.

1 · An index is a trade, not a freebie

A database index makes specific reads fast by paying for it on every write: insert, update, or delete on that table now has to update every index on it too, not just the row itself. So the real question is never "would this query benefit from an index" — almost any query would. It's: does this table get read that way often enough, relative to how often it's written, to justify slower writes forever? A table written once and read a million times a day is an easy yes. A table written on every request and read for a monthly report is not.

2 · Selectivity: the question before the index

An index only helps if the column actually narrows things down. Selectivity is that measure: a user_id or email column is highly selective — most values pick out a handful of rows, so an index skips almost the entire table. A boolean is_active column is nearly the opposite: if 95% of rows are true, "find all active users" still means reading almost the whole table, index or not — the database may reasonably ignore the index entirely and scan the table anyway, because scanning is cheaper than jumping around for that many matches.

Frontend-habit trap: "this column is in a lot of WHERE clauses, so index it" skips selectivity entirely. The column that actually deserves the index is the one that's both frequently queried and narrows the result to a small slice of the table.

3 · The trap that isn't about indexes at all: N+1

Before reaching for an index, check whether the real problem is how many queries run, not how slow one of them is. The classic version: fetch a list, then loop over it fetching something per row —

orders := SELECT * FROM orders WHERE user_id = 42        -- 1 query
for each order:
    SELECT * FROM products WHERE id = order.product_id   -- N more queries, one per order

That's the N+1 query problem: 1 query became 1 + N, and every one of those N pays full network-and-parse round-trip cost. No index fixes this — the fix is restructuring the access pattern itself: one JOIN, or one batched WHERE id IN (...) covering every product at once. This is the single most common runtime-behavior bug ORMs introduce silently (lazy-loading a related field inside a loop looks like a normal field access in the code, hiding the extra query completely) — worth naming by symptom: a list page that gets linearly slower as the list grows is almost always N+1, not a missing index.

4 · Three questions before you touch the schema

In order, before adding an index to anything:

  1. What exact query needs to be fast? Not "this table" — a specific WHERE/ORDER BY shape. An index tuned for one query can do nothing for a differently-shaped one on the same table.
  2. How does its read frequency compare to the table's write frequency? Hot-read, cold-write tables are the easy case; the reverse usually isn't worth it.
  3. How selective is the column? (Section 2.) Low-selectivity columns rarely pay for themselves alone — they sometimes help as the second column in a composite index instead, once one query pattern is already proven to need it.

Only after those three does "what index, exactly" become a Postgres syntax question — B-tree vs. composite vs. partial, and reading EXPLAIN to confirm it actually got used. That's precisely the Go course's Day 5–6 material: worth a look this week if the syntax itself is the gap, not the reasoning.

Q: "A list endpoint got slow as the table grew. Walk me through your first move." A strong answer, out loud: "Before touching the schema, I check whether it's actually N+1 — one query per row hiding inside a loop, which no index fixes. If it's genuinely one query getting slow, I ask what specific WHERE/ORDER BY shape it uses, how often it runs versus how often the table is written, and how selective the filtered column is — a low-selectivity column rarely earns an index on its own. Only then do I reach for EXPLAIN to confirm the index I'm considering would actually get picked, instead of guessing and hoping." The subtext: indexes are a considered trade, not a reflex — and half of "slow query" bugs aren't about indexes at all.
