1 · The bug you can't see: shared state

Your MemStore from Day 1 has a quiet flaw. Look at it with today's eyes:

type MemStore struct {
    links map[string]Link   // no protection — fine for ONE goroutine, a bomb for many
}
func (s *MemStore) Save(ctx context.Context, l Link) error { s.links[l.Code] = l; return nil }
func (s *MemStore) Get(ctx context.Context, code string) (Link, error) { l, ok := s.links[code]; /* ... */ }

The moment a real HTTP server calls this — and it will, next lesson — two requests run Save and Get at the same instant, each on its own goroutine, both touching s.links. That's a data race. Worse: Go's runtime actively detects concurrent map writes and panics on purpose — fatal error: concurrent map writes, un-recoverable, straight to a crash. This is the single most common way a Go service that "worked in dev" falls over under load.

The fix is synchronization: force the goroutines to take turns.

2 · Mutex: one goroutine at a time

A mutex is a turnstile around the shared data. One goroutine holds it; everyone else waits:

type Counter struct {
    mu    sync.Mutex           // guards everything below it
    total int
}
func (c *Counter) Inc() {
    c.mu.Lock()                // wait my turn
    defer c.mu.Unlock()        // release on the way out — even on panic
    c.total++                  // now safe: I'm the only one in here
}

Three rules that separate people who "know mutexes" from people who ship deadlocks:

  • defer the Unlock the line after Lock. Every early return in between then releases automatically — forget one and every other goroutine blocks forever.
  • Pointer receiver, always. (c *Counter), never (c Counter). A sync.Mutex must not be copied — a copy is a different lock guarding nothing. go vet catches this; it's a classic interview gotcha.
  • Keep the critical section tiny. Lock, touch the data, unlock. Never do I/O (an HTTP call, a DB query) while holding the lock — you'd serialize your whole server behind it.

3 · RWMutex: many readers or one writer

A plain Mutex blocks readers against each other too — but two goroutines just reading can't corrupt anything. When reads dominate (linkshort does far more Gets than Saves), a RWMutex lets all the readers in at once and only makes them wait for a writer:

func (s *MemStore) Get(...) (Link, error) {
    s.mu.RLock()               // shared: other readers may hold it too
    defer s.mu.RUnlock()
    // ...read the map...
}
func (s *MemStore) Save(...) error {
    s.mu.Lock()                // exclusive: everyone else waits
    defer s.mu.Unlock()
    // ...write the map...
}

Rule of thumb: reach for RWMutex only when reads clearly dominate. If writes are frequent, the extra bookkeeping makes it slower than a plain Mutex — measure, don't assume. That "measure it" instinct is exactly what interviewers probe.

4 · The real question: lock it, or hand it off?

Go's most-quoted proverb: "Don't communicate by sharing memory; share memory by communicating." Two philosophies for the same problem:

ApproachBest when
Mutex — guard shared state in placeSmall, long-lived shared data many goroutines poke at: a counter, an in-memory cache, a config map. Cheap, direct.
Channel — pass ownership alongHanding a unit of work from one stage to the next, so only one goroutine owns it at a time. Orchestration, pipelines, fan-out (tomorrow).

The mistake is treating it as religion. A shared click counter behind a Mutex is correct and simple; forcing it through a channel-owning goroutine would be cargo-cult. The senior move is naming the trade-off out loud: "few writers, tiny critical section → mutex; handing work between stages → channel." You'll do both this week.

5 · Why "it's just an int" is still a race

Tempting thought: "reading an int another goroutine wrote is fine — no lock needed." Wrong, and this is the memory model talking. Without a synchronizing operation — a mutex, a channel send/receive, or an atomic — the language gives zero guarantee that the reader ever sees the writer's value, or sees a whole value rather than a half-written one. "Works on my machine" is a data race that hasn't lost the coin flip yet. The compiler and CPU are both allowed to reorder unsynchronized memory access.

The lock isn't only mutual exclusion — it's also the happens-before edge that makes one goroutine's writes visible to the next. (Day 12 meets sync/atomic, the lightweight way to get that guarantee for a single counter.)

package link

import (
    "context"
    "sync"
)

var _ Store = (*MemStore)(nil)

type MemStore struct {
    mu    sync.RWMutex          // GIVEN — guards links
    links map[string]Link
}

func NewMemStore() *MemStore {
    return &MemStore{links: make(map[string]Link)}
}

func (s *MemStore) Save(ctx context.Context, l Link) error {
    // TODO 1: this WRITES the map — take the exclusive lock.
    //         s.mu.Lock(); defer s.mu.Unlock()
    s.links[l.Code] = l
    return nil
}

func (s *MemStore) Get(ctx context.Context, code string) (Link, error) {
    // TODO 2: this only READS the map — take the shared read lock.
    //         s.mu.RLock(); defer s.mu.RUnlock()
    l, ok := s.links[code]
    if !ok {
        return Link{}, ErrNotFound
    }
    return l, nil
}

    counts map[string]int
}

func NewClickCounter() *ClickCounter {

}

// Inc records one click for code.
func (c *ClickCounter) Inc(code string) {
    // TODO 3: Lock, counts[code]++, defer Unlock.
}

// Count returns the clicks recorded for code so far.
func (c *ClickCounter) Count(code string) int {
    // TODO 4: Lock (defer Unlock), return counts[code].
}

// Snapshot returns a COPY of every tally — GIVEN. Note the pattern: copy
// under the lock, return the copy, so callers read it without holding it.
func (c *ClickCounter) Snapshot() map[string]int {
    c.mu.Lock()
    defer c.mu.Unlock()
    out := make(map[string]int, len(c.counts))
    for k, v := range c.counts {
        out[k] = v
    }
    return out
}

Step 3 — prove it with -race. Create link/concurrent_test.go (given): a thousand goroutines hammering one counter, plus concurrent reads and writes on the store.

package link

import (
    "context"
    "sync"
    "testing"
)

func TestClickCounterConcurrent(t *testing.T) {
    c := NewClickCounter()
    var wg sync.WaitGroup
    for i := 0; i < 1000; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            c.Inc("go")
        }()
    }
    wg.Wait()
    if got := c.Count("go"); got != 1000 {
        t.Fatalf("Count(go) = %d, want 1000", got)
    }
}

func TestMemStoreConcurrentReadWrite(t *testing.T) {
    s := NewMemStore()
    ctx := context.Background()
    _ = s.Save(ctx, Link{Code: "go", URL: "https://go.dev"})
    var wg sync.WaitGroup
    for i := 0; i < 100; i++ {
        wg.Add(1)
        go func() {
            defer wg.Done()
            _, _ = s.Get(ctx, "go")
            _ = s.Save(ctx, Link{Code: "go", URL: "https://go.dev"})
        }()
    }
    wg.Wait()
}

Step 4 — feel the race. Temporarily gut Inc down to just c.counts[code]++ (no lock) and rerun. This is a production bug caught in CI:

Write at 0x00c000123abc by goroutine 8:
  linkshort/link.(*ClickCounter).Inc ... counter.go
Previous write at 0x00c000123abc by goroutine 7:

Two goroutines, same address, your Inc named on both sides. Also try go test ./... without -race a few times — you'll see Count(go) come back as some number below 1000, different each run: the lost updates the lock prevents. Then put the lock back.
