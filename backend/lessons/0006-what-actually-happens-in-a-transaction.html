What actually happens inside a transaction

Same runtime-behavior track as Lesson 5 — no dependency on it, no dependency on Lessons 2–4 either. "Wrap it in a transaction" is the reflex answer to almost any data-safety worry, frontend or backend. Today's lesson is about the specific, common case where that reflex is wrong.

1 · What a transaction actually promises

A transaction guarantees two different things, and the trap is treating them as one. The part everyone already gets right: atomicity — creating an order and its line items as two INSERTs either both land or neither does; a crash mid-way never leaves an order with no items.

The part "wrap it in a transaction" quietly assumes, and doesn't actually deliver for free, is isolation: what happens when two transactions run at the same time, touching the same row. Postgres's default isolation level lets each individual statement see whatever was committed at the moment it runs — which sounds safe, and is, for a single statement. It stops being safe the moment your "one operation" is actually two statements: a read, then a write based on what you read.

2 · The bug: check-then-act, twice, at once

Picture a flash sale with one ticket left. Two requests arrive within the same millisecond, each inside its own transaction:

-- both transactions run this first, both see remaining = 1
SELECT remaining FROM tickets WHERE id = 7;

-- both transactions then run this — both think it's safe
UPDATE tickets SET remaining = remaining - 1 WHERE id = 7;

Both transactions read 1. Both decide, correctly given what they saw, that selling one more is fine. Both commit. The ticket count goes to -1 — sold twice. Nothing here violated atomicity; each transaction's own statements all committed together, exactly as promised. What broke is that neither transaction knew about the other's read — this specific failure has a name, a lost update.

Frontend-habit trap: "I wrapped my read and my write in one transaction, so it's safe from races" — atomicity says those two statements commit together as a unit; it says nothing about whether another transaction's read-then-write interleaved with yours in between. Those are different guarantees, and only one of them is automatic.

3 · Two real fixes — pick one, know why

The fix is never "wrap it harder." Either make the check part of the write itself, or explicitly claim the row before reading it.

Make the write atomic and self-checking — the check and the act become one statement, so there's no gap for another transaction to land in:

UPDATE tickets SET remaining = remaining - 1
WHERE id = 7 AND remaining > 0
RETURNING remaining;
-- 0 rows returned = someone else got there first; your code checks that, not a prior SELECT

Or, when the logic between read and write is too complex to fold into one statement, explicitly lock the row on read with a row lock:

SELECT remaining FROM tickets WHERE id = 7 FOR UPDATE;
-- the second transaction's FOR UPDATE now BLOCKS here until the first commits or rolls back
-- ...application logic here, safely...
UPDATE tickets SET remaining = remaining - 1 WHERE id = 7;

FOR UPDATE makes the second transaction wait instead of racing ahead on a stale read — the isolation guarantee the plain SELECT never gave you. (Postgres also offers stricter isolation levels — REPEATABLE READ, SERIALIZABLE — that catch this class of conflict automatically and force one transaction to retry; worth knowing the knob exists, though the two patterns above cover the vast majority of real cases without reaching for it.)

Q: "Two requests could both try to decrement the same limited-stock counter at once. How do you prevent it going negative?" A strong answer, out loud: "A plain transaction around a SELECT-then-UPDATE doesn't fix this — that's a classic lost update, both transactions can read the same value before either writes. I'd either fold the check into the UPDATE itself — WHERE stock > 0, check the row count returned — or, if the logic between read and write needs to be more than one statement, take an explicit row lock with SELECT ... FOR UPDATE so the second transaction waits instead of racing on stale data." The subtext: naming the failure mode (lost update) and knowing two concrete fixes reads very differently from "I'd wrap it in a transaction."
