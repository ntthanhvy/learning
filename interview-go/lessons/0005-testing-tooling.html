Go Screening Kit · Part 5 · 2 questions · pick 1 per screen

Testing & tooling

One good question here is enough for a screen — testing habits correlate strongly with everything else, so a candidate who's strong in Parts 1–4 is rarely weak here. Use the second question only if you're on the fence and have minutes to spare.

Question 17

"How do you test code that depends on a database or an external API? Walk me through your actual setup on your last project."

Why ask: "on your last project" turns a theory question into an experience question. Everyone knows the words "mock" and "interface" — few can describe a working setup.

Answer guideline — a strong answer covers

  • Design for it: the service depends on a small consumer-defined interface (ties back to Q10), so unit tests substitute a hand-written fake — in Go this rarely needs a mocking framework.
  • Table-driven tests as the default idiom: a slice of cases, t.Run(tc.name, ...) subtests; t.Parallel() awareness is a bonus.
  • The unit/integration split: fakes prove your logic; they don't prove your SQL. Real queries need a real database — testcontainers / dockerized Postgres, a shared test DB with per-test transactions or truncation, or sqlite-in-memory with the caveat that it's a dialect lie.
  • External APIs: httptest.Server to fake the remote end and exercise the real HTTP client — timeouts, retries, error decoding. Bonus: httptest.NewRecorder for testing their own handlers.

Strong signals

  • Says fakes don't test SQL — knows what each layer of tests proves
  • Names httptest — the stdlib tool a real Go tester can't avoid
  • Describes a working CI-ready integration setup, with its warts

Weak / red flags

  • "We mock everything" — 100% mocks, zero integration coverage, no discomfort about it
  • Or the inverse: every test hits a shared dev database, flaky and slow, no isolation story
  • Can't explain what a specific test they wrote actually verified

Follow-up probe

"Your integration tests share one Postgres. Two tests run in parallel and interfere. What are your options?" (Good: per-test transactions rolled back, per-test schemas/databases, truncate between tests, or serialize the suite — with trade-offs of each.)

3/5 = interface + fake for units, some real-DB story. 5/5 = layered strategy with isolation trade-offs and httptest fluency.

Question 18

"What does CI look like for a Go service you'd set up today — what runs on every push, and what gates a merge?"

Why ask: cheap question, quick answer, and it exposes whether they've owned a pipeline or only pushed into one someone else built.

Answer guideline — a strong answer covers

  • The core four: go build ./..., go vet ./..., a linter (golangci-lint is the de facto standard), and go test -race ./... — race flag in CI is the mark of someone who's absorbed Part 2.
  • Formatting non-negotiable: gofmt/goimports checked, not debated.
  • Module hygiene: go.mod/go.sum tidy check; builds are reproducible by default thanks to the module cache + sumdb.
  • Bonus: integration tests as a separate (possibly gated) job, coverage as a signal not a gate, govulncheck, multi-stage Docker build with a small final image.

Strong signals

  • -race in the test invocation without being asked
  • Distinguishes fast per-push checks from slower merge gates
  • Has set one of these up themselves, can name the annoyances

Weak / red flags

  • "CI runs the tests" and nothing else after prompting
  • Never heard of go vet or any linter
  • All of it was "the DevOps team's job" — fine once, a flag if every infra topic lands there

3/5 = build/vet/lint/test list. 5/5 = race flag + owns the pipeline shape and its trade-offs. Skip this question entirely if Q17 went deep and time is short.
