GroupBy: split, apply, combine

If Lesson 3's practice isn't all green yet, no problem — today reuses the exact same file and the exact same coerced amount column, so finishing this one also finishes that one. Today's question is the one every interview wrangling task eventually asks: totals, counts, or averages per group — the pandas answer to SQL's GROUP BY.

Recap, in one line

Lesson 3 decided what to do with a missing value in one column at a time. Today's operations look at the whole table again, but sliced into groups first — same vectorized instinct, one level up.

1 · Split, apply, combine — and its SQL cousin

A groupby operation is always the same three steps, and naming them is the whole mental model: split the rows into groups by a key, apply some function to each group independently, combine the per-group results back into one answer. df.groupby("customer") alone does only the first step — it produces a lazy "grouped" object, nothing computed yet — until you chain an aggregation onto it:

SQLpandas
SELECT customer, SUM(amount) FROM orders GROUP BY customerdf.groupby("customer")["amount"].sum()
SELECT customer, COUNT(*) FROM orders GROUP BY customerdf.groupby("customer").size()
SELECT customer, AVG(amount) FROM orders GROUP BY customerdf.groupby("customer")["amount"].mean()

The result of .sum()/.mean() on a grouped single column is a Series indexed by the group key — customer becomes the row label, not a regular column, until you call .reset_index() to put it back as one. That's the "combine" step made visible: one row per group, automatically.

2 · .size() vs .count(): the NaN gotcha

These look interchangeable and aren't, the moment a column has missing values. .size() counts rows per group, full stop — it doesn't look at any column's values. .count(), called on a specific column, counts only the non-null values in that column per group:

df.groupby("customer").size()               # rows per group — NaN or not, doesn't matter
df.groupby("customer")["amount"].count()    # non-null amount values per group — NaN rows don't count

SQL's COUNT(*) matches .size(); SQL's COUNT(amount) — a much less commonly reached-for form, but it exists precisely for this — matches .count() on that column. Picking the wrong one silently answers a different question: "how many orders does this customer have" (.size()) is not the same fact as "how many orders have a known amount" (.count()), and an interview task that asks for the first while you reach for the second will look right until someone checks a customer with a gap.

3 · Multiple aggregations at once: .agg()

Chaining .sum() directly gets one number per group. The moment you want several — total and count and average, say — .agg() takes a dict (or named arguments) and returns them all in one DataFrame, one pass over the data instead of three separate groupby calls:

df.groupby("customer")["amount"].agg(
    total=("sum"),
    orders=("count"),
    avg=("mean"),
)

Each keyword becomes a column name in the result — this is named aggregation, and it's the version worth typing from muscle memory: the plain-dict form of .agg() works too, but its default column names are uglier and easy to get wrong under interview pressure.

4 · The SQL HAVING step: filtering groups

SQL's GROUP BY ... HAVING count(*) > 1 filters on the aggregated result, not the original rows — pandas has no separate keyword for this, because it doesn't need one: the groupby result is just a Series/DataFrame, and you filter it the same boolean-mask way as any other:

totals = df.groupby("customer")["amount"].sum()
counts = df.groupby("customer").size()
totals[counts > 1]     # SQL: ... GROUP BY customer HAVING count(*) > 1

Nothing new here mechanically — it's Lesson 1's boolean-mask filtering, just applied to a groupby result instead of the raw table.

Q: "Given an orders table, find customers whose total spend is above $150, ordered highest first." A strong answer, out loud: "That's split-apply-combine: group by customer, sum the amount column per group — that's the SQL GROUP BY plus SUM. Then I filter the resulting Series the same way I'd filter any Series, totals greater than 150 — that's the HAVING step, just as a boolean mask instead of a separate keyword. Last, sort_values descending. One line per step, no loop over customers anywhere." The subtext: naming split-apply-combine explicitly, and knowing HAVING isn't a separate pandas concept, reads as fluency rather than pattern-matching a snippet.
