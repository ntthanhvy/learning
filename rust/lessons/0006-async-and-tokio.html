1 · A future does nothing by itself

In Go, go f() starts running immediately — the goroutine is scheduled the instant that statement executes, whether or not anything ever reads its result. Rust's future is the opposite: writing async fn check_link(...) and calling check_link(l) builds a future value and does nothing — no code inside the function body runs — until something drives it forward. That "something" is either .await, which drives the future to completion right where you write it and pauses the surrounding async function while doing so, or tokio::spawn, which hands the future to a scheduler to run independently — section 3.

2 · No built-in runtime — that's what tokio is

Go's runtime — the scheduler that actually runs goroutines — is compiled into every Go binary; you never install it separately. Rust's standard library ships async/await as language syntax but deliberately no runtime to execute futures on — that's a library decision, not a language one, so different projects can pick different tradeoffs (embedded systems need a tinier one than a web server does). tokio is that runtime — the industry-standard choice, and what axum (Day 7's capstone) is built on:

#[tokio::main]
async fn main() {
    // everything in here can .await
}

#[tokio::main] is sugar: it rewrites that async fn main() into an ordinary, synchronous fn main() that starts a tokio runtime and blocks on your async body — the one seam where a plain, non-async function actually kicks off an async program at all.

3 · tokio::spawn: fan-out, Go's shape, Rust's rules

tokio::spawn(future) is Rust's go f() — it hands a future to the runtime and returns immediately, letting that future run concurrently with everything else. The difference from Go shows up at the return value: tokio::spawn gives back a JoinHandle<T>, and .awaiting that handle is how you get the task's result back — closer to Go's pattern of a goroutine sending its result down a channel for the caller to receive, made explicit instead of needing a channel you wire up by hand.

let handle = tokio::spawn(async move {
    // runs concurrently with the rest of main
    42
});
let result: i32 = handle.await.unwrap(); // .await gives Result

That Result wrapper is not decoration: .await on a JoinHandle returns Ok(value) if the task finished normally, or Err(JoinError) if it panicked — a panic inside a spawned task doesn't crash your program the way an unrecovered panic on the main thread does; it's reported back through this Result instead, another Rust error-as-value moment (Day 3).

The rule that trips people up first: a spawned future must be 'static — it can't borrow anything from the function that spawned it. The runtime may run that task on another thread, at some later, unknown time, possibly after the spawning function has already returned — a borrowed reference could be dangling by then. The fix is always the same one Go never forces on you explicitly: own your data, don't borrow it. That's why section 5's check_link takes link: Link by value, not &Link — the task owns the link it's checking, for exactly as long as it needs it.

4 · Threads vs. tasks, and Send/Sync in one sketch

std::thread::spawn — Rust's direct equivalent of an OS thread, like Go used to require before goroutines existed — creates one real OS thread per call: heavier, but true parallelism, no runtime needed. A tokio task is much cheaper: tokio's multi-threaded runtime schedules many tasks cooperatively across a small pool of OS threads — the same M:N idea behind Go's own goroutine scheduler, implemented as a library instead of built into the language.

Both still answer to two compiler-checked traits, automatically derived for almost every ordinary type — nothing to write by hand, unlike yesterday's #[derive(...)]:

  • Send — safe to move a value to a different thread. Link, being just two Strings, is Send automatically.
  • Sync — safe for multiple threads to hold &T to the same value at once.

tokio::spawn requires its future's output to be Send (it might finish on a different worker thread than it started on) — the same requirement std::thread::spawn's closure has always had. Tasks and threads share the ownership rules; tasks are just far lighter weight to create.
